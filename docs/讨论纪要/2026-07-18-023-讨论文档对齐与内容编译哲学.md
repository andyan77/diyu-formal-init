# 《讨论.txt》对齐审计与内容编译哲学

日期：2026-07-18

状态：**用户明确观点已写入项目记忆；架构细化进入 D-031 待决策，未把讨论中的助手方案冒充定稿。**

> **后续修订提示（2026-07-18）**：本纪要关于 D-004“四类候选”的表述保留为历史对齐记录；当前已由用户确认 P1—P5 五类正式基线，见 [ADR-013](../架构决策/ADR-013-五类内容产品正式基线.md)。D-031 的增益与记忆边界仍待裁决。

## 为什么必须对齐

《讨论.txt》提出的不是数据库产品选型变化，而是内容生产哲学的进一步修订：笛语不应把 LLM 限定为检索资料后的复述器，而应允许它从用户低负担输入出发，结合品牌/账号设定、商品与领域知识，完成语义理解、专业解释和内容创作；证据治理只在承重硬事实和高风险表达处收紧。

这与已确认的 D-012 混合技术路线并不冲突，但会改变 D-004 内容产品合同、D-011 内容记忆回写、D-013 证据与创意口径及 D-016 最小语义模型的运行语义。

## 来源状态说明

《讨论.txt》是未标注说话人的多轮对话拼接，且第 673 行存在文本粘连。可明确识别的用户原话集中在第 1、1132、1371 行；其余大量产品数量、字段、机制、质量门和 Gold Case 数量属于对话中助手提出的方案，不能自动登记为用户已确认结论。

## 用户明确表达、需要稳定继承的观点

1. 普通内容生产不能被“事实、证据、来源”普遍绑架。系统应能基于一句粗糙的门店或生活输入，调用知识进行专业解释、情境组织和内容创作；若只能复述用户已经完整写好的材料，产品价值不足。
2. 品牌调性、表达风格、账号人设和店铺人设主要是企业配置的内容预设；它们决定系统以什么身份、立场和方式理解输入，不应普遍要求外部举证。
3. 输入可以来自广泛生活片段，包括试穿、经营、夫妻、婆媳、育儿、工作、年龄、身体感受、朋友关系和城市生活；是否进入成品由品牌、账号角色和内容产品决定。
4. 用户对自己输入或配置的人设、经历、事件以及最终发布承担责任；系统不是人生事实调查机构，也不负责判断用户会不会因虚构或前后矛盾而“人设崩塌”。

## 必须纠正、不能原样照录的两句话

### 1. “结构化入库的数据就是事实、证据和来源”不成立

结构化只证明系统记录了该输入，不证明事件在外部世界客观发生。准确口径是：

> 用户输入或配置的数据，可以直接成为其声明并允许系统使用的**内容前提**；系统记录 `origin = user_provided`、适用账号/角色和使用范围，不要求为低风险内容做外部真实性调查，但也不把它标成系统已验证事实、商品硬事实或行业知识。

为避免“授权”被误解为第三方人物、媒体或跨工作区的法定使用权，正式名称使用**用户声明内容种子**或**运行时生活种子**，不使用“用户授权即权利完备”的表述。

### 2. “领域通用专业知识都属于硬事实”过粗

领域知识还包括条件性规律、审美判断、经验启发和创意候选。只有稳定术语、材料参数等确定性部分接近硬事实；版型适配、显高显瘦、风格判断和穿搭解释必须保留条件、适用范围和判断性质，不能与商品成分、价格、活动日期并列为无条件真理。

## 内容生成需要区分的六种信息性质

这不是六套数据库，也不要求六个核心本体类别，而是生成、检查和回写时必须分开的语义性质：

| 信息性质 | 示例 | 系统使用边界 |
|---|---|---|
| 权威硬事实/硬约束 | 商品成分、当前价格、活动时间、门店地址、有效政策、身份及定量经营信息 | 优先调用当前有效来源；不得擅自改写，缺失、冲突或过期时降级、提示或停止对应主张 |
| 条件性专业知识 | 衣长对比例的可能影响、版型对不同身形的适用判断 | 允许生成条件化解释和替代建议，不得说成人人适用的确定效果 |
| 表达与人格预设 | 品牌立场、账号人设、语言风格、禁区、内容落点 | 作为配置约束，不作外部人生真实性核验 |
| 用户声明内容种子 | 一句门店观察、家庭琐事、情绪片段、评论问题、用户配置的经历 | 可直接作为本任务内容前提；来源是用户声明，不等于系统已验证事实 |
| 运行信号与状态 | 趋势、评论、上一集进度、当前门店状态 | 带时间和作用域使用，不自动固化为永久领域知识 |
| 系统派生与内容资产 | 专业解释候选、叙事结构、情景演绎、脚本、成品 | 保留派生关系；不能循环回写成原始用户事件或权威硬事实 |

## 内容模块的准确定位

内容模块可以概括为：

> **用户声明内容种子 + 品牌/账号预设 + 选择性硬知识与条件性专业知识 + LLM 语义和叙事增益 → 符合内容产品合同的成品。**

这可以称为“生活语义编译”或“内容编译”，但只是内容模块的运行哲学，不是整个笛语系统的四层总架构。租户、组织、品牌、工作区、权限、账号、素材、版本、规则、留痕和零售经营型陈列仍然属于全局架构。

生活输入可以很宽，但系统不能退化为泛家庭矛盾或情绪流量生成器。输入必须能自然连接目标人群、账号角色、穿着/消费/门店生活或品牌立场；不相关时不应强行接商品、强行制造冲突或强行导流。

## 模型允许增益，但增益必须分层

当前建议由每个 `ContentProduct` 配置轻量 `enrichment_policy`，而不是建设独立“增益引擎”或新的权利系统：

- 可自由生成：内容结构、节奏、比喻、开头结尾、通用场景连接和表现方式；
- 可条件化生成：基于商品和领域知识的专业解释、适用边界、替代选择；
- 需要情景标识或轻量确认：读者会理解为现场真实发生的具体对白、动作、心理活动；
- 不得无输入依据新增：人物核心身份、重大人生经历、商品参数、价格、精确交易结果、功能效果和定量经营事实。

这不是替用户调查真假，而是区分“用户提供的内容前提”和“系统自己新增的断言”。系统也不能承诺绝对保证外部事实正确或平台一定放行；它只能优先使用当前有效权威数据和版本化规则，在缺失、冲突或过期时按合同处理，最终由发布者复核。

## 回写与知识升级边界

至少保持以下隔离：

1. 原始用户内容种子；
2. 系统专业解释候选；
3. 叙事和表达资产；
4. 用户采用的内容成品；
5. 经业务确认后形成的稳定表达规则；
6. 另有权威依据支持的事实型知识。

用户采用或确认系统成品，只表示该成品可以作为内容资产或受控内容记忆复用，不表示其中所有系统补全细节已经被客观核验。生成内容、高播放和用户接受都不能自动把内容升级为商品事实、人物履历、定量经营事实或通用专业知识；ADR-010 与 D-011 的防循环污染边界继续有效。

## 对现有决策的影响

| 决策/文档 | 对齐结果 |
|---|---|
| D-012 / ADR-001 混合路线 | 不变；此次改变的是信息权威与回写路径，不是数据库技术选型 |
| D-013 证据与创意 | 需要细化：证据是承重事实和高风险表达的风险阀门，普通内容种子不普遍举证 |
| D-014 分层治理 | 需要补充用户声明种子、表达预设和系统派生内容的语义分层 |
| D-027 / ADR-006 责任原则 | 不推翻；补充“系统不调查用户人生真假，但要区分用户前提与自身新增断言” |
| D-004 内容产品 | 后续已确认 P1—P5 五类正式基线；合同仍需逐类确认接受种子、增益边界和回写去向 |
| D-011 知识升级 | 必须把事实型知识升级与内容记忆复用分开 |
| D-016 最小语义模型 | 暂不新增 `ExperienceEnrichmentMechanism`、`BrandGenerativeKnowledge` 等核心类别；先用任务输入、派生类型、状态和来源谱系验证 |

## 尚未由用户定稿的事项

- P1—P5 的名称和稳定价值已经确认；逐类合同细节仍需审阅；
- 每类产品允许多大的专业解释、情景演绎和叙事增益；
- 用户采用后的系统派生内容，什么条件下进入可复用内容记忆；
- 广泛生活主题由哪些品牌、角色和账号启用；
- 系列状态、趋势信号和内容记忆哪些进入第一版；
- 具体合同字段、质量门和 Gold Case 数量。

这些统一登记为 D-031，与 D-004、D-011、D-016 联动裁决，不能从《讨论.txt》中的助手建议直接升级为正式架构。
