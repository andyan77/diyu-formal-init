# 2026-07-17 企业主导渠道协作系统与责任边界

## 一、本轮用户补充

1. 系统不能按面向社会任意注册的普通 SaaS 理解，而是品牌企业采购后用于内部及渠道协作。
2. 品牌企业统一开设账号，授权区域代理和加盟终端使用，并可停止某个账号继续使用系统的权限。
3. 用户希望品牌企业作为主方，使总部、区域和门店之间的约束尽量简化，避免首期系统复杂度失控。
4. 用户确认系统硬约束至少覆盖：
   - 法律、隐私、知识产权和人物授权；
   - 商品、价格、库存等可验证事实；
   - 品牌授权范围；
   - 各自媒体平台带版本和生效时间的规则。

## 二、产品身份的准确表述

“企业内部系统”与“SaaS”描述的是两个不同维度：

- SaaS 描述软件由服务方统一交付和持续维护；
- 企业主导描述谁采购、谁开账号、谁治理使用网络。

因此，首期正式定位为：

> **品牌企业采购并治理、区域代理与独立加盟店受邀使用的多组织渠道协作型 SaaS。**

面向非技术业务人员可以简称“企业主导的渠道协作系统”。它不是公开注册、自助购买、任何门店都能自行加入的开放平台。

这里仍须保留一个事实：代理商和加盟店虽然由品牌企业授权使用系统，但其经营身份不是品牌企业内部部门或员工，不能因此合并工作区、数据权利和行为责任。

## 三、首期最小账号治理模型

首期只需要以下生命周期：

`企业建立租户 → 企业管理员邀请账号 → 绑定区域/门店工作区与角色 → 启用 → 暂停/恢复 → 终止`

具体规则：

1. 一个试点签约企业对应一个租户与安全边界。租户代表合同、数据治理和安全责任主体，不等于一个品牌；首期虽然只有一个品牌，模型仍允许同一企业租户后续管理多个品牌。
2. 总部企业管理员统一邀请或创建实际操作人的登录身份，再分别授予区域代理、加盟店工作区的成员资格和权限。每个自然人一人一号，禁止多人共用“门店登录”；系统接口身份另行管理，不用于人员登录。
3. 每个账号必须绑定所属经营主体、工作区、角色和可使用范围，不能只靠“总部/区域/门店”一个标签决定全部权限。
4. 企业可暂停或终止自然人登录身份在本企业租户中的成员资格及相应授权；系统应同时阻断其进入该租户和发起新业务动作，并使该租户相关会话/凭据失效。自然人登录身份不可转交，可交接的是岗位、人设、企业发布账号运营权和业务任务。
5. 停用不删除历史审计，不改变素材权利归属，不抹除已完成的发布、导出和例外记录。停用后的门店私有素材如何导出、保留、删除或交接，必须另定退出规则，不能让资料因账号失效而悬空。
6. 首期不建设公开注册、自助购买、代理商自行创建租户、复杂跨租户共享身份等能力。

这已经能满足试点，不需要先做一套完整的互联网 SaaS 商业化账号体系。

## 四、必须纠偏：治理主方不等于替所有参与方承担一切

品牌企业可以是系统采购、账号和官方知识治理的主方，但架构不能直接声明其替独立加盟店承担所有发布、本地素材和违规操作的责任。谁提供资料、谁控制行为、谁作出发布决定，必须能够在系统中区分并留痕。

建议职责分工如下：

| 参与方 | 首期系统职责 |
|---|---|
| 品牌企业 | 采购系统；授权/停用账号；维护官方品牌规则、商品资料、活动素材及其授权范围 |
| 区域代理 | 对自己上传、补充、维护和向辖区交付的区域资料及操作负责；承担培训、辅导与异常协助 |
| 独立加盟店 | 对本店采集的原始素材、人物/媒体授权、手工录入的本地事实和自己最终发布的内容负责 |
| SaaS 提供方 | 对系统安全、访问控制、审计能力、受托数据处理及已约定规则的正确实现与执行负责；不承诺法律或平台规则绝对完整、实时、无遗漏 |

这张表定义的是产品中的职责、权限和审计归属，不直接替代合同或适用法律对最终法律责任的认定。法务可在试点合同阶段据此校准条款，但系统必须先能区分上述行为主体。

## 五、约束体系如何简化而不失控

用户列出的四类约束不应实现为四套独立审批产品，也不能反过来发明一套能解释所有法律和业务的万能规则语言。第一版采用：

> **统一规则登记册 + 统一检查编排、结果合同和审计 + 分类型校验器。**

权利授权、事实时效、品牌授权、平台红线和风格软策略共用入口与留痕，但分别使用适合自身语义的检查逻辑。首期不做通用规则语言、任意管理员脚本或自动法律推理。

| 规则信息 | 用途 |
|---|---|
| 类别与来源 | 区分法律/权利、经营事实、品牌授权、平台规则和品牌软策略 |
| 适用范围 | 指向租户、品牌、区域、门店、内容产品、平台与账号 |
| 版本和时间 | 记录已知版本、生效/失效时间、观察时间与最后核验时间；平台没有正式版本号时也可追溯 |
| 所需证据 | 指定授权文件、权威商品资料或库存快照；用户确认只适用于软建议/例外理由，不替代硬事实证据 |
| 维护状态与责任人 | 记录谁维护、谁批准、是否过期/被替代；每次任务冻结所用规则快照 |
| 处置能力 | 按规则类别限制可用处置，不能让风格评分随意升级为硬阻断 |

首期只保留三种结果：

1. **硬阻断**：结构化可判定的明确禁项、缺少必要人物/媒体授权、超出品牌授权范围、确定性事实无来源/过期/冲突等。
2. **警告并确认**：品牌风格偏离、主观经营建议存在不确定性或可容忍的低风险歧义等；用户确认不能替代事实证据或越过硬约束。
3. **允许并留痕**：规则满足或不构成风险，但保留所用规则版本与检查结果。

这三种只是面向用户的最终结果。内部还要分开记录：检查状态（通过、失败、未知/过期、不适用）和补救动作（补证、删除、条件化改写、专业复核）。事实证据不足时，可以把“本店现货充足”改为“库存请到店确认”，但改写后必须重新抽取事实主张并复检；原始失败版本仍保持阻断，不能把一次改写本身算作放行。

这里的统一治理入口不是自动法律顾问，也不能保证平台最终审核通过。系统只执行已经登记来源、负责人、审核状态和有效期的规则；第一版只覆盖 D-004 最终选定目标平台的明确红线。平台规则还需记录原始来源、适用内容形态/账号/地区、最后核验时间、新鲜度状态和替代关系；规则缺失、过期或含义不清时不得显示“已合规”，必须提示、降级表达或阻断高风险动作，不能让大模型自行猜一条规则。

硬检查只约束结构化事实主张、素材使用、授权范围和目标平台，不把整篇内容逐句强绑证据。生成前检查素材与权限，生成后抽取事实主张复检；叙事、比喻、节奏、情绪和创意演绎仍保持自由，前提是不伪造事实或越权。

## 六、本轮结论与剩余确认

可以直接进入架构基线的内容：

- 产品是企业主导、封闭邀请制的渠道协作系统，同时仍可采用 SaaS 交付；
- 一个试点签约企业形成一个租户/安全边界，不把租户写死为单品牌；
- 企业统一邀请、授权、暂停和终止渠道成员资格；人员一人一号并可追溯到具体使用人，不要求建设面向社会的公民实名核验；
- 停用不删除历史审计；
- 四类约束共用规则登记、编排、结果和审计，但使用分类型校验器，不建设四套系统或万能规则引擎。

用户后续已明确渠道承担最终复核与发布主责，并进一步提出“赋能而非监管、各方对各自行为负责”的原则。当前有效责任表述为：

> 品牌企业是系统采购、账号和官方知识治理主责方；区域代理对自身上传和交付负责；独立加盟店对本地素材、人工事实输入和自主发布负责；SaaS 提供方对系统安全、访问控制、审计、受托处理、已约定规则的正确执行及适用法律直接规定的义务负责。系统赋能但不代替任何一方进行最终复核或发布。
