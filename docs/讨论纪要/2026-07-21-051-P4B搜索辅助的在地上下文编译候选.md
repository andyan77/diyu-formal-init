## 一、用户本轮核心观点

用户提出：为入驻品牌减负，系统可以在工作流中增加搜索工具，由系统主动获得“地域内容基线 + 可选当次在地信号”，避免要求品牌填写和确认额外地域资料，并把内容生产本地化建设成系统核心能力。

这个方向中的“搜索替代查资料劳动、取消独立地域问卷”成立；“搜索可以完全替代品牌—地域关系判断”不成立。公开网页能告诉系统当地发生了什么，却不能证明某家门店实际服务谁、某个账号有资格怎样说，或者这个品牌愿意以什么关系进入当地生活。

## 二、正式名称与通俗解释

**正式名称：搜索辅助的在地上下文编译。**

通俗说，就是：

> **品牌不需要先填写地域问卷。系统使用已经存在的品牌战略基线、账号身份和经营区域，在真正需要本地信息时后台搜索，直接把少量有效线索编译进当前成品；用户只在不准确时自然纠正。**

真正的核心能力是“内容生产本地化编译”，搜索只是其中一个可替换的信息入口，不能把产品能力绑定为某一家搜索服务或某一个模型供应商。

## 三、三种方案比较

| 方案 | 做法 | 优点 | 主要问题 | 建议 |
|---|---|---|---|---|
| A 搜索完全接管 | 搜索城市、品牌和热点后直接生成并沉淀地域结论 | 入驻负担最低，看起来最智能 | 搜索不能证明品牌经营选择、实际客群和账号资格；容易形成城市标签、SEO/旅游内容拼贴和长期错误 | 否定 |
| B 搜索后逐条确认 | 系统列出来源和判断，要求区域人员逐条审核 | 表面最稳妥 | 把品牌从“填表”变成“替系统审资料”，还需审核页、状态和冲突处理，首版过重 | 否定 |
| **C 零表单搜索启动 + 使用中渐进校准** | 不新增地域入驻表单；系统按需搜索、编译临时上下文并直接交付成品，用户只在不准时自然纠正 | 减负、可立即工作、保持品牌约束，并能随使用逐步变准 | 不能把搜索不到的经营事实写成确定结论；跨账号长期复用仍需有权限的人主动保存 | **推荐** |

## 四、推荐主干

### 4.1 不新增地域入驻任务

系统直接复用三项已有信息：

1. D-014 已确认的品牌内容战略基线；
2. 企业 IP 矩阵中当前发布账号的人设、组织层级和表达资格；
3. 账号已经具备的经营区域或门店位置。

不再要求品牌额外填写“当地人怎样生活、当地品牌关系姿态、方言和民俗禁区”等独立地域表单，也不把 P4 设为必须先完成地域基线确认才能使用。

### 4.2 搜索只解决它真正能解决的内容

系统可以按需搜索：

- 相对稳定的公开地域背景，如季节、气候、交通与生活空间；
- 公开的节庆、展览、公共活动和文化资料；
- 当天天气、公共事件和具有时效性的公开信号；
- 用于发现当地生活命题的公开讨论线索。

系统不能仅凭搜索断言：

- 某门店的实际目标顾客、客流和购买习惯；
- 顾客真实说过或经历过什么；
- 门店库存、价格、活动、营业状态和服务能力；
- “本地女性都怎样”“三四线消费者更怎样”等群体结论；
- 品牌愿意采用某种方言、民俗、幽默或关系姿态。

品牌关系姿态由 D-014 品牌战略基线和账号人设共同约束，不从搜索排名中推导。

### 4.3 按需调用，不把每次生成变成研究项目

只有本地语境或当前公开信息会实质改变成品时，系统才调用搜索。普通内容不强制搜索；搜索不可用、结果不足、过期或互相冲突时，删除相应表达并继续完成仍然成立的成品。

首版只需要一个系统全局共享、供应商可替换的搜索工具入口。优先使用模型供应商或商用 SDK 已提供的托管搜索能力，不建设爬虫、搜索引擎、持续采集管道、城市知识图谱、独立搜索中心或专职搜索 Agent。

### 4.4 最小事实边界留在后台

当公开动态事实被实际写进成品时，后台只保留支持该表达所需的最小来源、地域范围和检索时间；这些不作为默认成品的一部分展示给普通用户，也不保存模型隐藏推理。

搜索结果默认只服务当前任务，不自动写入品牌知识库或地域长期记忆。用户的自然语言纠正在当前任务立即生效；只有具有相应权限的人主动要求“保存为区域长期设置”时，才允许后续跨账号复用。高播放、系统采用或反复搜到都不能自动升级为品牌事实。

## 五、一个具体场景

“折线之间·成都某门店”准备周五内容。系统已有品牌内容战略、门店账号身份和经营位置，后台搜索发现当周降温以及附近一项公开活动，因此可以主动发现一个与步行、室内外切换相关的当地内容机会。

但系统不能因此写“成都女性都追求松弛感”，也不能虚构“本店顾客最近都在问”，更不能擅自使用方言。最终内容怎样看待这段生活，仍由“折线之间”的品牌关系姿态和该门店账号身份决定。

如果用户自然纠正“我们这里顾客下班后更常去运动，不是接孩子”，当前成品立即按此修改；只有用户具有区域配置权限并主动要求长期保存时，这句话才成为区域共享设置。

## 六、它在系统中的位置

内容生产本地化继续是 P1—P5 共享能力：

- 主要帮助当地受众选择，路由 P1；
- 主要解释商品在当地条件下的取舍，路由 P2；
- 主要积累本地账号人格，路由 P3；
- 主要经营品牌与当地人的关系，路由 P4；
- 主要创造真实商品的当地视觉欲望，路由 P5。

搜索工具属于系统全局编排能力，不单独形成第六类产品，也不为 P4 建一套专属工作流。

## 七、最小验收条件

1. 除已有账号区域信息外，品牌无需填写新的地域表单即可开始生产；
2. 同一品牌和主题换区域后，生活问题、判断或关系表达发生实质变化，而不只是替换城市名；
3. 同一地区换品牌或账号后，品牌立场和发声资格仍发生实质变化；
4. 写入成品的公开动态事实具有最小内部来源和检索时间；无可靠结果时省略、不猜；
5. 搜索不可用时可以安全降级，不阻断不依赖实时地域事实的完整成品；
6. 搜索结果和用户临时纠正均不自动污染品牌知识库或跨账号地域设置。

## 八、当前技术可行性说明

当前主流模型 API 已提供托管式网页搜索或服务端搜索工具，因此首版可以把搜索作为可替换工具接入，不必自行建设爬虫和索引系统。具体供应商、SDK、费用上限和调用策略仍属于后续技术选型，不在 P4-B 冻结。

- OpenAI API 官方文档：[使用内置 web search 扩展模型](https://platform.openai.com/docs/quickstart/make-your-first-api-request)
- Anthropic 官方文档：[服务端工具与 web search 的调用方式](https://docs.anthropic.com/ko/docs/agents-and-tools/tool-use/overview)

## 九、正式确认与唯一下一动作

用户于 2026-07-21 正式确认，以本条取代 050 中“必须先由当地经营者确认一份地域基线”的初始候选：

> **P4-B 采用“搜索辅助的在地上下文编译：零表单启动、使用中渐进校准”。品牌无需另行填写和确认地域内容基线；系统以已有经营区域、D-014 品牌战略基线和账号人设为约束，按需搜索公开地域背景与当次信号。搜索不能代替品牌关系判断，不能创造目标受众、顾客或门店经营事实；不可靠信息直接省略，搜索失败不阻断内容生产。用户可以自然纠正，只有有权限者主动保存后才成为区域长期设置。该能力由 P1—P5 共享，只有关系经营成为主要成果时才路由 P4。**

用户同时备注：系统初期大概率通过 DIFY 编排落地交付。该备注记录为高概率实施路线，不把 DIFY 冻结为 P4 产品语义、永久供应商或唯一运行框架；P4-C 继续使用与编排平台无关的成品合同表达。

P4-B 据此关闭，当前唯一下一动作进入 P4-C“在地关系命题的成品化编译”。

## 十、P4-C 确认时归位的 P4-B 补充

用户随后在确认 P4-C 时要求将两项边界明确放回 P4-B，而不是作为 P4-C 的顺带说明：

1. **主动发现的来源范围**：系统使用已有品牌知识、账号身份与历史、用户已保存的区域设置，以及本合同允许的有来源、有地域范围和检索时间的按需公开搜索结果；所谓平台信号只有在合法可访问且具有时效信息时才能使用。第一阶段不承诺不受限实时本地搜索、持续平台抓取、POI 数据流、ERP/POS 或每次任务必搜。
2. **D-006 非交易边界正式关闭**：第一阶段本地生活只建设在地内容、用户明确提供的门店事实和可选自然到店指引；不建设团购、预约、下单、支付、核销、评价运营、广告归因或即时零售。这一能力边界不把 P4 的关系价值缩窄成门店事实清单。

上述补充与 P4-B 原确认口径共同有效。P4-C 随后亦已确认，当前唯一下一动作进入 P4-D。
