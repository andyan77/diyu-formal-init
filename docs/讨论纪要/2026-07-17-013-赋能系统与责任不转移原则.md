# 2026-07-17 赋能系统与责任不转移原则

## 一、用户提出的核心哲学

> 笛语系统本质上是赋能系统，不承担监管责任；企业方和对应责任人对各自行为负主体责任。

这个方向正确，而且应当成为内容生产、陈列搭配及未来应用共同遵守的顶层原则。它解决的是：系统提供建议、素材、规则检查和效率工具，并不因此接管用户的经营决策与最终行为。

## 二、必须纠正的一个绝对表述

“不替客户监管”成立；“系统提供方没有任何义务”不能成立。

软件本身不是独立承担责任的行为主体，但 SaaS 提供方仍须对自己提供的部分负责，包括：

- 系统安全与可用性承诺；
- 租户、组织和工作区的权限隔离；
- 审计记录按约生成且不被任意覆盖；
- 受托数据处理与必要保护；
- ERP/POS 等连接器按照约定读取、映射和标识数据；
- 已明确承诺的规则检查、必要阻断和自愿反馈采集正确执行。

产品定位和免责声明不能排除适用法律的强制义务，也不能把系统自身缺陷产生的责任全部转嫁给使用者。

## 三、建议定稿的一句话

> **笛语负责赋能，不替客户监管；不接管各经营主体的事实确认、素材授权、专业判断、最终复核、制作、执行与发布责任。品牌企业、区域代理和加盟店等经营主体分别对自身输入、决定和行为负责；具体操作人与所属组织的内部责任由组织制度和协议确定，系统只作审计追溯。系统提供方对自身系统能力、安全、权限、审计、受托数据处理、明确承诺功能的正确执行以及适用法律规定的其他不可排除义务负责。**

这句话同时避免两个极端：既不让系统成为企业内容的最终审批人，也不允许系统提供方用“只是工具”否认自己的产品责任。

## 四、按具体行为理解

| 行为 | 主要责任主体 |
|---|---|
| 总部提供商品事实、品牌规则、活动素材和授权范围 | 品牌企业 |
| 区域代理补充区域事实、素材和交付内容 | 区域代理 |
| 加盟店补充本地事实/素材，自主修改、拍摄、剪辑和发布 | 加盟店及其最终复核人 |
| 员工在总部工作区按岗位人设生成内容 | 业务控制主体为品牌企业；实际操作人用于身份验证和审计追溯，内部责任由企业制度/协议确定 |
| 系统权限隔离、审计、数据安全和约定规则执行 | SaaS 提供方 |
| 最终内容选择、修改、制作和发布决定 | 对应发布经营主体 |
| 第三方平台最终是否审核放行 | 由第三方平台按其规则决定；系统和发布主体都不能保证结果 |

一个错误或经营后果可能由多段行为共同造成，不能强行只保留一个责任人。例如，总部提供了错误商品成分，门店未经复核又把它拍成视频发布：总部对错误源资料负责，门店对最终复核和发布行为负责；系统提供方是否有相应责任，要看它是否违反了明确承诺的检查合同或不可排除义务，而不是一概“有”或“一概没有”。

## 五、系统规则检查的定位

规则检查是辅助防错，不是监管接管：

- 系统发现明确越权、缺少强制证据或 A 级事实冲突时可以阻断；
- 系统对软性风格和经营建议可以提示；
- 系统只按当前已登记且仍有效的规则检查；
- 系统不得宣称“已经保证合法”“平台一定通过”或“发布方无需再审”；
- 用户完成最终复核并发布后，仍对自己的最终内容和行为负责。

因此，设置硬阻断与坚持“赋能而非监管”并不矛盾。硬阻断是产品安全边界，不是系统接管企业的日常审核岗位。

## 六、对两个应用场景的统一影响

### 内容生产

系统负责提供知识、生成草稿、事实提示、规则检查、版本链和自愿反馈入口；发布主体负责最终选用、修改、素材组合、成片制作、复核和发布。反馈截图不得成为继续生成的强制门槛，最新口径见 [ADR-011](../架构决策/ADR-011-发布反馈自愿提交与禁止生成门控.md)。

### 零售经营型陈列

系统负责提供培训、方案和库存/空间约束下的执行候选；门店或企业专业人员负责确认现场库存、空间、安全、人员条件并决定是否执行。系统建议不替代现场责任人签字或实际检查。

未来增加商品管理等模块时也沿用同一原则：系统扩大能力，不自动扩大为经营监管主体。
