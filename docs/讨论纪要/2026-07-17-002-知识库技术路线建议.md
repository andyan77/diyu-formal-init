# 2026-07-17 知识库技术路线建议

## 当前结论

本项目应当：

> 业务任务先行，轻量语义建模同步；关系型 SQL 作为权威事实底座，原始文档和媒体作为证据，全文与向量作为混合检索；知识图谱按复杂关系查询需要演进，GraphRAG 按全库综合分析需要实验性引入。

这不是“本体、知识图谱、SQL、向量、GraphRAG”五选一，因为它们解决的是不同层级的问题。

## 五个概念的准确边界

| 概念 | 本质 | 本项目中的位置 |
|---|---|---|
| 本体/语义模型 | 定义概念是什么、关系是什么意思、什么约束成立 | 必须有，但第一阶段只做任务驱动的最小语义模型 |
| 知识图谱 | 把具体对象和关系实例化为可遍历网络 | 可先由 SQL 实体表和关系表承载，不等于必须使用图数据库 |
| 关系型 SQL | 保存确定性的事实、权限、版本、有效期、状态、审批和审计 | 第一阶段权威事实底座 |
| 文档切块与向量索引 | 从长文、手册、案例和品牌表达中寻找语义相关证据 | 第一阶段检索能力；必须保留原文并结合关键词、元数据和重排 |
| GraphRAG | 从非结构化语料中由模型抽取实体、关系、主张和社区摘要，用于实体邻域或全库综合问答 | 派生检索索引，不是事实源；暂不进入第一阶段主链路 |

W3C 把本体描述为特定领域的形式化词汇及其关系；它是一种语义表达，不等于必须选 RDF、OWL 或图数据库作为第一阶段存储。Microsoft GraphRAG 的标准流程则明确包含模型抽取实体、关系和主张、社区发现以及模型生成的社区摘要，因此它更适合作为探索与综合分析索引，而不是商品、价格、库存和权限的权威数据源。

## “始于本体”应如何理解

### 应该做

从首批业务必须正确回答的问题反推：

1. 核心对象；
2. 术语和同义词；
3. 关系类型与方向；
4. 关系成立的条件和例外；
5. 来源、证据和责任人；
6. 租户、组织和公开范围；
7. 版本、生效、失效和审核状态。

这可以命名为“最小应用本体”或“轻量语义模型”。它应该从第一周就存在，并作为数据字典和接口语义合同持续演进。

### 不应该做

在内容产品、陈列任务和真实数据尚未验证前，先花数月建立全行业 OWL 本体、三元组库、推理机和庞大的风格/气质分类。这样很可能把主观判断过早固化成“领域真理”，最后得到图很完整、业务却不用的系统。

## 第一阶段推荐架构

1. **原始证据层**
   - 保留完整文档、表格、图片、视频、音频、检测报告和授权文件；
   - 切块、转写、摘要和向量都是可重建的派生索引，不能取代原件。
2. **SQL 权威事实层**
   - 保存租户、组织、商品、SPU/SKU、材料、版型、门店、区域、库存快照、价格、活动、规则、策略、授权、任务、内容、版本和审计；
   - 稳定且需要约束的字段使用正常关系表，未稳定的扩展属性可暂用 JSON 字段。
3. **轻量语义与关系层**
   - 保存概念、别名、关系类型、条件化陈述、证据链接和适用范围；
   - “适合”“可搭配”等关系必须带场景、气候、人群/穿着任务、例外、证据和审核状态，不能只是无条件边。
4. **混合检索层**
   - 先按租户、组织权限、有效期和审核状态硬过滤；
   - 再并行进行关键词和向量召回，随后重排并回到权威实体与原始证据；
   - 禁止先跨租户召回后再靠生成模型删除越权内容。
5. **受控 Agent 层**
   - 商品、价格、库存、权限、授权和合规先查权威数据并做确定性校验；
   - 品牌语气、方法解释和历史案例再调用文档检索；
   - 输出必须关联使用的事实、规则和证据。

第一阶段可用 PostgreSQL 一类关系数据库同时承载关系数据、JSON 扩展、全文搜索和向量，减少多套系统之间的一致性成本；是否最终使用 PostgreSQL 尚不需要在立项讨论阶段锁死。

## 典型问题与正确技术

| 典型问题 | 正确主路径 | 不应依赖 |
|---|---|---|
| 某 SKU 的面料、尺码、洗护、价格和当前库存是什么？ | SQL 或对应企业权威系统 | 向量检索、GraphRAG |
| 这条功能宣称由什么证据支持，谁有权批准？ | SQL 中的陈述、证据、责任与审批关系，再回到原文 | 仅靠相似文档 |
| 找符合品牌语气的历史内容或培训说明 | 权限过滤后的全文加向量混合检索 | 精确 SQL 单独完成 |
| 广州门店有货、适合湿热通勤、符合品牌规则并能搭配主推外套的商品有哪些？ | SQL 条件过滤、显式关系和规则校验 | GraphRAG 自动推断 |
| 一组商品为什么适合共同陈列？ | 商品/系列/色彩/库存/陈列约束加解释证据 | 仅用知识图谱或大模型生成 |
| 数百份门店反馈中有哪些跨区域共同主题与隐含冲突？ | 后期可比较普通混合检索与 GraphRAG | 实时事务 SQL 单独完成 |

陈列优化尤其不能被简化为知识图谱问题。货架容量、库存下限、邻接、色彩连续性、主推款曝光等属于规则和约束优化；图可以帮助表达关系和解释，但不能替代约束求解。

## 建议的最小语义对象

第一版只需要覆盖能支撑首批任务的核心对象，例如：

- 治理：租户、组织单元、适用范围、来源、证据、版本、策略、规则、授权；
- 服装：品类、SPU、SKU、材料、版型、属性、穿着任务、场合、气候、搭配、陈列区域；
- 内容：品牌角色、渠道、媒体资产、内容任务书、内容成品、案例、评测。

最低关系包括：SKU 属于 SPU、商品使用材料、商品具有版型、商品适用于穿着任务、商品可搭配/不宜搭配商品、商品适合陈列于区域、资产描绘商品/人物/门店、资产获授权用于渠道、策略适用于组织/角色/渠道、陈述由证据支持。

## 何时升级

### 形式化本体

当多个差异品牌和数据源反复出现同义、冲突与映射问题，核心词义已经稳定，并且确实需要机器推理、跨系统语义交换或形式化一致性验证时，再考虑 OWL/SHACL 等技术。

### 独立知识图谱或图数据库

当高价值任务持续需要不固定深度的关系遍历，SQL 出现大量脆弱递归与关系拼接，并且对照评测证明失败原因确实是关系检索而不是数据缺失、权限、规则或提示词问题时，将 SQL 权威数据投影为图。图首先是查询投影，不建立第二套人工维护的真相源。

### GraphRAG

当跨大量文档的全局主题、模式、冲突和网络分析成为高频、高价值任务，普通混合检索在固定评测中覆盖明显不足，并且项目能承担模型索引、实体消歧、增量更新和逐租户隔离成本时，再进行受控实验。

GraphRAG 若用于多租户，必须在实体抽取和社区摘要之前完成租户隔离。不能把多个品牌先混入一张图或同一批社区摘要，再在问答阶段过滤。

### 独立向量数据库

只有关系数据库内的向量能力经过真实并发、延迟和召回率测试证明不足时才引入；不能因为“专用向量库更先进”就提前增加一套基础设施。

## 路线验证方法

用同一批模拟品牌数据建立可比较基线：

- A：SQL + 关键词/向量混合检索；
- B：A + 最小语义模型、术语归一和显式条件关系；
- C：B + 从 SQL 投影的人工审核知识图谱；
- D：B + 按租户独立构建的 GraphRAG。

黄金问题至少覆盖精确事实、语义证据、多条件/多跳、全库综合和跨租户攻击五类。比较事实正确、证据可追溯、硬约束违规、租户泄漏、更新时效、延迟、索引成本和人工维护时间。

租户泄漏和硬约束违规是零容忍指标。只有 C 或 D 在反复发生的高价值问题上带来稳定、可复现且足以覆盖新增复杂度的提升，才进入生产路线。

## 外部依据

- [W3C OWL 2 概览](https://www.w3.org/TR/owl-overview/)：本体是领域术语及其关系的形式化表达，不等于数据库选型。
- [Microsoft GraphRAG 索引概览](https://microsoft.github.io/graphrag/index/overview/)：标准流程从原始文本抽取实体、关系和主张并生成社区摘要。
- [Microsoft GraphRAG 查询概览](https://microsoft.github.io/graphrag/query/overview/)：Global Search 用于数据集整体理解且资源消耗较高，Local Search 面向具体实体。
- [PostgreSQL 全文搜索](https://www.postgresql.org/docs/current/functions-textsearch.html)：关系数据库可以同时承担结构化查询和基础全文检索。
- [pgvector](https://github.com/pgvector/pgvector)：可在 PostgreSQL 中结合元数据过滤、全文与向量检索，适合作为早期验证方案。
