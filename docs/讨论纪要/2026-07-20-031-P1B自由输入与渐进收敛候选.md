## 一、用户核心观点

笛语应参考 ChatGPT、Claude 一类自然语言协作产品：用户不需要先理解系统产品分类、字段和任务书，可以自由输入。即使只输入一句 `hello world`，系统也应友好回应，而不是要求选择 P1—P5、填写表单或重新描述需求。

用户进一步明确：租户实际具有全国三级渠道时，其中的账号、人设和发声能力必须依据战略基线中的 IP 内容矩阵规划，不能由模型临时想象。总部、区域代理和终端门店各有自己的角色位置与能力边界，但角色层级不应被做成知识读取权限；不具备这些层级的租户不得被系统补造。领域通用知识默认供所有租户任务调用；进入某品牌专属共享知识层的数据默认供该品牌范围内所有有效账号调用，不再按总部、区域、终端分权。账号人设只决定生成时默认选择什么知识、从什么视角表达，不能把共享知识变成人设私有知识。P1—P5 是系统内部的内容成果合同和闭环逻辑，前台可以提供定向创作入口，但不得成为自由输入的必填项。

## 二、必要纠偏：输入自由不等于每句话都创建任务

“没有限制”在交互层应理解为：**不设置业务表达格式门槛**。用户可以输入文字、语音、图片、商品、评论、素材，或者接着上一轮说“就按这个做”；无需填写题材、人设、平台、时长和内容产品编号。

但它不能工程化为绝对无限：账号权限、租户隔离、支持的文件类型/大小及明确安全边界仍然存在；笛语也不因一句问候、闲聊或随口观察自动创建内容任务、写入长期记忆或启动 P1。

## 三、正式名称：自由输入、语境收敛、成果路由

通俗说，就是用户什么都可以自然地说；系统先像协作伙伴一样理解当前这句话，在需要形成成品时才悄悄收敛到正确的业务合同。

```text
一个自由输入入口
→ 本轮自然判断：回应交流 / 提出内容机会 / 执行成果
→ 只有执行成果时，按主要价值路由 P1—P5 或经营陈列
→ 自动组装必要上下文
→ 交付一个完整成品
```

“回应 / 建议 / 执行”只是本轮临时处理结果，不是用户模式、持久状态机、三套页面或三个 Agent。首版可以由同一个对话编排器完成，不建设独立意图分类服务、意图图谱、置信度阈值或长表单。

## 四、内容生产意图不是用户字段

用户不需要主动声明“我的内容生产意图已经成立”。系统根据当前对话和操作语境识别以下信号：

1. **明确要求成品**：如“写一条”“做成短视频”“直接给完整稿”；
2. **接受系统建议**：如“就做第一条”“按你推荐的来”；
3. **继续既有成果**：如“短一点”“换个开头”“接着写第五集”；
4. **进入明确创作语境并提交**：例如未来在创作/连续系列工作区选中商品、评论、选题或既有系列后输入；功能区只提供上下文，不限制用户怎样表达，也不要求选择 P1—P5。

满足任一项即可形成或继续正式任务，系统不得再问“你是想讨论还是生成内容”。没有这些信号时，系统可以自然回应、主动提炼内容机会或给一个有价值的方向，但不自动创建 `ContentItem`、系列前情或长期记忆。

## 五、P1-B 的三种当轮处理

### 5.1 普通交流：友好回应，不硬拉业务

用户输入：

> Hello world。

系统可以回答：

> 你好，我在。你可以随便聊，也可以丢给我一句今天的见闻、一件商品或一个门店观察，我会帮你看看它有没有值得做成内容的角度。

本轮不创建任务、不调用完整品牌知识、不要求用户选择产品类型。

### 5.2 模糊种子：主动给价值，不抢跑生成

用户在普通对话中输入：

> 这条裤子最近很多人拿起来看，最后又放下了。

它可能形成：

- 帮受众判断什么情况下值得选——P1；
- 解释版型取舍和不适合谁——P2；
- 从门店观察回应顾客犹豫——P4。

系统不向用户展示 P1/P2/P4 标签，而应结合当前品牌、账号和连续前情，先推荐最有价值的一个方向；只有几个方向确实会形成不同内容成果且无法合理择优时，才给不超过三个简短方向。建议尚不是正式任务，用户接受后再生成完整成品。

若用户已经处于明确创作语境，或同时说“直接做成内容”，内容生产意图已经成立，系统应直接择优完成，不再走一轮“要不要做”的确认。

### 5.3 正式任务：信息够就直接交付

用户输入：

> 给门店账号做一条短视频：明天见客户，胯宽，这条裤子能不能穿？直接出完整成品。

系统直接路由 P1，调用当前工作区、发布账号、内容角色、目标人群、品牌表达和已有商品资料，输出符合媒体合同的完整成品；不展示意图识别、P1 标签、检索过程和检查报告。

只有用户明确要求判断某件具体商品，而当前没有选中对象、图片或任何可用商品资料时，才问一个最小问题：

> 请选中具体商品或提供正反面资料，我再把判断落到这件商品上。

不得扩展成长资料清单。

## 六、按成品主要价值路由，不按关键词路由

同样出现“裤子”，路由仍可能不同：

| 用户真正要获得的成果 | 路由 |
|---|---|
| 目标受众怎样选、何时不该选 | P1 |
| 商品为什么这样设计、解决什么又牺牲什么 | P2 |
| 借这件事表达账号怎样看生活 | P3 |
| 回应评论、门店观察或本地关系 | P4 |
| 用真实商品形成造型与视觉欲望 | P5 |

因此，“为什么这件风衣没有做得更修身？”应进入 P2；“我明天见客户，这件宽松风衣该不该选？”才进入 P1。用户不需要知道这些编号。

## 七、品牌关联由系统自动补齐，但必须真实改变成品

用户不需要每次重新输入品牌、目标人群、人设和表达风格。系统从当前租户、工作区、发布账号、内容角色、已选对象、当前系列和必要私人协作偏好中选择性调用；不能为了一个 `hello world` 全量检索企业知识。

一个正式 P1 的品牌关联至少应实际改变以下一项：

- 选择依据；
- 条件优先顺序；
- 反选标准；
- 对目标人群的理解；
- 专业判断方法；
- 账号角色的表达视角。

只写品牌名、放 Logo、结尾加“欢迎到店”或强插商品均不合格。没有具体商品时，P1 仍可通过品牌目标人群理解、审美/版型判断方法或门店服务经验成立；找不到自然品牌关联时，不把通用建议伪装成合格品牌 P1。

## 八、实际组织与角色方案先于账号人设

正式名称是：**实际组织与角色权威约束下的自由创作路由**。

通俗说，就是用户可以随意表达，也可以直接指定想做哪类内容；但系统不能临时编一个“听起来合适”的人设。租户实际存在总部、代理或终端时，账号不能假装成另一个层级发声；单店租户则不得被补造这些层级。

### 8.1 只采用企业确认的真实角色架构，不任意扩写

“一核、三层、五类角色资产”只作为当前 D-019 试点和“折线之间”三级渠道实例的参考结构：

| 层级 | 在 IP 矩阵中的核心动作 | 合法角色与内容权威的主要来源 |
|---|---|---|
| 品牌总部 | **定义** | 品牌母账号、品牌价值角色、设计/商品等专业权威角色，以及作为终端角色母模型的零售/服务专业角色 |
| 省级代理 | **翻译** | 区域经营角色，把总部母题和品牌核心转译为当地气候、城市、商圈、消费节奏和加盟经营语言 |
| 加盟或直营终端 | **证明** | 店长、导购、搭配师、门店团队等终端关系角色，以真实门店人物、商品、服务和一线问题形成在地表达 |

商品、系列、栏目和服务仪式可以成为非人物符号资产，但不能被当作一个自然人的随意人设。总部第一阶段的“零售/服务专家”是专业方法的母模型，不等于总部账号可以冒充任意门店店长或声称自己经历了某家店的现场事件。

本文只继承上述组织型 IP 与角色矩阵基线。附件中关于阶段排期、逐级审核、权利流程或其他治理建议，不因本次引用自动升级为笛语首版决定；加盟店自主生产、私人素材隔离、不接 ERP 等现有正式裁决继续有效。

### 8.2 五个维度必须分开

| 维度 | 回答的问题 |
|---|---|
| 组织节点与渠道层级 | 这条内容属于总部、某区域，还是某个加盟/直营终端 |
| IP 矩阵角色 | 它以品牌价值、专业权威、区域经营还是终端关系身份发声 |
| 发布账号 | 最终由哪个业务账号承载和持续经营 |
| 发声与业务能力边界 | 该角色可以解释、主张和生产什么，哪些内容不得冒充 |
| 本次内容目标 | 选择 P1—P5 中哪个主要成果，以及什么题材、平台和内容形式 |

实际操作人的私人协作偏好只回答“系统怎样更好地协助这个人”，不属于以上业务角色矩阵；目标人群回答“内容服务谁”，也不等于操作者画像或账号人设。

### 8.3 一条最小硬规则

每个用于生产和发布内容的账号，必须绑定：

```text
一个组织节点
+ 一个经该组织授权的矩阵角色
+ 一份相应的发声/业务能力边界
```

由此得到四条直接规则：

1. 区域代理和终端账号默认可以调用当前品牌共享知识，无需为同一条商品知识再配置总部、区域、终端三级读取权限；但它们不能创建或冒充总部品牌价值角色、设计师等总部权威角色；
2. 终端门店可以在终端角色框架内策划本地人设和内容，但不能获得区域经营或总部权威账号的发声能力；
3. 总部若新增一家直营终端，应先新增该直营门店的组织节点，再在终端矩阵下建立店长、导购或搭配师等角色账号；不能拿总部设计、品控或品牌价值账号代替终端账号；
4. 共享知识可用不等于共享身份。总部设计或商品知识可以支持区域、门店成品，但最终账号不能因此自称设计师；总部专家账号也不能把门店观察、顾客服务或当地生活写成自己的第一人称经历。

这里还要避免反向过度限制：**“总部定义、区域翻译、终端证明”约束的是权威来源、第一人称身份和结论适用范围，不是 P1—P5 的层级独占表。** 总部、区域和终端都可以按自身角色生产 P1—P5；区别在于以什么身份说、哪些内容可以说成亲历或决策、结论可以覆盖多大范围。

如果用户请求超出当前账号角色边界，系统不能悄悄换身份完成。它只需给出一个最短可行动选择：切换到本人有权使用的正确账号/角色，或者把任务改写成当前角色可以合法表达的角度；不建立逐级审批流。

### 8.4 修正此前错误示例

此前“总部员工小李运营企业抖音账号，以‘懂 35+ 通勤的品牌店长’人设发声”的例子无效，现正式删除。它把实际操作人、总部发布账号、终端店长角色和目标人群拼成了一个模型臆造人设，违反 IP 矩阵。

正确例子是：

- 总部员工可以被授权运营总部“设计/商品专家”账号，围绕设计逻辑、版型取舍和适用边界生产内容；员工的私人协作偏好仍只属于本人；
- 总部若拥有杭州某直营店，可以在该直营店组织节点下建立“门店店长/搭配师”角色账号，生产真实的终端内容；总部设计知识可以作为背景支持，但发声身份仍是该门店角色；
- 加盟店账号输入“这件落肩大衣为什么这样做？直接给一条内容”，即使没选 P2，系统也应在后台路由 P2。它可以引用总部商品资料并加入本店真实试穿观察，但不能写“我们设计这款时决定”，也不能把一家店的观察说成全国事实。

### 8.5 P1—P5 是后台合同，也是前台可选捷径，但不是输入门槛

这里需要纠正一个概念混合：**P1—P5 是内容成果目标，题材是讲什么，内容形式是用短视频、图文或其他媒介怎样呈现；三者不是同一个维度。**

交互 UI 可以同时支持两条路径：

1. **随心输入**：用户直接说话、上传素材或继续上一轮，系统在后台自动判断主要内容产品；
2. **定向创作**：用户可选“帮顾客做选择、讲清商品取舍、经营观点与连续叙事、回应一线与本地生活、造型与视觉表达”等业务语言入口，再按需指定题材、平台、时长或内容形式。

前台不必显示 P1—P5 编号，也不得强迫用户先选。用户选择某类成果，只是向后台路由器给出明确意图，不会覆盖当前发布账号的组织层级和角色边界。用户没有选择时系统自动判断；选择与角色冲突时，系统按第 8.3 节给出一次有效改路，而不是生成越权人设内容。当前明确指令优先于旧的 UI 选择或历史默认。

首版不为三级渠道或五类产品各建一套 Agent、能力图谱或复杂规则引擎。只需由同一个编排器在生成前读取“当前组织节点 + 发布账号 + 已授权矩阵角色 + 简明能力边界”，再选择 P1—P5 合同和所需共享能力。对象怎样落库继续由 D-016 决定，平台/时长/媒体成品怎样具体呈现继续由 D-032 决定。

### 8.6 知识可用域与人设编排域分离

正式名称是：**知识默认可用，人设按需编排**。

通俗说，就是知识库回答“系统知道什么”，账号人设回答“这个账号通常应该选什么、从哪个角度说”；不能把两者做成同一套权限。

| 对象 | 当前规则 |
|---|---|
| 领域通用知识 | 默认供所有租户在对话与生成中调用；平台统一维护，不包含任何租户私有数据 |
| 品牌共享知识 | 已正式进入当前品牌专属共享知识层的品牌定位、商品资料、设计解释、服务方法和表达规则，默认供该品牌范围内所有有效账号调用；不再按总部、区域、终端配置知识读取权限 |
| 账号人设 | 决定默认关注主题、选材优先级、解释视角、语气和身份/权威边界；不是知识访问控制器 |
| 特殊能力或数据模型 | 可以由平台按租户/品牌做简单开关配置；它属于产品能力配置，不改变通用知识和品牌知识的上述默认规则，也不建设逐知识、逐角色策略引擎 |

这里的“开放”必须精确解释为：**可在当前任务中检索、问答和生成使用**，不等于所有账号都能打开数据库、批量浏览、下载、修改或管理知识原件。品牌知识维护、版本启停和原始资料管理仍由相应管理职责控制；普通账号能够使用共享后的知识，不因此取得维护权。

“品牌专属数据库对所有账号开放”还必须受品牌范围约束。单品牌试点中就是该品牌合作网络的全部有效账号；一个企业租户经营多个品牌时，当前品牌 A 的任务默认使用企业明确共享基线和品牌 A 知识，不能因为同属一个租户就自动混入品牌 B 知识。明确的多品牌/集合店任务才组合多个已选品牌。

品牌共享知识层也不等于品牌全部原始资料。总部未整理的设计草稿、内部会议文件、成本与供应商资料、区域/门店未共享记录、个人私有素材以及系统未经确认的推断，继续留在原有资料空间，不因同属一个品牌而被所有账号调用。这里不要求新增第三座“品牌共享库”：**本项目所称品牌专属数据库，本身就应理解为已经整理、可供该品牌账号共同使用的正式知识；原始文件和私有素材继续使用既有资料/素材空间。** 最简工程边界只是这两者语义分开，不给每条共享知识再增加三级角色权限。

人设对知识的作用分成两层：

1. **默认选材是软规则**：样衣工账号默认优先调用版型、结构、工艺和打样知识，不会因为数据库里有陈列知识就主动把内容写成陈列培训；
2. **身份与权威是硬边界**：无论调用什么知识，都不能把共享知识说成该账号亲历、亲自设计或拥有的组织权威。

必要纠偏是：**“默认不编排”不能等于“用户明确要求后仍永远禁止调用”。** 在当前租户、品牌和原始资料边界内，本次明确任务应优先于人设的默认选材，但不能改变人设身份和权威边界。否则自由输入和多样化需求会被人设默认值反向锁死。

例如样衣工账号日常生成时不会主动调用陈列知识；但用户明确说“从样衣结构角度讲讲为什么这件西装挂在模特上不能压塌肩线”时，系统可以调用陈列通用知识和品牌商品知识，仍以样衣工视角说明结构影响。若用户要求该账号“代表总部发布全国门店陈列标准”，系统则应建议切换到相应陈列/零售专业角色，不能把样衣工人设升级成陈列权威。

首版运行优先关系保持极简：

```text
租户隔离 + 当前品牌范围 + 原始资料/私人素材边界
→ 用户本次明确任务
→ 当前内容产品
→ 账号人设的默认选材与表达偏好
→ 始终生效的人设身份与权威边界
```

### 8.7 三级配置分层，由一次运行编排合成

正式名称是：**系统基线、品牌叠加、人设叠加的三级编排合同**。

通俗说，就是系统先决定“平台会什么、这是什么任务”，品牌再决定“本品牌怎样做”，账号人设最后决定“由谁、从什么视角说”；三层分别维护，但生成一次内容时只合成为一份执行计划。

| 编排范围 | 负责什么 | 不能做什么 |
|---|---|---|
| 系统全局编排 | 应用模块与 P1—P5 合同、领域通用知识、行业角色模板、通用共享能力、租户隔离及平台级能力目录 | 不写入任何具体品牌的人设、声纹和经营策略 |
| 品牌租户全局编排 | 品牌 IP 矩阵、品牌表达与目标人群、账号框架、内容组合重点、品牌专属知识、特殊能力/数据模型开关 | 不修改系统产品合同，不把一个品牌配置扩散到其他品牌或租户 |
| 账号/人设编排 | 当前账号的渠道层级、矩阵角色、默认主题、选材优先级、视角、语气、连续定位和身份/权威边界 | 不取得额外知识权限，不覆盖系统/品牌硬边界，不因一次任务自动改写长期人设 |

当前用户任务不是第四套持久路由层，而是本次执行输入：它可以改变品牌和人设的软默认选材，不能修改三层已经确认的硬边界。

“解耦”不应被工程化为三个 Agent、三套服务或三个依次调用的大模型。首版只需要三份各自负责、可独立更新的配置，由同一个上下文编译/任务编排过程合成：

```text
系统全局基线
+ 当前品牌租户叠加
+ 当前账号/人设叠加
+ 用户本次任务
→ 一份执行计划与一个主要内容产品合同
```

“前后项兼容”至少保证：系统新增可选能力不会自动改变既有品牌和人设；平台行业人设模板更新不会静默覆盖企业已经定制并启用的人设；企业修改人设采用新版本，历史内容仍能追溯生成时所用版本。具体版本字段和存储方式留给 D-016，不在 P1-B 预建配置中心或规则引擎。

### 8.8 入驻后按真实组织形成最小账号与角色方案

租户完成 D-014 的品牌或企业/店铺表达基线后，再进入最小账号与角色方案策划，而不是直接面对一批空白发布账号。系统提供服装行业通用岗位/人设模板作为起点，企业可以删改、组合或新增符合自身真实组织与内容战略的角色；模板只是建议和初始化材料，不能自动成为该企业真实人设。

为避免规划过度，首期只要求实际运行所需的最小方案；多个组织或账号存在协同时可以形成 IP 矩阵，单店独立租户也可以只有一个真实组织和首批必要账号：

1. 先确定当前主要品牌或企业/店铺表达身份，以及真实存在的组织和经营关系；
2. 为首批准备启用的账号确认组织节点、矩阵角色、人设属性和身份/权威边界；
3. 系统可在策划阶段继续与企业讨论、生成矩阵草案，但未绑定有效人设的账号不得冒充正式品牌账号生产对外成品；
4. D-019 试点只激活总部、一个区域代理和 3—5 家终端所需账号；其他租户按自身实际拓扑激活，不为未来层级预建空组织、空账号或空流程。

行业模板与企业定制采用“来源模板 + 企业确认版本”的语义即可：模板未来升级只能提示企业比较和选择采纳，不能自动改写已经运行的品牌人设。是否复制、继承或保存差异继续由 D-016 决定。

## 九、P1-B 最小验收规则

1. 用户可以自由输入，不填写 P1 字段、不选择 P1—P5；
2. `hello world` 得到友好回应，不创建业务任务；
3. 模糊种子得到自然理解和主动内容机会，不只得到“请补充需求”；
4. 用户明确要求、接受建议或继续既有成果时，信息足够就直接执行，不重复确认；
5. 默认只交付一个最佳成品；真正分叉时最多先给三个简短方向，不默认生成多篇完整稿；
6. 能用条件化表达或省略解决的缺口不追问；只有缺少不可替代信息时问一个问题；
7. 产品路由按成品主要价值判断，不按“裤子、商品、门店”等关键词判断；
8. 品牌和账号上下文由系统自动调用，品牌关联必须实质改变选择判断或表达视角；
9. 系统不向用户展示意图分类、上下文编译、评分或后台轨迹；P1—P5 编号可以不显示；
10. 私人协作偏好、发布账号策略、矩阵角色、目标人群和权限不得合并；
11. 每个业务发布账号必须绑定明确的组织节点、矩阵角色和发声/业务能力边界；模型不得在生成时临时发明有效人设；
12. 领域通用知识默认供所有租户任务调用；进入当前品牌共享知识层的数据默认供该品牌全部有效账号调用，不按总部、区域、终端设置知识读取权限；
13. “总部定义、区域翻译、终端证明”不形成 P1—P5 的层级独占表；三个层级均可在自身角色与权威范围内生产 P1—P5；
14. UI 可以提供定向选择某类内容成果、题材和内容形式的功能区，同时必须保留无需选择 P1—P5 的自由输入；
15. “知识开放”只表示可供当前任务检索、问答和生成，不自动授予原始资料浏览、下载、编辑、启停或跨品牌调用能力；
16. 账号人设决定默认选材、视角和表达，不充当知识访问控制器；用户明确任务可以改变默认选材，但不能改变人设身份、权威和事实边界；
17. 用户指定的成果与当前角色身份/权威不兼容时，系统只建议切换到正确账号/角色或改写为当前角色可表达的角度，不静默越权，也不新增审批流程；
18. 第一版不为共享知识增加逐账号、逐角色、逐层级权限标签，不建设知识权限图谱；只保留租户/品牌边界、共享知识与原始资料隔离及简单租户能力开关。
19. 系统全局、品牌租户全局和账号/人设是三层独立配置范围，由同一个运行时编排过程合成为一份执行计划，不建设三套 Agent 或物理路由服务；
20. 每个租户按真实组织完成最小账号与角色方案；存在多组织/多账号协同时形成相应 IP 矩阵，单店租户不被强制补齐三级渠道。系统行业人设模板只提供起点，企业确认版本才可用于正式内容生成；
21. 行业模板更新、系统新增能力和品牌/人设新版本不得静默改变既有配置或历史成品；具体版本物理结构后置到 D-016；
22. 首期只激活试点范围所需账号，不要求一次性创建全国全部账号，也不因未来扩展预建空账号或空流程。

## 十、P1-B 正式裁决（2026-07-20）

用户确认 V0.3 的其他表述，并补充三级编排与品牌入驻规则。P1-B 由此正式关闭：

> **笛语允许自由输入和可选定向创作，执行时由系统全局基线、当前品牌租户叠加、当前账号/人设叠加和用户本次任务共同合成一份执行计划。领域通用知识默认供所有租户任务调用，品牌专属正式知识默认供该品牌全部有效账号调用，不按总部、区域、终端分权；人设决定默认选材、视角和表达，而不是知识权限。用户明确任务可以改变软默认，但不能跨品牌、穿透原始资料/私人素材边界或改变账号身份与权威。每个品牌入驻先依据系统行业角色模板规划最小可用 IP 矩阵，再由企业定制、确认和逐步激活账号。三级配置独立维护并保持版本兼容，但首版由同一个编排器合成，不建设三套 Agent、服务或复杂规则引擎。**

P1-B 的确认不冻结 UI 页面、对象名称、数据库表、配置格式或模型/SDK 选型；这些分别继续由交互工作台设计、D-016 和后续实现阶段决定。

下一项是 **P1-C“专属合同收敛”**——通俗说，就是不再讨论用户怎样输入或账号怎样路由，而只确认：一份“帮助目标人群作出穿衣选择”的 P1 成品，最低必须给用户什么判断、行动和完整内容交付，哪些内容绝对不能猜造。
