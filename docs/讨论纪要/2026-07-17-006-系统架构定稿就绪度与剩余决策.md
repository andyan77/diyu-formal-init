# 2026-07-17 系统架构定稿就绪度与剩余决策

## 一、结论

**现在可以形成无歧义的“笛语系统逻辑架构基线”，但还不能诚实地称为“首个试点可直接施工的最终生产架构”。**

这不是架构本身尚未成形，而是两类变量必须显式填入：

1. 会改变首期权限、流程和验收的三个业务决策包；
2. 会改变物理部署、安全和容量的试点部署配置档案。

“无歧义”不等于对所有未来客户只能有一张物理部署图。正确的架构文档应当明确区分：不变架构、可配置档位、试点决策门与未来扩展点。把未知参数偷填为唯一答案，才是伪“无歧义”。

## 二、现在已可冻结的不变架构

| 架构面 | 已可冻结的结论 |
|---|---|
| 知识路线 | 业务任务先行；15—25 类轻量语义模型；SQL 为权威底座；原始证据/媒体独立保存；全文与向量为可重建索引；图与 GraphRAG 按评测后置。 |
| 数据分层 | 通用领域知识、企业/品牌私有知识、媒体与证据、动态经营上下文、个人/敏感数据、任务运行与审计分层治理。 |
| 多租户与组织 | 租户、法律/组织主体、品牌是不同维度；同一组织、人员和门店可服务多品牌；总部/区域/门店使用一套租户数据底座中的逻辑工作区，不复制物理数据库。 |
| 双应用语义 | `content-production` 使用内容表达型搭配/陈列；`display-merchandising` 使用零售经营型陈列和私有 `DisplayArtifact`；两者分开任务、产物、状态、真实性和验收。 |
| 模块扩展 | 稳定共享核心 + 可注册应用模块 + 版本化合同；第一版只实现两个当前模块；未来模块未立项前不建空壳。 |
| 内容真实性 | 只对可验证主张分级绑定证据；创意结构、节奏、镜头、比喻等在不伪造事实时保持自由；授权、来源、失效和撤回可追溯。 |
| 库存边界 | 冻结供应商无关的 `InventorySnapshot` 输入与降级合同；库存不进入长期知识库。外部系统按需接入，具体源系统口径、时效和读取方式仅在租户选择连接器时配置。 |

## 三、三个仍需领域确认的业务决策包

### A. 第一家试点企业里，谁能看、谁能用、谁负责

> “试点与数据权利边界”仅作为内部架构索引语；与领域专家确认时，按具体人物、动作和门店场景逐条说明。

需确认：

1. 首期签约租户和数据控制主体是谁；
2. 区域代理、加盟商是租户内组织作用域、关联租户，还是受限外部协作者；
3. 门店事实与素材默认能否向区域/总部复用，能否横向给其他门店复用，由谁授权和撤回；
4. 谁对商品事实、功能宣称、人物素材、内容审批与发布负责；
5. 首期是否正式排除纯散货、买手/集合店与门店自行采集儿童真人素材。

架构默认建议：首个试点选择稳定自有/完整授权品牌、有商品主数据与总部—区域—加盟网络的企业；跨租户和横向复用默认禁止，向上复用需显式授权。

### B. 内容生产首期合同

需确认：

1. 首期验收的 2—3 类内容产品；
2. 首期验收的两类媒体形式与目标平台；
3. 每类内容产品的输入、输出、合格/不合格样例与评分标准；
4. 总部、区域、门店分别能创建、改写、审核和发布什么；
5. 首期“本地生活”是否确认只做本地语境、门店事实与到店引导，不进入团购、预约、支付、核销和广告归因。

这一包不能用“全媒体都支持”代替。如果没有具体内容产品与成品验收样例，系统只能证明会生成，无法证明对企业有用。

### C. 陈列搭配首期深度

需确认：

1. 首期只做培训指南与概念案，还是包含门店执行方案；
2. 是否进入巡检、整改和复核闭环；
3. 首期选择哪一种货架、橱窗或陈列区域；
4. 空间、安全、品牌、商品和库存中哪些是不可违反的硬约束；
5. 哪些方案下发、异常判断和完成状态必须由陈列专业人员签字。

架构默认建议：以一个标准货架或橱窗、一次活动和少量门店验证“指南→概念案→执行方案→照片预检→人工确认/整改”最小闭环，不承诺完全替代陈列岗位。

## 四、试点企业需提供的部署配置档案

这些不是领域本体问题，也不应由领域专家凭空回答。下表前四项属于生产部署配置；最后一项只有租户主动选择外部系统连接器时才需要：

| 配置门 | 必须提供的信息 | 主要责任人 |
|---|---|---|
| 数据与部署地域 | 是否要求指定地域、专有实例/私有化，品牌资料和门店媒体是否允许发送给外部模型服务 | 企业 IT、安全、法务/采购 |
| 隔离档位 | 共享 SaaS 逻辑强隔离、独立数据库/密钥，还是专有实例 | 业务、安全、采购 |
| 两年规模量级 | 租户、品牌、门店、并发用户、每月图片/视频与生成任务的数量级 | 产品负责人、渠道运营 |
| 服务与灾备 | 中断时是否允许使用最近有效结果，是否存在合同级可用性、恢复时间和数据丢失容忍要求 | 产品负责人、企业 IT |
| 可选连接器配置（仅在选择接入时） | 源系统名称、部署位置、可用的文件/API/Webhook/VPN 等读取方式，以及字段口径的业务责任人 | 源系统负责人、相应业务专家 |

## 五、架构师直接负责的默认决策

下列选择不应继续上抛给领域专家，在没有试点非功能证据推翻时，作为第一版架构默认值：

1. 采用**模块化单体**，内容、陈列与共享核心保持代码/数据所有权边界，不起步即拆成微服务群。
2. 使用关系型 SQL 数据库保存权威事实、权限、版本、状态和审计；对象存储保存原始文件与媒体；全文/向量索引为派生数据。
3. 任何 SQL、对象存储、缓存、队列、作业、索引、导出、日志和备份都必须携带租户/作用域；检索必须先做权限过滤，不能检索后再删结果。
4. 外部企业系统首期只读、单向、可审计；Agent 不持有数据库或 ERP/POS 凭据。
5. 模型调用通过统一模型网关执行权限、脱敏、成本、审计与供应商路由；原始提示、检索片段、门店照片与库存默认不进入公共训练/跨租户评测集。
6. 跨租户学习默认关闭；首期不自动发布，不预建图数据库、GraphRAG、未来模块、多区域多活或全量流式库存。

## 六、D-023 的最终处理

D-023 已关闭为供应商无关的最小合同：

`tenant + organization + store/location + SKU + mapping version + inventory measure + quantity + source + source version + as-of + received-at + expires-at + data quality`

并强制：

- 缺少某个库存状态表示“未知”，不是 0；
- 库存过期、SKU 映射失败或关键口径未知时，方案降级为模拟/不可执行，或要求人工复核；
- 执行方案绑定库存快照、空间版本、规则版本和有效期；
- 具体任务必须声明所用的在手、可售、预留、残次、在途或陈列样口径；某租户选择外部系统连接器时，再由该租户的源系统资料和库存业务人员完成字段映射与更新频率配置。

## 七、文档定稿路径

1. **现在**：将已确认内容编入《笛语系统逻辑架构基线 v0.1》；对三个业务决策包和部署配置门使用明确编号，不用模糊文字掩盖。

3. **试点配置档案到位后**：冻结《试点生产物理架构与安全部署档案 v1.0》，开始施工级估算与计划。
