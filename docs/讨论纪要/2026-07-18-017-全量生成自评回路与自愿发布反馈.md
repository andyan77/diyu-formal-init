# 2026-07-18 全量生成自评回路与自愿发布反馈

## 一、用户本轮核心观点

1. 系统生产的全部内容，无论使用者最终是否采用，都必须留痕并进行有价值的分析；这也是系统自我评估和未来迭代进化的回路。
2. 该机器回路不意味着品牌企业可以查看具体生成内容；企业无权因数据飞轮或系统自评打开这些内容。
3. 完整的分析与自我迭代能力放在系统稳定发展阶段工程化实现，当前阶段只需保留接口。
4. 系统可以提倡用户上传此前发布内容的平台截图，但不能强制；如果过去关于强制反馈门的裁决与本次冲突，以本次为准。

## 二、必须执行的工程纠偏

### 1. 不能只留空接口，必须从第一版开始积累真实事件

如果第一版只定义一个未来接口而不记录生成事件，成长阶段无法回溯未被采用的内容，也就没有全量评估样本。因此第一版必须真实保存每次生成运行和输出版本的最小事件链；成长阶段才新增分析消费者、评测流水线和候选改进机制。这是“当前不做完整飞轮”与“未来有历史数据可用”之间的最小可行边界，不是提前建设数据飞轮平台。

### 2. “系统自我进化”不能等于自主改写生产系统

全部生成记录可以发现失败模式、质量漂移和采用规律，但不能直接修改生产提示、规则、路由或模型。未来任何改进都必须先生成候选版本，经隔离评测、回归测试和人工发布审批后上线。

### 3. 本轮没有授权跨租户训练

系统机器分析与企业人工查看分离，并不自动解决跨品牌、跨租户复用问题。D-010 未确认前，具体内容、派生特征和模型训练都不能跨品牌/跨租户；生成内容也不能自动成为事实知识或品牌规则。

### 4. “全部留痕”不等于无限期保存全部正文

所有生成运行必须有事件、版本和结果状态，但完整正文、机器分析样本和最小审计摘要应分别设置保留期。到期可删除或匿名化正文，同时按政策保留不可还原的编号、哈希、版本谱系和聚合评价；不能把无限期保存敏感数据写成默认承诺。

### 5. 本轮“内容”不能顺带覆盖经营陈列产物

本轮上下文来自内容生产和发布截图，因此 D-028 先覆盖 `ContentItem` 及其版本。陈列模块的 `DisplayArtifact` 同样需要事件和审计留痕，但它要用库存有效性、空间约束、执行偏差和复核结果等不同指标，不能把播放量、采用率等内容指标直接套过去。

## 三、第一版与稳定发展阶段的明确分工

| 第一版必须真实完成 | 稳定发展阶段再实现 |
|---|---|
| 每次生成、失败、编辑、采用、放弃、导出、发布和自愿反馈的版本化事件 | 全量质量分析、失败聚类和趋势评估 |
| 模型、提示模板、知识/规则版本及输入来源引用/哈希 | 候选提示、路由、规则和模型改进方案 |
| 输出版本、规则检查、错误和最小审计链 | 隔离评测、回归验证和受控发布流水线 |
| 工作区权限隔离、保留策略和机器处理合同 | 经单独授权的更高级学习方式 |

第一版不记录模型隐藏推理，不把私人原件复制到分析库，也不建设模型微调、自修改代理或空分析后台。

## 四、企业“不能查看”的准确含义

- 总部、区域和租户管理员不能借系统自评回路打开其他工作区的生成正文、原始提示、反馈截图或私人素材。
- 企业仍可按正常业务权限查看自己总部组织工作区中的岗位工作成果；这来自组织工作区的业务所有权，不是数据飞轮权。
- 使用者仍可按本人和所在工作区权限使用自己的内容。
- 企业管理员仍可查看账号状态、安全事件和运行所需的最小元数据，但这不包括跨工作区具体正文。
- 系统运营人员也没有日常人工查看权；故障应急访问必须限时、最小化、审批并留痕。

## 五、反馈截图的最新裁决

此前“可配置开启强制上传上一条截图，否则不能继续”的方案已经废止。现行规则是：

1. 系统可在内容被标记已发布后温和提醒上传平台数据和评论截图；用户可以跳过、关闭或稍后上传。
2. 未上传不能阻断生成、修改、定稿、导出或后续任务，不能形成反馈债务、违规记录或管理员例外审批。
3. 用户自愿上传时，截图仍需关联内容版本、平台、发布账号、发布时间和截取时间，并默认脱敏、标记为用户自报证据。
4. 没有上传反馈不能被当作负向样本；主动上传的数据存在自选择偏差，机器评估必须结合全量生成、采用、修改、明确拒绝和发布信号。单纯未采用或用户离开只能记为结果未知。
