# D-016：两条黄金路径关系校验与关闭

- 日期：2026-07-22
- 版本：V1.0-confirmed
- 状态：**用户已确认 D-016-A V0.2；两条既有黄金路径完成最小关系校验，四处残余归属歧义已在不新增类别的前提下修复；D-016 由 [ADR-019](../架构决策/ADR-019-第一版25类最小语义模型与兼容扩展边界.md)正式关闭**
- 语义母稿：[D-016 第一版最小语义对象](2026-07-22-079-D016最小语义对象候选.md)

## 一、本次只验证什么

只复用两条已经存在的单品牌黄金路径，验证 25 类及 R1—R10 能否无歧义表达：

1. 一个总部品牌账号怎样基于 P3 任务形成内容成品；
2. 一个加盟店怎样根据本次商品清单形成并局部修订墙面挂杆成果。

不新增样例、评分器、字段或全量 ERD；不验证数据库实现、DIFY 工作流、真实企业部署、多品牌真实运行或未来全部横向扩展。

## 二、校验发现并修复的四处残余歧义

| 歧义 | 最小修复 | 是否新增类别 |
|---|---|---|
| `ContentRole` 归谁控制、能否跨品牌适用未明确 | 每个 `ContentRole` 恰由一个 `OrganizationalParty` 控制；它同时属于一个租户。品牌不是所有者，而是独立适用范围，可为企业共同表达或一个/多个明确品牌 | 否 |
| `PublishingAccount` 的组织控制关系未单列 | 每个 `PublishingAccount` 恰由一个 `OrganizationalParty` 业务控制；多人运营通过 `AuthorizationGrant` 表达，授权不改变账号控制方 | 否 |
| 无成果或失败运行时，`BusinessTask` 无法从成果反推所属组织 | 每个 `BusinessTask` 恰有一个发起/控制 `OrganizationalParty` 和一个顶层应用；各成果再保存其代表或执行组织 | 否 |
| `Product` 只能依赖文件语境判断租户、目录控制方及品牌关系 | 稳定商品恰属一个 `Tenant` 并由一个 `OrganizationalParty` 控制其目录记录；品牌关联必须显式，但为兼容无稳定品牌散货与真实联名/集合商品，不强制恰为一个。当前 ZX 商品均明确关联“折线之间”品牌；品牌关联不等于取得独立内容人格，D-021 不变 | 否 |

这些是已确认 25 类之间的关系补齐，不是第五次扩类。`ContentRole`、`PublishingAccount` 和 `Product` 的稳定归属不能依赖某篇成品反推，否则没有成品的任务、账号交接和商品检索都会产生两种解释。兼容性复核另外明确：`BusinessTask` 必须保存企业共同表达、单品牌或显式品牌集合之一，内容成果绑定“当前已注册的一个主要产品合同”，第一版注册集才是 P1—P5；这两项只消除未来扩展的文字锁死，不新增对象或运行流程。

## 三、内容生产黄金路径

### 夹具

使用[P3-D 总部账号离线替换成品《别人只说了四件事，我们别替她补第五件》](2026-07-21-045-P3D两包三版候选黄金成稿比较稿.md)。该夹具明确把发布身份换为“折线之间品牌母账号 + 总部零售/服务专家”，并明确本次生活片段只是离线测试输入，不表示真实跨工作区读取。

### 关系闭合

```text
折线之间租户
  → 折线之间品牌总部（任务与账号控制组织）
  → 折线之间品牌/当时有效品牌战略基线/目标人群
  → 品牌母账号 PublishingAccount
  → 总部零售/服务专家 ContentRole
  → content-production BusinessTask + P3 主合同
  → ContentItem 当前修订
  → GenerationRun / ActivityEvent
```

- `ContentRole` 和 `PublishingAccount` 都由品牌总部控制，但不是同一个对象，也不与实际操作人合并；
- 用户提供的合成生活片段只留在 `BusinessTask`，不自动创建 `MaterialAsset`、`KnowledgeStatement` 或账号长期履历；
- 该 P3 成品可以有零个 `Product`、零个 `EvidenceRecord`、零个 `ContentSeries`；内容仍须绑定任务、有效品牌表达、代表组织、唯一发布账号、唯一有效角色和 P3 主合同；
- `GenerationRun` 可以产生该成品修订，H 评测规则的本次检查结果由 `ActivityEvent` 记录；
- 账号替换使第一人称资格和关系姿态重构，证明发布账号和内容角色不是空标签。

补齐控制组织关系后，本路径语义歧义为零。

## 四、经营陈列黄金路径

### 夹具

使用[“折线之间·南城店”墙面双层挂杆黄金闭环](2026-07-22-078-D005D墙面挂杆黄金闭环候选.md)：门店提交六款共二十件本次可用商品，首次方案使用十五件；反馈右上马甲影响拿取后只修订受影响区域，整墙改为十四件。

### 关系闭合

```text
折线之间租户
  → 南城店 OrganizationalParty（任务与成果执行/控制组织）
  → display-merchandising BusinessTask
  → 本次六款商品来源行（均属于折线之间 Brand）
  → DisplayArtifact 首次修订（15 件）
  → ActivityEvent：门店自然反馈
  → 同一 DisplayArtifact 新修订（14 件）
```

- 二十件输入、十五件首次使用、十四件修订后使用均可逐项回指本次商品来源，未出现清单外商品，也未超过任一商品输入数量；
- 三个上下搭配区属于本次 `DisplayArtifact` 内部结构，不为通过校验虚构三个独立 `ProductComposition`；
- 首次成果和局部修订保持同一成果根、任务与组织归属，版本通过通用修订信封表达；
- 路径没有 `PublishingAccount`、`ContentRole` 或 P1—P5 合同，符合对内组织身份边界；
- 本次商品“实际可用”仍是门店用户声明，不是 ERP 或笛语核验真值。

补齐任务组织及商品租户、目录控制方和显式品牌关联后，本路径语义歧义为零。

## 五、哪些内容已经验证，哪些只是设计兼容

| 范围 | 当前证据状态 |
|---|---|
| 单品牌下的内容任务、账号/角色、低商品举证、成品修订与运行留痕 | 已由总部 P3 单品牌合成路径验证 |
| 单品牌加盟店的陈列任务、商品来源、数量约束、成果修订与反馈留痕 | 已由南城店墙面挂杆合成路径验证 |
| A/B/C、D/E、F、H 四类全局资产的稳定语义归宿 | 已由现有 170 项导入候选及逻辑合同证明有真实消费者；本轮未逐项运行 |
| `ContentPlan`、`ContentSeries`、跨账号矩阵与公共素材 | 有已确认业务合同，但不属于本次两条路径 |
| 同一组织多品牌、集团多品牌、跨品牌 `ContentRole`/`PublishingAccount` 适用范围 | **已设计、未验证**；D-019 首期本就是单品牌，第一家多品牌租户进入时由 D-017 和真实数据校准 |

因此，D-016 的关闭证明第一版单品牌双应用语义模型能够承接当前合同，不证明所有未来经营形态已经无需调整。

## 六、横向扩展、前后向兼容与鲁棒性结论

“完全兼容所有未知未来”不能成为诚实的工程承诺。D-016 冻结的是以下**有边界的兼容合同**：

1. **稳定身份不改义**：`BusinessTask`、`ContentItem`、`DisplayArtifact` 等已保存根身份和所属应用不得在后续版本被重解释；内容成果不能通过改字段变成陈列成果。
2. **应用内横向扩展优先增加合同或子型**：新增内容题材、媒体语法或 P1—P5 之外的新内容产品，在注册新版本合同后仍以 `ContentItem` 为成果根；新增橱窗、中岛、模特、培训、概念、巡检等陈列任务，仍以 `DisplayArtifact` 为成果根。只有出现新的独立业务身份和生命周期时，才依据 D-016 准入门槛增加核心类别。
3. **加法优先，破坏性变化必须换版**：可选能力和新子型以版本化合同增加；改变旧字段含义、旧必填条件或旧关系基数时必须发布新合同并显式迁移，不能静默覆盖旧记录。
4. **新版本向后读取旧数据**：新系统必须在承诺范围内继续读取旧任务、旧成果、旧规则和旧修订；历史任务保留当时实际使用的品牌基线、角色、规则和方法版本。
5. **旧消费者对未知扩展安全处理**：旧消费者可以忽略明确标记为可选且不影响旧语义的新信息，或对未知合同/子型明确拒绝；不得把未知陈列子型误当执行方案，也不得把未知内容产品误路由为 P1—P5。
6. **新应用按模块接入**：商品管理、库存管理、空间数字孪生、多媒体生成等若拥有新的数据真源、权限和生命周期，应通过新应用合同及必要的新对象接入；不为“兼容”提前把它们塞进 `Product`、`DisplayArtifact` 或万能 JSON。
7. **失败仍可解释和回退**：未知合同、缺失来源或迁移失败必须安全拒绝或降级，并保留 `GenerationRun`/`ActivityEvent`；不得损坏原始任务、成果版本和知识来源。

租户、组织和品牌同时保持三条独立作用域轴：租户负责隔离，组织负责业务控制，品牌负责适用范围；同属一个租户不能自动跨品牌调用。未来能力若不改变成果的真实性、业务控制、权限和生命周期，优先增加合同或子型；只要任一项发生实质变化，就采用新主版本、显式迁移或模块私有成果对象，而不是扩充万能载荷。

因此可以确认的是：

> **当前模型对两个已知应用内部的横向扩展具有稳定扩展缝，并对旧数据提供版本化向后兼容；它不会、也不应宣称对所有未知未来零改造兼容。**

## 七、关闭结论

- 25 类不增不减；
- 四类卡归宿不变；
- R1、R3、R6 的归属口径和 R7 的可注册产品合同/实际基线集合口径完成最小补齐，R1—R10 仍是一套承重关系，不另增平行关系模型；
- 两条既有黄金路径补齐后均无剩余语义歧义；
- 多品牌和未来横向扩展的证据状态已如实区分为“设计兼容、尚未实证”；
- 兼容性采用稳定身份、版本化合同、加法优先和显式迁移，不使用万能对象、空表、空接口或未知未来承诺。

D-016 正式关闭。该关闭只冻结最小语义对象、承重关系和兼容扩展边界，不表示物理数据库、知识图谱、DIFY 工作流或生产软件已经实现。
