# D-005-A：首期对内陈列成果深度候选

- 日期：2026-07-21
- 版本：V0.2-confirmed
- 状态：**D-005-A 已由用户确认；D-005 整体仍未关闭**
- 本轮只决定：第一版对内陈列成果的业务上限，不同时决定具体区域、载体、字段、页面或工作流。

## 一、通俗地说，这次决定什么

当门店负责人进入“陈列搭配（对内）”并说“帮我把这组货安排到店里”时，笛语第一版究竟：

1. 只讲方法、给灵感；
2. 能结合门店本次手工提供的真实条件，给一份门店可以照着做的文字执行建议；
3. 还要继续检查门店是否执行、判断偏差、发整改并复核关闭。

## 二、三套候选

| 方案 | 具体场景 | 优点 | 主要问题 |
|---|---|---|---|
| A. 培训＋概念案 | 区域经理问“春季墙面怎样做色彩节奏”，系统讲方法并给一个尚未落店的构想 | 输入少、实现最轻、风险低 | 容易退化成培训讲义，门店仍不知道这次具体怎样摆，无法证明系统能够承担陈列专员的部分重复工作 |
| **B. 单场景轻量执行反馈闭环（已确认）** | 门店手工提供一处区域/载体的容量、本次实际可用商品和陈列目标，系统给出文字执行方案；门店人工执行后用自然语言反馈实际差异，系统据此修订 | 直接产生经营价值；不需要 ERP、多媒体生成、审批或巡检系统；同时能验证 `DisplayArtifact` 的真实必要性 | 依赖门店人工输入和人工执行确认；系统不能证明现场真的照做，也不能宣称结果有效 |
| C. 巡检—整改—复核闭环 | 门店上传执行前后证据，系统判断偏差、打分、下发整改任务并复核关闭 | 远程督导价值最高 | 需要图像理解、证据口径、评分、责任人、期限和状态流；会把赋能系统推向监管系统，明显超出第一版 |

## 三、专家建议

用户确认选择 B，并把第一版能力写成：

> **只选择一个轻量但极具代表性的真实场景，完成“最低条件输入—文字执行方案—门店人工执行—自然语言反馈—方案修订”的闭环；不铺开全部陈列场景，巡检、整改和复核监管闭环不开发。**

原因：

- 只选 A，系统很容易成为“会讲课的知识问答”，达不到帮助门店实际工作的目标；
- 选择 B，系统可以在没有 ERP 的情况下使用用户本次手工输入，形成有实际价值但不冒充已执行事实的建议；
- 选择 C，会提前引入多模态识别和监管流程，与当前文字型成品、赋能而非监管、反过度工程化三条边界冲突。

## 四、B 方案的极简边界

1. 对内陈列使用组织身份和业务任务，不套用对外 IP 人设。
2. 第一版不接 ERP；商品、数量、空间和目标由用户按本次任务手工提供，系统不判断库存现实真值。
3. 输入不足时可以降为培训或概念建议；只有用户明确要可执行方案且缺少不可替代条件时，才问一个最小问题。
4. 系统交付的是文字型 `DisplayArtifact`：方案导读、摆放关系、执行步骤、替代办法和必要自检，不生成效果图或现场成片。
5. 门店负责人决定是否采用，并在执行前确认现场条件；不建设总部审批、电子签字、系统批准或合规放行。
6. 用户执行后可以自然告诉系统“太挤、重点不明显”等实际差异，系统据此修订方案；这一步构成轻量执行反馈闭环，但不创建巡检、整改、复核状态机。

## 五、本轮不决定什么

- 首个黄金场景究竟选橱窗模特、墙面挂杆还是中岛；
- 精确需要哪些尺寸、数量和商品字段；
- 是否直接读取门店照片；
- DIFY 节点、数据库表、评分和 UI 页面。

这些只在 B 方案确认后进入 D-005-B，用一个具体“任务目的 × 陈列区域 × 陈列载体”组合继续收敛。

## 六、建议裁决原文

> **D-005-A 正式确认：第一阶段只建设一个轻量但极具代表性的陈列闭环场景，不铺开全部陈列类型。闭环为“最低条件输入—文字执行方案—门店人工执行—自然语言反馈—方案修订”；系统不接 ERP、不核验现实库存、不生成效果图，也不声称方案已批准、已执行或已验证。巡检、整改和复核监管闭环不纳入第一版。新增其他陈列场景时沿用稳定的任务目的、陈列区域和陈列载体合同进行版本化扩展，不破坏首期场景。**

用户已确认本小项。下一步只进入 D-005-B，选择一个首期代表性场景；D-005 仍不会因本小项自动关闭。
