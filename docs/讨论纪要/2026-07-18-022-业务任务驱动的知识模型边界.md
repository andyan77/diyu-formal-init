# 业务任务驱动的知识模型边界

日期：2026-07-18

状态：**对既有 D-012 路线的澄清，尚未新增或关闭决策项。**

## 本轮用户确认问题

用户询问：当前建议是否是以内容产品为导向，反推知识库的对象类型和模型。

## 精确结论

方向正确，但必须限定范围：

1. **内容生产模块**以首期 `ContentProduct` 及其真实任务书为主要驱动，反推完成任务所需的对象、关系、业务问题、事实强度、检索方式和评测标准。
2. **整个笛语系统的共享语义核心**不能只从内容产品反推，还必须同时接受零售经营型陈列任务，以及租户/组织/品牌、账号与工作区、规则、证据、素材、版本和留痕等跨模块约束。
3. 反推得到的不是“每个内容产品一套知识库”，而是先汇总各任务需求，再抽取可复用的稳定对象与关系；只服务单一模块的对象留在模块内部，跨多个任务稳定复用的对象才进入共享核心。
4. 业务任务决定“系统必须回答什么”，语义模型定义“这些信息是什么、彼此如何关联”，SQL、文档/全文、向量索引或后续图谱决定“怎样保存和检索”。这三层不能颠倒。

## 反推步骤

对每类内容产品依次回答：

1. 它替谁解决什么业务问题；
2. 触发任务需要哪些事实、判断、素材和运行时信息；
3. 哪些主张必须有来源，哪些部分允许创意；
4. 输出什么媒体成品，怎样判定合格、同质化失败或事实错误；
5. 总部、区域和门店分别能提供、使用和修改什么；
6. 哪些概念跨多个产品复用，哪些只是主题、词表或模块私有对象。

例如，“真实穿衣问题与购买决策”会验证 `AudienceProfile`、生活场景、商品、商品事实、商品组合、专业判断和内容任务等对象；“门店一线回应与本地生活”会验证门店/组织主体、本地场景、一线观察、媒体素材、发布账号和内容关系链等对象。之后还要用零售经营型陈列任务验证库存快照、门店空间、陈列区域/载体和 `DisplayArtifact`，才能冻结整个第一版最小语义模型。

## 一句话口径

> **不是先造一套本体再寻找用途，而是先用首期内容产品和陈列任务证明系统必须回答哪些问题，再抽取共享的最小语义模型；知识内容可以持续扩充，核心对象类型只在真实任务证明必要时增加。**
