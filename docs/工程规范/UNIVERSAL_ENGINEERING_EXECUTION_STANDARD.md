# 结果优先、风险相称、最小充分、强制收敛的工程执行首要准则 v1.0

## 1. 最高原则

> 所有软件、数据、人工智能和基础设施项目，都必须以与真实风险相匹配的最小充分成本，尽快交付可使用、可验证、可回退的业务结果。

“最佳工程实践”不是最大化流程、证明和治理，而是在正确的位置使用足够的严谨：

- 高风险事项严格验证；
- 普通可恢复事项快速实现和定向验证；
- 业务结果达到验收后立即收口；
- 不得用文件、命令、检查、待办或报告数量代替真实能力。

“最小充分”不等于降低质量。它保留能够防止真实失败的验证，删除重复、无下游、低收益的步骤和产物。

## 2. 风险相称

### 2.1 高风险事项

以下事项允许增加必要的备份、回滚、失败关闭、反向测试和独立验证：

- 登录、权限、企业或租户隔离；
- 敏感信息、隐私、内部秘密和密钥泄漏；
- 资金、计费和不可控外部成本；
- 数据删除、不可逆迁移和生产数据损坏；
- 生产发布、自动对外写入和无法快速回退的变更；
- 法律、平台安全或人身安全明确禁止的行为。

即使属于高风险，也只建设证明当前风险所需的最小保护。

### 2.2 普通事项

以下事项默认使用直接实现、定向测试和人工判断：

- 可回退的业务代码和界面修改；
- 内容、文案、创意、审美和人工可复核质量；
- 命名、目录、格式、注释和普通兼容问题；
- 局部测试、类型、代码规范和小范围重构；
- 不影响当前用户结果的未来扩展建议。

不得把最高风险等级的治理机械套用到普通事项。

## 3. 固定执行顺序

每个执行包按照以下顺序推进：

1. **定义真实结果**：明确最终由谁使用、用户能做什么、哪些核心数字或行为证明完成。
2. **先查最便宜的阻断**：权限、凭据、版本、外部上限、容量、磁盘、网络、依赖和写入边界优先于大规模实现。
3. **交付最小纵向闭环**：先让一条真实路径从输入走到结果，再扩展数量和边界。
4. **范围内持续修复**：普通失败由执行方在同一指令内定位、修复、复测，不为每个问题增加新指令。
5. **开发期定向验证**：修改什么就先验证什么，不在每次微小改动后运行全仓全量检查。
6. **稳定候选一次全量验证**：候选稳定后运行一次必要回归和远程检查；真实修复后只补原阻断及必要回归。
7. **两份并行独立审查**：默认各一次；修复后只做一次定向复审。第三审只解决两审的实质分歧。
8. **验收成立立即收口**：提交、推送、报审并停止扩展。

前置侦察不得扩展成独立工程项目。阻断排除后必须立即进入真实交付。

## 4. 最小充分证据

每项证据必须直接回答以下至少一个问题：

- 用户能力是否真实存在？
- 当前变更是否破坏关键安全边界？
- 失败能否被发现并安全恢复？
- 下游能否直接消费交付？

新建报告、台账、模板、检查器、数据表或治理文件前，必须回答：

1. 它对应哪条既定验收或真实风险？
2. 当前缺少什么证据，现有入口为什么不能覆盖？
3. 不创建是否会影响本次交付？
4. 谁会直接消费？
5. 成本和终止点是什么？

答不清楚，默认不创建。

默认复杂度预算：

- 一个当前有效任务合同；
- 优先复用一个既有检查入口；
- 一套最小测试和失败案例；
- 一个结果与交付请求；
- 两份独立审查。

禁止默认建设：

- 检查器再检查检查器；
- 同义合同、平行数据模型或重复账本；
- 为每个批次复制任务框架；
- 多层摘要、回执、证明链和审查角色；
- 没有当前用户价值的“未来完备性”设施。

## 5. 测试和审查

- 测试重点覆盖本次改动、真实用户路径、关键失败和高风险边界。
- 同一事实已有有效证据时直接复用；只有对象改变、外部状态实质漂移、前次证据不可信或高风险终验明确要求时才重新核验。
- 不以测试数量、代码覆盖率、检查器数量或报告页数代表业务完成度。
- 同一候选默认只做两份独立正式审查；审查意见必须指向既定验收、真实证据和当前损害才能阻断。
- 命名、风格、额外字段和未来建议默认列为非阻断。
- 修复后只复核原阻断和受影响的必要回归，禁止重新开启无边界全量找问题。
- 本地模拟、结构覆盖和平均分不能替代真实生产路径、逐项结果或最终验收。

## 6. 强制收敛

每份执行指令必须明确：

- 核心业务目标和核心数量；
- 成功边界；
- 写入与外部授权边界；
- 测试和正式审查上限；
- 允许新增的最少工程与治理产物；
- 达到验收后必须立即停止的规则。

执行侧不得因为“更完整”“更严谨”“以后可能需要”继续增加代码、检查器、数据结构、文档、摘要、审查或平台。

如确需超过默认预算，必须在实施前说明：

1. 要防止的具体真实损失；
2. 现有保护为何不足；
3. 新增成本和明确上限；
4. 何时必须终止扩展。

无法回答以上四项，不得扩大流程。

## 7. 状态汇报

状态汇报先给业务核心结果：

```text
核心目标：完成多少 / 总量
用户能力：已经能够做什么
真实失败：还有几个，影响什么
成本：实际时间、调用或费用
下一步：唯一最短动作
```

“待办全部完成”“检查全部绿色”“文件全部齐全”只能作为辅助证据，不能冒充业务目标完成。

## 8. 完成定义

任务只有同时满足以下条件才算完成：

- 约定的用户结果真实可观察；
- 关键失败路径和高风险边界得到风险相称的验证；
- 必要测试和审查完成；
- 不存在与当前验收直接对应的阻断；
- 交付可被下游直接消费或安全回退；
- 执行侧已经停止继续扩展。

如果业务结果尚未产生，即使准备清单、脚本、报告和检查均已完成，也只能称为“准备就绪”或“诚实停止”，不能称为业务完成。

## 9. 对后续所有项目的约束

所有后续规划、执行指令和审查必须显式承接本准则，而不能只写“符合最佳工程实践”。

如果项目本地规则与本准则冲突：

- 安全、法律和不可逆风险采用更严格规则；
- 普通工程流程采用更小、更快、可验证的路径；
- 任何额外流程必须说明真实风险、直接消费者和收敛上限。

一句话判断：

> 如果一个新增步骤不能直接降低本次交付的真实风险、补齐既定验收证据或服务明确下游，就不应执行；如果全部必需验收已经成立，就必须停止。
