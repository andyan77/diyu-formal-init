> ① §5-2、§9、§10-2 中"采用此版应为唯一红色主动作 / 补 adopt 端点"的建议**已撤回**——依据 ADR-026，采用是行为事件，V1.1 采用"场景化主动作 + 设为当前版本 + 存为素材 + 纳入系列"口径；
> ② §3 中"分店也能用 / 平常聊的不会被一直记着 / 用谁的口吻发 / 不会卡住整个店"四处替换文案**已被创始人纠正**，最终口径见规范 §2.3；
> ③ 全文 `runtime_verified` 一律读作 `prototype_verified`（原型验证，不证明生产运行）；"一线水准"读作"有竞争力的高保真候选"。§8 的 S14—S21 收敛为三类复用模式；§5 移动端三段切换器改为"对话+成品，上下文进抽屉"。
>
> **性质**：针对《笛语交互UI完整架构设计_v1.0》与《笛语HTML高保真状态模型_v1.0》的独立评审与修订规格
> **日期**：2026-07-23
> **评审方式**：通读设计文档全文 48 节 + 亲验 3 张预览图 + 抽查高保真 HTML 源码；另派 7 个独立评审视角（全量文案盘点 / 文案凝练 / 认知负荷 / 大师审美 / 店长视角 / 后端一致性 / 查漏批判）交叉评审，关键结论逐条复核。

## 0. 总评

**骨相是对的，病在"嘴太碎"。**

桌面主流程已是一线水准：自动版本取代"保存"、成品右扩为视觉中心、陈列只改受影响区、追问只问一个问题——这些核心下注全部真实落地，不是口号。名词主干（任务/成品/版本/DisplayArtifact/五类成果）与主仓代码契约是同一套，"依据项目记忆.md"的 6 条关键继承声明抽查全部属实（`static_verified`）——这份设计不是空中楼阁。

但它有一个贯穿性的毛病：**这套自称"安静"的界面，太爱解释自己的安静**。全站散落着十几条"我们不显示模型""不核验真值""不阻塞租户"式的自我旁白——每一句都是系统在清嗓子说"看我多克制"，恰恰把藏起来的复杂度又搬回了店长眼前。这就是"啰嗦像说明书"的病根。

修复路线一句话：**删掉自我旁白、把黑话翻成店长的话、把朱砂红还给"采用此版"这一个按钮、把移动端的路走通。**

## 1. 必须保住的（真功夫，修订时严禁误伤）

| 亮点 | 为什么是顶级的 |
|---|---|
| 首屏「今天要完成什么？」 | 取代烂大街的"有什么可以帮你"，第一句就指向目标 |
| 「更短 / 更口语 / 自定义」选区按钮 | 2-4 字动词，贴真人说话，教科书级微文案 |
| 「只修改了 C 区，A、B 区和下杆不动」变更横幅 | 店长最怕反馈一句被重排整面墙，这句直接建立信任 |
| 「要给出准确数量与间距，我还需要知道这组双杆大约能挂多少件」 | 缺什么 + 为什么问，一句讲透 |
| 正挂 / 侧挂 / 上墙 / 主焦点 / 主来客 | 用了店长真实行话，没把用户当外行硬翻译 |
| 「这次成品没有完成。你的输入已保留，可以重试。」 | 先安抚（东西还在）+ 给下一步，无术语无指责 |
| 演示叙事文案（"三天只带一件，我先把轻的放回去"） | 内容本身的文笔是真功夫 |
| 无"保存"按钮、无假进度条、无蓝紫渐变、无机器人头像 | 敢说不，兑现了自订铁律的大半 |

## 2. 病灶一：自证式旁白 ——"越声明克制，越不克制"

**原理**：隐藏复杂度（§2.3）的正确做法是**不提**，而不是**声明自己没显示**。一旦要写"轨迹不显示在这里"，复杂度就从界面搬进了"关于界面的一句话"里。

**分流判据（本次评审最重要的可复用规则）**：一条"免责/边界"句要不要留，不看它是不是自我叙述，看它**改不改变用户的预期或动作**：

- **改变预期/动作 → 保留但翻成人话**（见 §4 改写表）
- **只描述用户从没问过的后台内部 → 删**（见下表）

### 2.1 直接删除清单（不改写，删）

| 位置 | 现文案 | 处置 |
|---|---|---|
| 内容/陈列侧栏底部常驻 | "底层检索、模型与运行轨迹不会显示在这里。" | **删**。声明"我藏了什么"=把后台请进界面 |
| 13 搜索与状态 页面副标题 | "加载、错误、权限和输入缺口都不暴露模型、检索或工作流轨迹。" | **删**，换正面一句："出问题时，只告诉你结论和下一步。" |

| 03 内容空状态 引导语尾句 | "笛语会在必要时只问一个问题。" | **删尾句**。让用户亲历这份克制，不要先读它的说明书 |

## 3. 病灶二：内部黑话泄漏（违反自订 FE-R03「普通用户不见内部语言」）

| 黑话 | 出现位置 | 店长会怎么误解 | 换成 |
|---|---|---|---|
| **成果合同** | 首屏第一句（landing 副标题） | 以为要签法律合同 | "选好要做哪件事，笛语就照这件事帮你。用平常说话的方式跟它讲就行。" |
| **业务域** | 首屏 + 应用切换 | 无此概念 | "选一个要做的（做内容 / 排陈列）" |
| **租户** (tenant) | 11 管理台 admin-note | 租房子的租客？ | "缺哪项只影响哪项，不会卡住整个店。" |
| **经营真值** | 08 库存输入 input-note | 完全读不懂，反而更慌 | "这份清单只用来排这次的墙，不会动你店里的库存账。" |
| **服务端** | 陈列身份栏副标题（"任务条件由服务端固定"） | 以为被系统锁死 | "挂杆条件已按本店设好，不用每次重填。" |
| **下行可用** | 12 素材范围徽标 | "下行"=下滑趋势？ | "分店也能用" |
| **黄金成品** | 05 成品页脚注 | 团队内部词，零意义 | 整词删（见 §2.1） |
| **作用域** (scope) | 全局搜索底部 | 程序员词 | "搜索范围：南城店 · 你自己的素材" |
| **长期记忆** | 输入框脚注 | AI 黑话 | "平常聊的不会被一直记着，除非你主动存起来。" |
| **内容角色** | 身份栏 / 管理台 | 和自己岗位分不清 | "用谁的口吻发（店长本人 / 品牌官方）" |
| **联动区 / 搭配区** | 文档叫"联动区"、界面叫"搭配区" | 以为是两个东西 | **全局统一为"搭配组"**，删"联动"二字（陈列侧最核心的看图词，必须统一） |
| **局部修订 / 局部新版** | 多处标签 | 公文腔 | "只改了这里" / "小改一版"（它自己的横幅原文"只修改了 C 区"就很好，标签向横幅看齐） |
| **挂杆条件** | 陈列侧栏 + 条件页标题 | "条件"像要达标的门槛 | "本店挂杆情况"（或"这面墙的挂杆设置"） |
| **特殊目标** | 08 库存输入提示 | 含糊 | "有特别想主推的，顺手说一句。" |
| **编排分集 / 跳集 / 插集 / 前情** | 07 系列页整套 | 导演行话太重 | "每一集的顺序你自己排" / "记到下一集，接着这条往下讲" |

**另一条同类问题**：店长说"右侧"，系统回应却切成代号"C 区"，中间缺一次翻译。修法：系统回应把方位词和代号绑在一起说——**"已把右边那组（C 组）减掉一件、重新摆正，别的都没动。"**

## 4. 承重边界保护清单（只能改写、不能删——防过度执行）

如果把"删掉一切旁白"照单执行，会误删承重墙。以下四条是**真业务契约/合规边界**，修订时必须保留语义、只做人话化：

| 边界 | 为什么必须留 | 人话化后 |
|---|---|---|
| 未成年人原件边界 | 合规/肖像安全声明（§3.8） | "第一版不能处理认得出真人未成年人的照片、视频或声音。请改用不带人物的画面或文字描述。"（可移到上传时才出现，但不许删） |
| "示意图非实拍" | §46 明列受控风险"陈列示意被误解为实景"，店长照着上墙前必须知道 | "这是摆放示意图，不是门店实拍。"（只删"也不表示系统已识别或验收现场"这半句系统自证） |
| "清单不动库存账" | §3.7 真业务契约（第一版不接 ERP/POS） | "这份清单只用来排这次的墙，不会动你店里的库存账。" |
| "管理员身份不自动取得素材维护权" | §3.6/§26 真实权限边界，可行动 | 保留原意，保留在素材/权限页语境里 |

## 6. 按钮与提示改写精选表

### 6.1 按钮（原则：上下文已给足时，按钮只留动词）

| 原 | 改 | 理由 |
|---|---|---|
| 生成挂杆方案 | **生成方案** | 整页都是挂杆场景，"挂杆"冗余 |

| 搜索 placeholder："搜索成品、任务、系列、方案或私人素材" | "搜索你的成品、任务、方案"（举 2-3 个代表即可，再多就成清单） |
| "粘贴清单即可；特殊目标可以顺手补一句" | "把清单粘进来就行；有特别想主推的，顺手说一句。" |

**整体效果**：按此口径全站文案可减字 30–40%，信息零损失。

### 6.3 "采用 vs 导出"的店长困惑（中优先）

单店店长做完一版只想拿去发小红书，"采用"（在多版里定一版为准、供系列接续和留档）的价值对他不可见，极易和"导出"混淆。修法不是改名，而是**给按钮点明后果**：hover/长按或首次使用时给一行注释——"采用此版 = 以后就以这版为准"；"导出 = 下载带走，不影响哪版为准"。

## 7. 三条可复用的文案心法（"怎么不啰嗦"的答案）

1. **边界文案分流判据**：改变用户预期/动作的 → 保留但翻人话；只描述用户没问过的后台内部的 → 删。（详见 §2 与 §4）
2. **正面陈述，禁否定句罗列**。全稿"不核验、不自动、不显示、不冒充、不阻塞、不覆盖"式否定句密度极高——这是规格书的写法（规格书要写"我不做什么"），不是界面的写法。界面只说"是什么、下一步做什么"。
3. **Toast 只确认，页面才教学**。点击瞬间弹出的反馈只回答"刚才那下成了没、结果在哪"，任何产品理念、语义区分不许出现在 toast 里。

## 8. 13 个状态之外，缺一整类"不顺路"状态

现在的 13 态是"顺风模型"。在把它定为"开发与视觉验收共同基线"（FE-R08）之前，建议先补以下状态设计（门店店长信号差的一天必然撞上）：

| 建议新增 | 内容 | 为什么必须有 |
|---|---|---|
| S14 搜索无结果 | 结论级空态："没有找到匹配的成品或方案。换个说法，或去我的成品里看看。" | 现在 searchMarkup 永远渲染 4 条写死的命中，搜一个拼错的词界面无法回应 |
| S15 列表零数据 | /我的成品、/连续系列、/历史方案、/私人素材、/内容计划 五个列表的"第一件从这里开始"引导 | 原型所有列表都是预填的；入驻就绪本身就假设了什么都没有的新账号 |
| S16 断网/弱网 | 面向用户的网络中断提示与降级行为 | 目标人群是卖场里的店长，断信号是高频事件；§35 错误表没有这一行 |
| S17 生成失败 + 重试落点 | 失败态界面 + 真实"重试"按钮 | §35 文案承诺"可以重试"，但 13 态里没有生成失败态，重试无处可点 |
| S18 删除确认 | 删除私人素材的确认弹窗 + 不可恢复性说明 | 现在全站有后果的动作都是点一下就 toast，没有一个确认 |
| S19 库存解析失败 / 识别 0 项 | 粘进乱码、识别 0 项、某行歧义的纠错界面；行内"编辑"铅笔的实际交互定义 | D01"解析确认（轻量纠错）"这个态在稿里没有真正的纠错界面 |
| S20 触屏选字方案 | 移动端"选中台词→更短/更口语"与**手机系统自带复制粘贴菜单的冲突**解决方案 | 局部修改整套机制依赖选字，触屏上会被系统选择手柄压掉；§39 对此只字未提 |
| S21 冷启动桥接 | 新账号品牌未确认时，"今天的建议"产不出来 → "先去确认品牌草案再来生成建议"的桥接文案 | 内容建议依赖品牌上下文，但新账号该项还是管理台待办——先后依赖没有引导 |

## 9. 给开发阶段的三个契约预警（设计 vs 主仓真实代码，`static_verified`）

| 漂移 | 事实 | 建议方向 |
|---|---|---|
| "采用此版"无后端端点 | 真实 /api/v1 无任何 adopt 端点，代码只有 `POST .../save`（主动保存） | **补后端，不降 UI**——"采用≠保存"是核心产品下注和 P0 验收 #6 硬指标，不能迁就现状让 UI 退回"保存"语义 |
| "内容计划"与代码撞名 | 代码里 `plan` 专指陈列墙面方案（GeneratedDisplayArtifact.plan）；设计的 content/plans 指五类内容上层编排 | 前端命名显式区分（如 content-programs），防止 codegen 串味 |
| 19 个 API 逻辑组只有 7 个真端点背书 | 严格对上仅 display revisions 一条；版本寻址设计按 artifact、代码按 task；文档自己声明了"待对齐"，属自知非隐瞒 | P0 开发前列明"后端待建"清单再喂 openapi→TS codegen；注意 P0 壳层承诺的"继续最近/我的成品/基础搜索"目前也无 list/search 端点 |

另有微差：display 修订端点真实名是 `.../revisions`（复数）；`session/context` JSON 端点不存在（身份目前只渲进 HTML），React 壳的身份条需要先补这个数据源。

## 10. 待创始人裁决项（不阻塞，供留意）

| # | 事项 | 选项与利弊 | 倾向建议 |
|---|---|---|---|
| 1 | **"成品"要不要换皮** | 店长视角：服装行业"成品"=成品衣服，会误读成货品清单。后端视角：这词与代码契约一致。两者可分层两全：后端资源名不动，UI 标签单独换（如"做好的内容"） | 先不动。店长实测时真有误读再换表现层标签，不阻塞任何事 |
| 2 | **adopt 端点补法** | (a) 后端新增 adopt 端点；(b) 服务端把"采用"映射为 save 之上的独立状态 | 两者皆可，但**方向必须是改后端**，不是让 UI 退回"保存" |

## 11. 一句话收尾

这套设计已经赢在骨架——它敢对"保存按钮、假进度条、聊天中心主义"说不。现在只需要它对**自己的解说欲**也说不：删掉自我旁白、把黑话翻成店长的话、把红还给"采用此版"这一个按钮、把移动端的路走通。做到这四件事，它就配得上自己写下的那五条铁律。
