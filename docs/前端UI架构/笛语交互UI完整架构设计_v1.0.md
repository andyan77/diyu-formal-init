# 笛语交互 UI 与前端完整架构设计 V1.0

> **状态：评审候选稿**  
> **范围：**内容生产（对外）、陈列搭配（对内）、最小企业管理台、统一工作壳、前端工程架构、状态模型与验收口径。  
> **依据：**`项目记忆.md`、`待决策清单.md`、`diyu-frontend-architecture.pdf`。  
> **边界：**本文继承已确认业务合同；本文新增的前端技术选型、路由、组件和视觉表达属于实施建议，正式开发前仍应由用户确认。

---

## 1. 总体结论

笛语不应做成“懂服装的聊天机器人”，也不应做成“把后端模块搬进菜单的 SaaS 后台”。推荐形态是：

> **一个统一、安静的业务工作壳；两个严格分流的业务域；自然对话负责理解与修改，正式成品负责承载业务结果。**

桌面端核心结构：

```text
顶部：当前应用 + 当前业务身份（只读） + 搜索 + 个人账户
左侧：当前业务域的任务、成品与可延续上下文
中间：自然协作、必要追问、修改指令
右侧：可直接采用的正式成品与版本
```

产品只需持续回答六个用户问题：

1. 我现在代表谁？
2. 我今天要完成什么？
3. 系统是否还缺一个真正必要的信息？
4. 正式成品在哪里？
5. 怎样只改需要改的部分，并保留旧版本？
6. 哪一版被采用，是否要继续到系列、计划或下一任务？

# 第一部分：产品哲学与界面铁律

## 2. 乔布斯式工程哲学的五条界面规则

### 2.1 聚焦

每个页面只设置一个首要动作：首次进入选择业务域；内容空状态开始输入；成品状态采用或继续修改；陈列输入状态提交库存；管理台处理当前唯一阻塞事项。

### 2.2 端到端整合

一次成果在一个任务空间中完成：

```text
自然交流 → 形成任务 → 成品 V1 → 自然语言修改 → V2 → 采用某版 → 导出/延续
```

用户不在“聊天、任务、草稿、保存、版本、历史”六套近似对象之间来回判断。

### 2.3 隐藏复杂度

普通用户与企业管理员均不看到数据库、图谱、索引、检索过程、Prompt、模型路由、评分、重试、淘汰候选、P1—P5 编号和内部工作流节点。普通质量问题由系统后台先修复；只有不可替代输入缺失、权限越界或明确硬禁项时，才给一句最短行动提示。

### 2.4 渐进显露

未形成任务不显示版本；未生成成品不显示成品工具栏；未选择系列不显示系列编辑器；用户未选中文字不显示局部修改；无权限不显示管理入口；无维护职责不显示组织素材编辑动作。

### 2.5 细节建立信任

必须始终清楚显示：实际登录人、代表组织、发布账号、内容角色、当前应用、任务归属、当前版本、采用版本，以及导出、采用、系列写入和私人素材保存的不同后果。

## 3. 已确认业务合同的 UI 映射

### 3.1 双应用顶层明确

- **内容生产（对外）**：交付可直接拍摄、制作或排版的文字型内容成品。
- **陈列搭配（对内）**：交付墙面双层挂杆的文字执行方案及自然反馈后的局部新版。

两个应用共享设计系统、会话、搜索和基础组件，但任务、成果、身份语义和数据严格分流。内容侧形成内容成品；经营陈列侧形成 `DisplayArtifact` 语义的方案。

### 3.2 自然对话、必要收敛、接受后才形成任务

普通交流、构思和讨论不自动创建任务。只有明确要求成果、接受系统建议、继续已有成果，或选择计划项/系列下一集执行时，才形成或继续任务。真正会改变成果的缺口只追问一个问题。

### 3.3 只向用户呈现成品

内容成品可按任务需要包含成品导读、标题、完整正文或台词、字幕、镜头、动作、图片顺序、场景、造型、声音、封面、发布配文和必要制作提示。这些是成品，不是后台轨迹。

### 3.4 一个独立成品一个主要成果合同

用户不理解也不选择 P1—P5。系统自动判断主要成果；构图、镜头、造型、商品解释和本地语境只是共享能力。只有用户明确要求另一份可独立采用的完整成果时，才创建第二成果。

### 3.5 版本、采用、导出、系列写入分开

- 每次生成与修改自动形成版本；
- **采用此版**：指定当前认可的业务成果；
- **导出**：带走当前版本，不自动采用；
- **作为下一集前情**：用户明确更新系列；
- **存为私人素材**：用户主动选择后才允许跨任务复用；
- 浏览、生成、中间稿选择和导出均不自动改长期记忆或系列。

### 3.6 私人素材与组织素材分开

每个有效自然人可拥有本人独占的私人素材；组织素材由明确授权维护人从本组织入口录入。第一版私人素材不转组织、不投稿、不审批；管理员身份不自动取得组织素材维护权。

### 3.7 人工库存是当前任务输入

第一版不接 ERP/POS，不做库存真值判断、来源评分、冲突裁决或对账。陈列页面应明确说明库存清单只服务本次方案。

### 3.8 真人未成年人原件边界

第一版不接收已知可识别真人未成年人的照片、视频或声音；私人素材、组织素材和临时附件入口一致执行。界面不承诺自动年龄识别，但上下文已知时必须在持久化或发送模型前阻止。

# 第二部分：总体信息架构

## 4. 一套工作壳、两个业务域、一个独立管理面

```text
笛语统一工作壳
├── 内容生产
│   ├── 新建内容
│   ├── 继续最近
│   ├── 我的成品
│   ├── 连续系列
│   ├── 内容计划
│   └── 私人素材
├── 陈列搭配
│   ├── 新建挂杆方案
│   ├── 历史方案
│   └── 本店挂杆条件
└── 企业管理（有权限时）
    ├── 入驻与就绪
    ├── 品牌表达与账号方向
    ├── 人员、账号与角色
    └── 组织素材
```

企业管理不与两个日常应用长期平级占据顶层导航，而从账户菜单或专用入口进入。

## 5. 推荐路由

```text
/                                  智能首页：首次选择或继续最近
/content                           内容生产空状态
/content/tasks/:taskId             内容任务空间
/content/artifacts                 我的成品
/content/artifacts/:artifactId      成品与版本
/content/series                    连续系列
/content/series/:seriesId           系列卡片与分集计划
/content/plans                     内容计划
/materials/private                 私人素材
/display                           陈列库存输入
/display/tasks/:taskId              陈列任务与版本
/display/history                   历史方案
/display/conditions                本店挂杆条件
/admin/readiness                   入驻与就绪
/admin/brand-expression            品牌表达与账号方向
/admin/access                      人员、账号与角色
/admin/materials                   组织素材
```

## 6. 进入规则

| 情况 | 默认行为 |
|---|---|
| 只有一个应用权限 | 直接进入该应用最近合法状态 |
| 首次使用且有两个应用 | 显示双应用选择页 |
| 返回用户 | 打开上次合法应用并优先“继续最近” |
| 从任务链接进入 | 直接进入任务 |
| 要求与当前应用冲突 | 提示一次切换，不静默混用合同 |
| 多个授权发布身份 | 服务端返回合法选项；任务形成后身份冻结 |

---

## 7. 统一工作壳

### 7.1 顶部栏

从左到右：笛语标识、应用切换、只读业务身份、全局搜索、帮助、自然人账户。推荐身份显示：

```text
正在代表：折线之间 · 南城店账号
内容角色：店长 / 门店经营者
工作组织：南城店
```

切换身份应新建任务，不在原任务中静默改变。

### 7.2 左侧导航

侧栏承载“我正在做的业务和可延续上下文”，不承载技术后台。只显示最近 5—8 项，更早内容通过搜索获取。

### 7.3 动态主工作区

| 阶段 | 中间协作区 | 右侧成品区 |
|---|---:|---:|
| 空状态 | 100% | 不显示 |
| 讨论/最小追问 | 65% | 可显示上下文摘要 |
| 成品生成后 | 38% | 62%，成为视觉中心 |
| 专注修改 | 可收至 320px | 占剩余空间 |
| 移动端 | 对话/成品/上下文独立切换 | 不并排 |

### 7.4 全局搜索

`Ctrl/Cmd + K`。结果优先级：已采用成品、当前任务、系列、计划、历史陈列方案、本人私人素材、具体对话片段。搜索先按租户、组织、品牌、账号和私人边界过滤，再做全文或语义匹配。

# 第三部分：内容生产工作台

## 8. 内容状态流

```text
C00 空状态
  ├─ 自由输入
  ├─ 接受系统建议
  └─ 选择计划项/系列下一集
        ↓
C01 自然协作 / 最小追问
        ↓ 明确执行
C02 生成中
        ↓
C03 成品 V1
  ├─ 采用此版
  ├─ 自然语言修改 → C04 V2/V3...
  ├─ 导出
  ├─ 保存所选片段为私人素材
  └─ 明确写入系列前情
```

## 9. 内容空状态

新账号不以“有什么可以帮你？”开场。系统利用品牌、账号角色、受众、平台、商品、素材和现实制作条件提出一个可直接接受的建议。建议未被接受前不创建任务。

返回首页优先显示：继续最近、最近采用成品、一个系统建议、系列下一步和计划中可立即开始的一项；不显示无行动价值的大盘。

## 10. 输入框与上下文

输入框保持自然，可显示少量上下文标签：发布账号、平台与媒体、独立单篇/当前系列、已选商品、已选素材、制作条件、用户明确时长。标签是辅助，不是必填。

可选业务语言捷径：帮助顾客做选择、讲清一件商品、写品牌/账号观点、回应本地生活、做视觉内容。编号只在内部存在。

## 11. 成品工作面

### 11.1 头部

显示标题、平台与媒体、制作条件、当前版本、采用状态、任务时间和所属账号。主要动作：

- **采用此版**；
- **继续改**；
- **导出**；
- 更多：复制全文、查看版本、结束任务。

所有版本自动保留，不把“保存”作为首要动作。

### 11.2 正文

按任务自然组合：成品导读、标题与入口、完整台词/正文、镜头/动作/图序、声音、发布配文和制作提示。默认展示干净成品。

### 11.3 局部修改

选中标题、台词、镜头或图序后，可直接输入“更口语一点”“缩短三分之一”“只改结尾，其他不动”。修改生成新版本，不覆盖旧版；变化摘要可展开，默认不显示整页红绿 diff。

### 11.4 版本

版本条示例：`V1 初稿 · V2 当前 · V3 已采用`。恢复旧版不删除新版本，而是将旧版复制成新的当前版本，保留谱系。

## 12. 我的成品

统一承载已采用、继续修改和未采用成果，替代多个“草稿/保存/历史版本”一级菜单。列表卡片展示成品首屏摘要，而不是聊天最后一句。

## 13. 连续系列

系列只维护一份轻量卡片：定位、品牌关联、账号角色、稳定设定、主题地图/叙事大纲、分集计划、当前进度和未完线索。用户可跳集、插集、改序和重设；生成、浏览和导出不自动更新系列。

## 14. 内容计划

内容计划是五类内容成果的上层编排，不是第六类产品。计划项显示预期受众回报、品牌关联、发布语境、最低输入和是否可直接开始。点击“开始制作”后才创建任务。首版不建设日历引擎、提醒、自动排程或自治运营 Agent。

## 15. 私人素材与私人协作偏好

私人素材保存用户主动选择长期复用的内容；私人协作偏好只保存“系统怎样更好地与本人协作”。本人可查看、纠正、删除、关闭偏好，并可开启不读取也不写入偏好的临时会话。

# 第四部分：陈列搭配工作台

## 16. 首期范围

首期只实现：

> **视觉陈列 × 卖场墙面 × 一组双层挂杆的文字执行闭环。**

固定挂杆条件首次记录，变化时更新；日常默认只提交本次实际可用库存。系统形成完整方案，门店人工执行后自然反馈，系统只修订受影响部分。

不做 ERP、库存核验、图片巡检、陈列打分、整改状态机、效果图生成和其他区域空入口。

## 17. 状态流

```text
D00 库存输入
  ↓
D01 解析确认（轻量纠错）
  ↓
D02 完整挂杆方案 V1
  ↓ 门店自然反馈
D03 局部修订 V2
```

## 18. 库存输入

支持粘贴、上传结构化清单、补充特殊目标或复制最近清单。解析后只显示商品名、数量和必要属性供纠错，不要求完整商品管理表。页面明确：本清单只用于本次方案，笛语不核验库存真值。

## 19. 挂杆条件

首次记录上/下双杆结构、大致容量、主要视线带、固定限制和主来客方向。条件更新只影响后续任务，不静默改写历史方案。

## 20. 方案工作面

右侧同时提供：

1. **结构示意**：上/下杆、A/B/C 三个上下联动区、主焦点、回应点、正挂、侧挂和数量；明确标记为结构化示意，不冒充实景。
2. **文字执行方案**：商品分组、左右次序、上下联动、主次正挂、侧挂节奏、数量间距、真实替代和执行步骤。

## 21. 自然反馈与局部修订

用户说“右侧两件外套太厚，挂不下”后，新版只改 C 区：减少一件、重新居中，A/B 区和下杆顺序不动，并显示变化摘要。禁止每次反馈重写整篇方案。

# 第五部分：企业管理台

## 22. 定位与模块

管理台只承载四类真实配置：入驻与就绪、品牌表达与账号方向、人员/发布账号/内容角色、组织素材。普通用户看不到数据库、图谱、模型、生成轨迹、万能规则编排和无业务消费者的大盘。

## 23. 入驻与就绪

最多显示 3—5 个真实待处理事项，例如品牌表达草案待确认、账号未绑定合法内容角色、首次挂杆条件未记录、组织素材维护人未指定。每项说明“为什么现在需要、完成后解锁什么、下一步是什么”，不使用统一完成度百分比。

## 24. 品牌表达与账号方向

系统先用行业原型、具体正反例和简短对话预填人话版草案。企业只需“像/不像、用/不用、A/B/都不对”、改几句话并确认版本，不填写抽象长问卷。

## 25. 人员、发布账号与内容角色

界面必须拆开自然人登录身份、组织成员资格、发布账号、内容角色和组织岗位。发布账号可多人运营，但不能成为共享登录账户；历史始终记录实际操作者。

## 26. 组织素材

只有授权维护人显示编辑动作。每项素材明确控制组织、适用品牌、仅本组织或明确下行网络、版本与维护人。“组织官方”不自动成为品牌事实、知识、公共素材或训练数据。

# 第六部分：状态模型与高保真稿

## 27. 用户可见状态

不建设重型八态工作流。内容任务只需呈现讨论中、等待必要输入、生成中、已有成品、正在修改、已采用和已结束；陈列任务呈现等待库存、已解析、已有方案、收到反馈和局部新版。后台可继续完整留痕，但不把事件名、评分和内部检查展示给用户。

## 28. 关键状态稿

| ID | 页面 | 关键验证 |
|---|---|---|
| S00 | 首次双应用入口 | 两应用边界清楚，不暴露内部编号 |
| S01 | 返回首页 | 继续最近、系统建议、最近成品优先 |
| C00 | 内容空状态 | 自然输入，不以问卷开场 |
| C01 | 最小追问 | 只问一个真正改变成果的问题 |
| C02 | 生成中 | 不显示虚假百分比与后台轨迹 |
| C03 | 内容成品 | 成品成为视觉中心，完整可制作 |
| C04 | 局部修改 | 不覆盖旧版，采用/导出分开 |
| C05 | 系列与计划 | 计划不等于任务，系列由用户编排 |
| D00 | 库存输入 | 清单是本次输入，不冒充 ERP |
| D01 | 双层挂杆方案 | 上下、左右、焦点、数量、步骤完整 |
| D02 | 反馈新版 | 只修订受影响区域 |
| A00 | 入驻与就绪 | 只显示真实阻塞项 |
| A01 | 素材边界 | 私人/组织入口和权限分开 |
| X00 | 搜索与系统状态 | 结论级、可行动、无后台泄露 |

# 第七部分：前端技术架构

## 29. 推荐技术选型

| 层 | 推荐 |
|---|---|
| 框架 | React + TypeScript + Vite |
| 路由 | React Router |
| 服务端数据 | TanStack Query |
| 局部 UI 状态 | 组件状态 + 小型 Zustand，仅存布局、选中版本等可重建状态 |
| 组件/无障碍 | Radix primitives + 自有组件层；可从 shadcn/ui 起步但重做视觉 |
| 样式 | CSS variables + Tailwind/utility layer，VI 全部 token 化 |
| 表单 | React Hook Form，仅用于管理台和结构化补充 |
| API 类型 | `openapi.json` 生成 TypeScript client/schema |
| 测试 | Vitest + Testing Library + Playwright |
| 部署 | 静态资源同 ECS/Nginx 或 CDN；API 保持 `/api/v1` |

### 29.1 P0 起使用同一工作壳

不建议先扩展一套 server-rendered 临时业务工作台，再重写 SPA。成品画布、版本、局部修改、搜索和身份是核心交互，不是装饰；双轨会制造两套状态与验收。登录、邀请等简单页可保留服务端渲染，但业务工作区从第一条纵向闭环起使用最终 React 壳。

## 34. 生成与流式

P0 可非流式，只显示“正在整理成品……”，超过合理时间后提供取消与继续等待，不显示百分比。后端稳定支持 SSE/fetch stream 后再流式，但半成品不能自动成为版本，断线可恢复，取消不留下不完整成品，流式不改变采用、导出和系列语义。

## 35. 错误设计

| 场景 | 用户提示 |
|---|---|
| 缺陈列容量 | “要给出准确数量与间距，我还需要知道这组双杆大约能挂多少件。” |
| 缺 P5 真实商品 | “这条视觉内容需要一件明确属于当前品牌的真实商品。” |
| 应用冲突 | “这项要求属于经营陈列。切换后会新建陈列任务，当前讨论会保留。” |

| 生成失败 | “这次成品没有完成。你的输入已保留，可以重试。” |
| 搜索不可用 | “当前无法获取公开信息，我会先按现有品牌与账号上下文完成一版。” |
| 已知未成年人原件 | “第一版不能处理可识别真人未成年人的照片、视频或声音。” |

# 第九部分：响应式、无障碍、安全与性能

## 39. 响应式

- 桌面：左侧导航 + 对话/成品并排；
- 平板：侧栏抽屉化，对话/成品约 42/58；
- 移动端：顶部身份摘要，底部“对话/成品/上下文”切换，输入框固定底部；不硬塞三栏。

## 40. 无障碍

目标 WCAG 2.2 AA：全键盘、清晰焦点、图标名称、颜色非唯一状态、对比度、正确弹窗焦点、生成状态 `aria-live`、版本切换后焦点回成品标题、支持减少动效。

## 41. 安全与隔离

- HttpOnly/Secure/SameSite Cookie；前端不存 token；
- CSRF 防护；所有作用域由服务端校验；
- 客户端隐藏菜单不是权限边界；
- 身份切换时清理相应 Query Cache；
- Markdown/富文本白名单渲染；
- 遥测不写入正文、Prompt、私人素材和私人偏好；
- 搜索、上传、导出、缓存、日志、流式连接遵守同一租户和作用域隔离。

## 42. 性能目标

建议：工作壳 LCP ≤ 2.5s；路由切换可交互 ≤ 200ms；输入延迟 ≤ 50ms；最近列表不超过 20 项；长成品按区块渲染；历史版本懒加载；内容、陈列、管理按路由拆包。

# 第十部分：测试、验收与实施阶段

## 43. 测试分层

- 组件：身份只读、版本、采用/导出分离、素材权限、陈列变化摘要；
- 合同：OpenAPI 变化阻断构建，前端不私自扩展作用域；
- E2E 内容闭环：登录 → 输入 → 任务 → 成品 → 局部修改 → V2 → 采用 → 导出；
- E2E 陈列闭环：库存 → 双杆方案 → 反馈右侧过厚 → 只改 C 区 → 切换 V1/V2；
- 隔离对抗：诱饵租户、品牌 A/B、私人素材、管理权限、应用边界；
- 可访问性与视觉回归：桌面/平板/移动、200% 缩放、键盘全流程。

## 44. P0 验收

1. 双应用边界清楚；
2. 当前身份始终只读可见；
3. 内容纵向闭环完整；
4. 陈列纵向闭环完整；
5. 版本自动保留；
6. 采用与导出分开；
7. 陈列反馈只局部修订；
8. 不显示后台轨迹；
9. 服务端验证权限和隔离；
10. 桌面与移动端都可完成核心任务。

## 45. 实施阶段

### P0：统一工作壳与两条纵向闭环

应用切换、身份条、内容输入/成品/版本/采用/导出、陈列库存/双杆方案/反馈新版、最近任务、基础搜索、响应式、最小错误态和两条 E2E。

### P1：正式工作台

我的成品、局部选区修改、版本恢复、连续系列、内容计划、私人素材与偏好控制、入驻与就绪、品牌表达、人员账号角色、组织素材、完整搜索、稳定后再加流式。

### P2：真实采用证明后的扩展

公共素材、多人共创、更多陈列载体、多模态、ERP/POS 连接器、实时协作、独立部署档位。未立项不预建空菜单、空表、空 API 和占位权限。

# 第十一部分：主要风险与推荐裁决

## 46. 主要风险

| 风险 | 控制 |
|---|---|
| 聊天再次成为视觉中心 | 成品生成后右侧自动扩展 |
| 内部编号进入用户界面 | 全部改为业务语言 |
| “保存”语义混乱 | 自动版本；采用、导出、系列写入分开 |
| 双应用混用 | 顶层固定，冲突提示一次切换 |
| 管理台变后台大杂烩 | 只保留四个真实管理模块 |
| 临时页面与 SPA 双轨 | P0 起单一 React 工作壳 |
| 搜索跨作用域 | 搜索前硬过滤，缓存按作用域隔离 |
| 陈列示意被误解为实景 | 明确“结构化执行示意” |
| 过早扩未来模块 | 注册式扩展，未立项不占位 |

## 47. 建议一次确认的前端决定

1. **FE-R01：**P0 起采用单一 React + TypeScript 工作壳；
2. **FE-R02：**左侧上下文、中间协作、右侧成品；成品生成后成为主视觉；
3. **FE-R03：**普通用户不见 P1—P5、DM01、Save、ReuseAsContext 等内部语言；
4. **FE-R04：**所有版本自动保留，采用、导出、系列写入、私人素材保存分别表达；
5. **FE-R05：**管理台从账户入口进入，只保留四个真实模块；
6. **FE-R06：**返回用户默认继续最近，双应用选择页主要用于首次和主动切换；
7. **FE-R07：**全局搜索优先成品与业务对象，不以聊天历史为中心；
8. **FE-R08：**本交付的 13 个高保真状态作为开发与视觉验收共同基线。
