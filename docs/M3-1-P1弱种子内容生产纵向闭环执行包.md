## 三、里程碑结果

交付一个可在本地启动、由普通用户直接使用的最小软件：

> “折线之间品牌母账号·抖音”的当前授权操作人“总部内容运营甲”，以“总部零售/服务专家”内容角色进入内容生产入口 → 自然输入一句没有明确提出穿衣问题的生活弱种子 → 系统从可信会话确定租户、组织、品牌、发布账号和内容角色 → 主动发现一个对目标受众有意义的穿衣选择问题 → 只调用当前获准的最小品牌资料与领域知识 → 生成一份符合 P1 合同、可直接制作的完整文字成品 → 用户以自然语言讨论并要求修改 → 系统形成不可覆盖的新版本 → 用户主动保存需要跨任务复用的成品。

冻结的主验收种子来自 P1 黄金包 1：

> 下午开完一个挺正式的会，转身就拎着电脑去接孩子，站在校门口才发现自己还穿着会议那一身。

黄金成稿只校准业务语义和质量边界，禁止固定回放、句式照抄或相似度验收。至少再用一条不在黄金文件中的弱种子证明系统会形成实质不同的选择问题与成品，不能只对一个输入成立。

## 五、最小软件范围

### 1. 小白友好的自然入口

- 顶层明确当前是“内容生产”，不要求用户填写租户 ID、品牌 ID、P1 或产品编号；
- 支持普通自然交流；用户只说 `hello` 或没有生产意图时，友好回应，不强制生成内容产品；
- 内容生产意图成立且上下文足够时直接交付一个择优成品，不用问题清单折磨用户；
- 只有缺少真正不可替代的信息时追问一次；
- 用户可以继续说“更口语一点”“改成单人正对手机讲”“保留判断但压缩重复”等，系统修改同一任务并生成新版本；
- 用户只看到对话、内容概要、完整成品和清楚的保存动作，不看到提示词、知识 ID、评分、主张谱系、上下文包、路由或内部运行轨迹。

不建设正式设计系统、管理后台、通用聊天产品、多应用导航或平台发布能力。界面朴素可用即可。

### 2. 用户最终看到的是完整成品

先用一段自然语言告诉用户：

- 这篇抓住了什么；
- 为什么采用这个方向；
- 用户可以怎样进入制作状态。

随后交付一份连贯、可直接制作的文字成品。媒体组成按本次表达需要自然包含标题/封面文字、完整台词或解说、画面与动作、必要镜头顺序、关键字幕、环境声或音乐方向、制作提示和自然收束。不是每次机械输出同一字段清单，但不能只交付提纲、观点、分镜或半成品。

首期只生产文字，不生成、编辑、匿名化或渲染图片、视频和音频。

### 3. P1 业务合同

- 不要求用户先提出明确穿衣问题；内容生产意图成立后，系统可以从弱种子创造本篇选择问题；
- 价值锚点是目标受众的真实选择机会，不是商品露出；
- 成品必须形成有依据、有边界、可采取下一步行动的条件性选择结果；
- 表达可用直接回答、选择方法、生活情境、口播、演示或其他最适形式，故事化不是默认要求；
- 系统可以充分增加节奏、比喻、幽默、动作、画面和情绪，但不能创造用户、商品或实际事件事实；
- 用户真实经历的身份、关系、事件结果及其明确时间、地点、数量不得被改写；
- 品牌战略、账号和内容角色必须实质影响观察角度、关系姿态与表达，不能只在结尾贴品牌名；
- 正确但像通用 AI 稿、课堂讲义、企业宣讲或拙劣隐性卖货的成品不合格。

### 4. 可信上下文

上下文至少按以下顺序形成：

1. 从服务端可信会话确定 `Tenant` 和当前操作人；
2. 验证其是否可以使用当前发布账号；
3. 确定代表组织、品牌、发布账号、具体内容角色和目标受众；
4. 读取当前有效的品牌战略版本；
5. 只读取当前任务需要、且已获准使用的最小领域资产；
6. 形成内部上下文后再调用生成器。

请求体不得通过填写或修改 `tenant_id`、`brand_id`、组织、账号等参数切换身份。模型不能自行扩张作用域。内部可以记录实际使用的版本和资产引用，但不保存模型隐藏推理。

### 5. 最小持久化与版本

只物理实现当前闭环直接消费的关系，不把 25 类机械变成 25 张表，也不为未来模块建立空壳。物理结构可以合并，但必须准确表达当前需要的：

- 租户、组织、用户与有效账号使用关系；
- 品牌、当前品牌战略、发布账号、内容角色和目标受众；
- 获准使用的最小领域资产及其状态；
- 业务任务、内容成品、不可覆盖的成品版本；
- 每次生成运行及追加式必要事件；
- 当前任务自然修改所需的最小会话关系。

初次生成形成 V1；自然修改形成 V2，V1 保持可读取。每次模型生成都留有运行记录，包括用户未采用的结果，但企业用户界面不能据此浏览他人的内部生成轨迹。

普通日常输入和成品默认不进入长期记忆。只有用户主动保存，或明确指定继续某条历史内容时，才在原账号/品牌/租户作用域内跨任务复用。关闭私人协作偏好不删除系列计划、成品或必要业务留痕；M3-1 不需要实现完整私人偏好管理。

### 6. 真实模型与测试替身

定义一个很小、可替换的生成器边界。测试可以使用明确命名的确定性替身来验证交互、作用域、持久化和版本，但：

- 页面必须清楚显示当前是否为“离线测试模式”；
- 替身不得冒充任意自然输入的真实创作能力；
- 不得把固定黄金成稿作为替身返回值；
- 关闭 M3-1 前必须完成一次经授权的真实模型端到端调用，并由人工按 P1 合同审阅结果；
- 至少一条新的弱种子必须在离线测试模式关闭时，从同一个用户页面经过同一可信会话、作用域编译、真实模型、版本保存和页面呈现路径完成；另写脚本直接调用供应商 API 的 smoke test 不算 M3-1 关闭证据；
- 没有凭据或付费调用授权时，继续完成所有离线工作；依赖真实调用的最终验收保持未通过，必要时按 `BLOCKED` 诚实记录；
- 禁止索取、打印、提交或写入日志的明文密钥，禁止未经授权的付费调用；
- 首期最多接一个真实供应商，不建设多模型路由、自动降级平台或供应商矩阵。

## 六、风险相称的验证

### 必须通过

1. 一个文档化命令可以启动本地软件；
2. 黄金包 1 弱种子无需技术表单即可得到一份完整 P1 文字成品；
3. 一条新的弱种子会形成实质不同的选择问题与成品，证明不是固定回放；
4. 用户自然修改后形成 V2，V1 未被覆盖且服务重启后仍可读取；
5. 只有用户主动保存的成品可在新任务中被明确复用；未保存日常内容不会自动进入新任务上下文；
6. 普通闲聊不被硬路由为 P1；
7. 用户不能用请求参数切换租户、品牌、组织或账号；
8. 用一个最小外域夹具证明当前上下文只能读取服务端确定的租户与品牌范围；相似名称、历史池、缓存和系统化诱饵攻击留给 M3-2；
9. 每次生成和版本变化有必要留痕，失败不产生半写入版本；
10. 未确认候选资产不能进入正式上下文；被使用资产都有明确作用域和版本；
11. 本地统一检查入口在没有模型密钥时也能运行确定性测试；
12. 真实模型输出经过一次人工语义审阅，不能用字面或向量相似度替代。

### 验证纪律

- 先做与当前闭环直接相关的单元/合同测试，再做一条稳定端到端运行；
- 高风险隔离、版本不可覆盖和失败原子性必须反证；普通展示细节不追求穷举；
- 日常生成只做快速必要自检，完整反事实或黄金对照只用于回归；
- 达到验收后停止，不增加“检查检查器”、全量评测平台、覆盖率竞赛或空远程 CI；
- M3-1 只提供稳定本地检查入口。远程 Linux CI 在 M3-2 承载已经存在的真实检查，不先建空流水线；
- 本地合成会话与最小外域夹具只证明当前软件路径，不证明生产认证、生产级强隔离或真实企业采用；
- 进入 `REVIEW` 前进行一轮独立、范围内的反证审查；若已有同等审查证据，不重复同一无争议检查。主控随后进行最终验收。
