# 笛语第一阶段总体技术架构与路线图映射基线

- 状态：已确认
- 确认日期：2026-07-22
- 最新补充：2026-07-22（第一阶段容量、数据库强制隔离与统一模型）
- 适用范围：第一阶段共享 SaaS 的设计、软件纵向闭环、部署、租户入驻与试点验收

## 一、唯一架构结论

笛语第一阶段采用：

> **六个逻辑责任层，一套模块化单体，一套部署在 ECS 上的共享应用，以及独立于平台部署的新租户入驻生命周期。**

六层是责任边界，不是六套服务；路线图中的软件里程碑是贯穿必要层次的纵向闭环，不采用“先做完 KE、再做完 RAG、再做完编排、最后才做 UI”的横向瀑布。

第一阶段取消 DIFY。应用核心通过一个极薄模型适配器直连 `DeepSeek-V4-Flash`。DIFY 不再是候选运行路径、数据真源或部署依赖；首期也不建设多模型路由或自动模型降级平台。实际 API 的 `model` 参数由部署配置承接，不在架构文档中猜测供应商线上的字符串。

## 二、六个逻辑责任层

| 层 | 责任 | 第一阶段边界 |
|---|---|---|
| L1 交互 UI | 内容生产、陈列搭配、自然对话、成品阅读、修改、保存和入驻入口 | 不向普通用户暴露 ID、提示词、检索过程、内部评分或运行轨迹 |
| L2 笛语应用核心 | 可信身份、租户/组织/品牌/账号/角色解析，任务与价值路由，版本、系列、保存和事件 | 系统级、品牌级、人设级编排均在这里合成，不拆成三套 Agent |
| L3 作用域上下文编译器（RAG） | 先确定合法作用域，再检索 SQL/全文及获准素材，组装模型最小上下文 | RAG 不等于向量库；第一条闭环优先精确查询，质量缺口成立后才增加派生检索 |
| L4 KE 与权威数据 | 候选导入、专业复核、激活、版本和适用范围；SQL 保存权威对象、关系、权限、版本与事件 | 领域通用、品牌专属、组织和私人来源保持显式范围；候选不能冒充正式知识 |
| L5 模型生成适配器 | 接收应用核心已裁减的上下文，调用 `DeepSeek-V4-Flash` 并返回生成结果 | 不掌握身份、权限、知识真源、成品版本或事件；不预建多供应商路由 |
| L6 部署与运行 | ECS 应用、共享 PostgreSQL、必要对象存储、HTTPS、密钥、日志、模型调用观测、备份和回退 | 首期不建设微服务、Kubernetes、消息队列或独立工作流平台 |

## 三、一次生成的固定顺序

```text
登录身份
→ 服务端解析 tenant_id
→ 解析组织、品牌适用范围、发布账号/岗位身份和私人所有者范围
→ 判定内容生产或经营陈列任务
→ 在合法范围内检索并编译最小上下文
→ 通过小型适配器调用 DeepSeek-V4-Flash
→ 保存 GenerationRun、不可覆盖的成品版本和必要事件
→ 只向用户呈现自然说明与完整成品
```

`tenant_id` 必须来自服务端可信会话，客户端不得通过请求字段切换租户、品牌、组织或账号。所有租户拥有的数据、缓存键、后台任务、检索投影、对象路径、调用记录、导出和恢复都必须承接相同作用域；外部模型只接收过滤后的上下文。

租户根隔离采用双保险：应用层每次访问都显式携带服务端派生的合法作用域；共享 PostgreSQL 对所有租户拥有表启用并强制执行行级安全（RLS）。请求应用角色不得拥有 `BYPASSRLS`、不得作为表所有者绕过策略；迁移/管理角色与请求应用角色分离。缺少可信租户上下文时数据库访问必须失败关闭，插入和更新也必须通过同租户 `WITH CHECK`。RLS 负责兜住租户边界，租户内部的品牌适用、组织权限、发布账号和私人所有者范围仍由应用核心继续收窄，不为第一阶段建设通用策略引擎。

## 四、共享 ECS 运行形态

“一套共享应用”表示一套租户中立的代码和部署，不表示永远只能有一台虚拟机。第一阶段按预计 2—3 个、最多 5 个租户，全平台合计不超过 200 个已启用账号设计；这是账号容量冗余，不等于 200 个并发生成请求。首期从一个应用部署单元开始，真实负载超过当前边界后再横向扩容，不改变租户模型。

- ECS 承载模块化单体应用；权威关系存储统一采用当前项目独立的共享 PostgreSQL 数据库和共享 schema，租户拥有表以 `tenant_id NOT NULL`、应用显式作用域和强制 RLS 共同隔离，不采用每租户 schema。
- 每条租户拥有记录显式承接 `tenant_id`；租户内继续按品牌适用范围、组织权限、账号角色和私人所有者收窄。
- 领域通用知识可以是系统公共范围，但租户不能修改系统真源；品牌、组织和私人数据不得因共用基础设施而串用。
- M5-3 首次启用私人/组织媒体素材持久化时才接对象存储，应用本地磁盘不作为长期素材真源。
- 不为品牌、区域、门店或账号分别部署应用、数据库、代码分支或 CI/CD。
- 只有未来真实合同明确提出独立网络、数据地域、专属实例或 SLA 时，才形成专属部署方案。

## 五、KE、RAG 与检索演进

1. M0—M2 已冻结业务问题、25 类语义核、来源和使用边界。
2. M1-1 的 62 个文件和 243 条候选是用户提供的首批 KE 输入。M3-1-C1 已将它们完整登记为系统公共目录，但全量登记不是生产激活。
3. M3-1 当前只按适用条件激活 `B-TPO-001`、`C-COMMUTE-001`、`D-DIRECT-001`、`D-CRAFT-001` 给 P1；H 类通过可信会话、RLS、上下文编译和最小合同承接，不进入普通知识上下文。不另建“全库先完成”阶段。
4. SQL 是权威真源；全文、摘要、向量和图查询都是可重建投影，不能替代租户隔离、对象 ID、版本、权限和审计。
5. 第一阶段不预选向量库。只有黄金任务或真实检索证明精确查询/全文稳定漏召回时，才裁决是否增加向量；GraphRAG 继续后置。
6. 未来如启用 pgvector 或其他向量引擎，向量投影必须承接权威来源作用域：租户专属向量使用 `tenant_id NOT NULL` 并受 RLS，系统通用向量进入独立公共投影或明确的 `system_public` 作用域，不把两者混成可空租户字段。查询先由服务端合成“获准公共范围 + 当前租户及必要品牌/组织/私人范围”，再执行相似度 Top-K；只在全库召回后做二次过滤不合格。外部向量引擎须使用不可由客户端覆盖的集合/命名空间和元数据过滤。

## 六、模型 API 与历史基础设施的复用边界

`M3-1-D1` 正式关闭：第一阶段不使用 DIFY，统一使用 `DeepSeek-V4-Flash`；离线确定性替身只是确定性测试模式，不是第二套架构。

用户已授权：真实模型调用需要既有凭据时，执行端可从 Windows `W:\home\diyu\笛语领域通用数据库` 对应的 WSL 只读路径 `/home/diyu/笛语领域通用数据库` 安全查找并复用与 `DeepSeek-V4-Flash` 兼容的凭据和连接参数，不再重复询问“复用还是新建”。执行端不得输出、提交、记录或复制明文密钥；若安全检查后凭据不存在、无效或无法调用已确认模型，只阻塞真实外部调用，并继续全部不依赖工作。

构建和测试阶段调用 `DeepSeek-V4-Flash` 不设人为成本预算或调用次数闸门，可按当前开发、黄金任务与人工语义审阅需要调用；不得以费用为由停工或重复请求预算。确定性 CI 仍使用测试替身，真实模型调用仍只服务当前里程碑，不扩张为无验收目的的批量生成或压力消耗。

同一历史仓库中的 ECS 与基础设施资料可以作为现场参数来源，但不得成为当前项目业务真源：

- Docker 运行方式和 PostgreSQL 引擎可以在当前里程碑确有消费者、现场兼容且隔离成立时复用；
- 复用 PostgreSQL 基础设施必须使用当前项目独立数据库或等效硬边界、独立数据库用户/凭据和独立迁移，不读取旧表、旧租户或旧业务数据；
- 现有向量设施的存在不构成启用理由；未来如评测触发，必须使用当前项目独立集合、作用域元数据和凭据；
- DIFY 即使仍在历史环境运行，当前项目也不得连接其工作流、数据集、记忆或运行状态；
- 禁止继承旧项目的品牌、组织、账号、知识、素材、向量、提示词、事件、用户记忆、Schema 含义和完成状态。

这属于“复用基础设施能力，不继承历史项目”。

## 八、贯穿性质量与商业运行闸门

### 1. 黄金基线是持续闸门

- M3-1 形成一个本地统一黄金回归入口，不按固定文案或向量相似度判定。
- M3-2 将确定性的合同、作用域、版本、路由和失败检查接入远程 CI。
- M3—M5 每次新增产品、修改 API、上下文、知识激活、模型适配或核心提示策略，都必须重跑受影响黄金套件。
- 真实模型语义质量不进入每次提交的确定性 CI，原因是结果非确定性且依赖外部服务，不是成本预算；模型、提示、知识编译或主要生成策略实质变化时，按需运行黄金任务并进行一次人工语义审阅。

### 2. 品牌入驻验收

- 每个真实租户在 M6-3 检查品牌调性符合度、专属知识缺口及是否混入“折线之间”或其他租户表达。
- 一个真实单品牌租户是第一阶段最低试点验收锚点；预计接入 2—3 个、最多 5 个租户是容量和渐进入驻边界，不成为等待客户资料的施工阻塞。
- 从第二个真实租户起，同一 M6-3 合同增量启用跨租户诱饵和串味检查。若这些租户均为单品牌，仍不构成“同一租户多品牌”真实运行证明；只有真实多品牌租户出现时才验证 D-017 的该部分。

### 3. 模型调用用量、限流与失败恢复

- 主要容量约束在 `DeepSeek-V4-Flash` 调用链，而不是 5 个租户对 ECS/数据库形成的计算压力。M5-4 至少按租户、任务/会话记录请求、供应商用量、延迟、限流和失败，不建设完整计费系统。
- 共享环境对模型调用设置简单的全局/每租户并发与速率上限，遵循供应商 `Retry-After` 或有界退避；达到重试上限后清楚失败，不产生半写入成品版本。第一阶段不为削峰建设分布式消息队列或调度平台。
- `DeepSeek-V4-Flash` 价格低不取消用量和错误观测，但构建测试不设成本预算闸门。生产限制服务于供应商速率边界、稳定性和防滥用，不由模型供应商账单替代笛语自己的租户归集。

### 4. 非功能验收

- M5-4 对共享发布候选按“最多 5 个租户、合计 200 个已启用账号”完成相称的启动、正常请求、小规模突发、模型限流、备份恢复和版本回退验证；200 个账号不转写成 200 并发压测。
- M6-2 根据试点实测峰值并发、数据量和合同恢复目标做定向验证；没有明确合同依据时不虚构企业级 SLA，不建设多地域、双活、分库分表、读写分离、分布式缓存/队列或大规模压测平台。

### 5. 素材与合规边界

M5-3 是媒体素材首次进入持久化边界，但不得把它升级成重型媒体审批或清理流水线：

- D-030 的文字规则从 M3 第一次生成起即生效：生成文字默认移除具体顾客个人标识并使用非识别性称谓；首期不建设独立 PII 扫描服务；
- 私人图片和视频保持原件、仅在原合法作用域作为创作参考，系统不编辑、打码或匿名化媒体；
- D-020 对已知或用户声明的可识别真人未成年人原件，在持久化或外部模型传输前拒绝；不建设年龄识别、监护人审批或自动打码；
- 平台红线约束生成文字和制作建议，不把素材入口变成监管审批系统。

## 九、技术选型与 API 契约冻结点

- M3-1：形成唯一、版本化、机器可读且可审阅的最小业务 API 契约；冻结客户端不能改变可信作用域的边界。
- M3-1-D2：已关闭为当前项目独立 PostgreSQL、共享 schema、应用显式作用域与强制 RLS；只建立当前纵向闭环直接消费的表和策略。
- 首次真实模型调用前：安全读取已授权既有凭据并确认 `DeepSeek-V4-Flash` 的实际连接参数；不重新讨论 DIFY 或模型供应商。
- 向量检索：仅在已有检索稳定无法通过黄金或真实任务时裁决。
- M5-4：确定 ECS 账号、区域、域名、共享数据库、对象存储、密钥托管和回退的实际参数。
- M6-2：根据真实试点合同冻结规模、SLA、恢复目标和租户专属差异。

## 十、第一阶段边界与停止线

本路线图只覆盖第一阶段。ADR-009 公共素材、ADR-010 完整机器自评与数据飞轮、ERP/POS 连接器、多模态生成、真实多品牌运行及其他成长能力均在 M7 之后另行立项，不能成为当前阶段的隐含待办。

第一阶段以“折线之间”推动软件构建，以一个授权真实单品牌企业作为最低试点验收锚点；平台按预计 2—3 个、最多 5 个租户和合计 200 个已启用账号保持可用。合成夹具、自动检查、共享 ECS 上线和真实品牌分别证明不同层级，均不得互相冒充。M7 验收成立后立即停止扩展。
