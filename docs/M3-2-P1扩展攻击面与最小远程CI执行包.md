## 三、范围内交付

### 1. 扩展最小诱饵攻击面

在现有测试入口内补足以下高信号反证，不复制测试框架：

- **跨租户**：另一租户放入与当前种子、品牌名或历史正文高度相似的诱饵，当前会话仍不得读取、复用或发送给生成器；
- **同租户跨品牌**：兄弟品牌拥有更“诱人”的账号、人设、受众、已生成正文或已保存版本，当前品牌上下文和自然承接不得串用；
- **同品牌跨账号/用户**：另一账号或另一操作人的历史版本不得成为当前用户“接着上一条”的候选，也不得通过直接版本 ID 读取、修改、保存或复用；
- **历史池**：当前作用域内的合法上一条可以继续；未授权范围的相似内容即使更新、更近或文本更匹配也必须排除。用户主动指定当前作用域的合法版本时仍能工作；
- **上下文反证**：使用可观察的测试生成器捕获最终 `GenerationInput`，证明发送到模型边界的品牌战略、账号、内容角色、受众、前情和资产只来自当前合法作用域；
- **失败边界**：复用现有超时、限流、无效返回和数据库事务证据，补足当前真正缺失的一条即可；失败仍记录为失败且不产生内容版本。

“缓存攻击面”按当前事实处理：项目尚未引入业务内容缓存，不为测试缓存而新增缓存。只需保证连接或进程复用不会残留前一作用域；未来真正增加缓存时，再把可信作用域纳入缓存键并扩展同一闸门。

### 2. 让既有本地入口可在普通 Linux CI 运行

- 保留 `make lint`、`make typecheck`、`make golden` 三个既有入口，不创建平行脚本体系；
- 将本地 PostgreSQL 启动脚本从固定机器路径收敛为可发现当前 Linux 已安装 PostgreSQL 工具的实现，仍使用项目私有临时数据目录和独立测试角色；
- 本地重复执行安全，已有测试 PostgreSQL 在运行时不误报失败；
- 不引入 Docker、Testcontainers、Kubernetes 或远程测试数据库。

### 3. 建立一个最小远程 CI

在 `.github/workflows/` 下只建立一个工作流：

- 触发：向 `main` 推送、针对 `main` 的 pull request，以及人工触发；
- 环境：单一 Ubuntu Runner、单一受支持 Python 版本，不建版本矩阵；
- 时长：单作业设置 `timeout-minutes: 10`；正常目标应显著低于该上限，超时直接失败，不靠延长时间掩盖脚本问题；
- 步骤：检出、安装最小依赖、运行 `make lint`、`make typecheck`、`make golden`；
- CI 使用确定性替身，不读取或要求 DeepSeek、ECS、生产数据库或其他密钥；
- 不上传内容正文、数据库文件或运行日志作为长期 artifact；
- 不建设 Codecov、Sonar、SAST 平台、发布流水线、CD、分支保护编排或通知机器人。

工作流文件存在不算完成；必须在公开远程仓库至少成功运行一次，并能回指提交和绿色结果。
