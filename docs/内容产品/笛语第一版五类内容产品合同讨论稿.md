## 1. 本文解决什么问题

D-004 已经确认第一版内容生产采用五类内容产品：

1. P1 真实穿衣问题与购买决策；
2. P2 商品真相与专业取舍；
3. P3 品牌/账号观点与生活叙事；
4. P4 一线问题回应与本地生活；
5. P5 造型灵感与视觉叙事。

本文把五个名称展开为可讨论的产品合同，重点回答：

- 用户提出什么问题时进入这一类；
- 系统必须具备哪些品牌、商品、账号和场景上下文；
- LLM 可以增加什么、不能增加什么；
- 商品怎样自然参与，内容怎样保持品牌相关；
- 输出什么成品，怎样进入账号连续记忆；
- 什么叫合格，哪些典型内容必须判为不合格。

本文不同时冻结首期平台与媒体范围、数据库物理表、最终评分阈值、模型供应商和自动发布能力。

### 1.1 规范优先级

不能把“最新口头意见”和“正式 ADR”笼统放在同一层。发生冲突时按以下方式处理：

1. **显式取代性决定**：最新用户决定明确说明替代哪个旧决定/ADR，并已写入项目记忆或形成更新 ADR；在补 ADR 期间标为 `owner_confirmed_adr_pending`，立即按新决定执行，不能让旧 ADR 继续支配实现；
2. **当前有效 ADR**：未被显式取代的已接受 ADR；
3. **用户已确认、待补 ADR 的新增规则**：不与有效 ADR 冲突，已经写入项目记忆，状态为 `owner_confirmed_adr_pending`；
4. **D-031 候选规则**：状态为 `d031_proposal`，必须经过裁决才能成为实施规范；
5. **评测假设、历史资料与匿名专家分析**：只能用于验证和启发，不构成生产规则。

用户的新表述只有在明确新增或取代某项规则并完成记录后，才改变规范状态；不能凭“时间更新”静默修改旧 ADR。旧战略资料继续提供方向和素材，不得覆盖后续正式裁决；《讨论1.txt》中的字段、枚举、文档拆分和实施顺序也不能自动升级为用户决定。

## 2. 五类产品的稳定边界

| 产品 | 从哪里出发 | 首要完成的任务 | 不能被哪一类替代 |
|---|---|---|---|
| P1 真实穿衣问题与购买决策 | 人的身体、场景、顾虑和约束 | 帮助选择 | 不能退化为 P2 的商品参数讲解 |
| P2 商品真相与专业取舍 | 真实商品、系列或设计决定 | 解释商品 | 不能用泛行业科普冒充品牌商品真相 |
| P3 品牌/账号观点与生活叙事 | 一句生活/工作/组织种子 | 建立人格 | 不能退化为无品牌去向的泛生活鸡汤 |
| P4 一线问题回应与本地生活 | 品牌与地域生活交集，以及可选的评论、一线观察和现实信号 | 经营关系 | 不能退化为总部稿加城市名，也不能吞并全部本地化内容 |
| P5 造型灵感与视觉叙事 | 商品组合、品牌审美、场景和视觉资产 | 创造欲望 | 不能混成内部经营陈列执行方案 |

最短定义固定为：

> **P1 帮助选择，P2 解释商品，P3 建立人格，P4 经营关系，P5 创造欲望。**

## 3. 五类共用运行合同（D-031 已确认语义）

> **本章效力说明**：`SH-01`—`SH-12` 承接 D-031 第 1—9 项、D-014 运行化口径及相关既有 ADR；`SH-13` 承接 ADR-011。第 1.4 节是稳定索引，本章及第 9 章是索引指向的展开正文。本文没有仍在悬空的 `d031_proposal`。邻近出现的字段名、代码枚举、对象身份、评分阈值、采样比例、未来多模态能力和物理结构，必须按第 1.3 节分别归入产品合同验证、评测假设、D-016/D-032 或后续独立扩展，不能因共享合同关闭而自动确认。

### 3.1 `[SH-01]` 一个共享内容编译器，五种目标合同

P1—P5 不是五套彼此复制的数据、知识库和 Agent 流程，而是共享内容编译过程的五种目标合同：

`顶层应用选择 → 应用内自然对话回合 → 当前工作意图 → 回应交流 / 主动引导与计划建议 / 最小追问 / 形成任务种子 → 应用内产品/成果路由 → 相应上下文编译与业务合同 → 后台自检与成品交付`

第一版 UI 在逻辑顶层明确提供“内容生产（对外）”与“陈列搭配（对内）”两个应用入口。自然对话是进入应用后的交互方式，不等于让系统在一个无边界入口中反复猜测当前属于哪个应用：选择内容生产后，正式内容成果才在 P1—P5 内路由并生成 `ContentItem`；选择陈列搭配后，成果按经营陈列合同生成 `DisplayArtifact`，不进入 P1—P5。若用户明确要求的成果与当前应用相冲突，只提示一次切换，不静默混用合同，也不新增重型意图识别或二次 P5/陈列选择器。

进入内容生产分支后，P1—P5 共享上下文解析、关键事实防错、版本事件和权限外壳；差异在于优先优化的价值、承重信息、必需输出、失败条件和连续状态。这里的“共享编译器”是**逻辑合同**，不等于必须只有一个 Agent、一次 LLM 调用或一条巨型 Prompt；实现可以由若干检索、生成、校验和评审步骤共同完成，但不得为五类复制五套相互漂移的基础语义。

普通用户仍可只说一句话，或只选择商品、评论、素材或已有任务。系统在当前权限内自动组装适用上下文：合同字段必须在**当前主产品、当前主张和当前交付真正适用时**解析完整，不等于全部候选字段都适用，更不等于让普通用户填写长表单。软信息不足时使用可见假设或条件化候选；只有缺少无法删除、无法条件化且会改变结论的承重信息时，才提出最少量追问。

#### 3.1.1 `[SH-02]` 自然协作，必要收敛

笛语不是无边界闲聊机器人，而是**以自然对话为交互方式的服装零售知识协作与业务成果助手**。用户先在逻辑顶层进入内容生产或陈列搭配，进入任一应用后的默认交互都是连续自然对话而不是生成表单；每一轮用户输入首先是该应用内的一个对话回合，不自动等于 `task_seed`。系统在会话中维护一份用户可随时纠正的“当前工作意图”摘要，只保留当前目标、希望收敛到的判断/成果、已接受约束、已否定方向和仍待解决的问题；它是运行时逻辑视图，不要求新增本体类、状态机或数据库表。

系统每轮只选择一个最有助于推进当前协作的动作：回应或继续交流；提出一项有依据的引导、挖掘或策划建议；提出一个确实会改变结果的最小追问；或在用户已经提出可执行成果要求、明确接受共同形成的方案、选择现有任务继续修改时形成正式任务。用户明确说“先聊聊”时不得抢跑生成；用户已经明确要求执行且必要信息足够时，也不得重复确认。

成品交付不强制结束当前协作。只要用户继续用自然语言讨论、质疑或修改，系统就沿当前成品、已接受约束和版本前情继续解释或续改，不要求填写表单、重新描述全部需求或点击“确认完成”；同一成果的这些修改继续进入既有任务版本链。系统可以用一句自然话告诉小白用户“哪里不像你、哪里拍不了，直接说”，但不得为了维持对话连续追问。用户不再输入、明确停止或切换话题时，系统安静等待或转入新话题，不自主发送消息；“持续协作”不等于无限闲聊、后台自治或自动记入长期内容记忆。

构思、策划、选题和规划属于知识协作及任务准备能力，不新增第六类内容产品。系统可以兼容核心业务以外的话题，但不以无限延展闲聊为产品目标；与当前工作有关的洞察可以继续讨论并成为创作种子，无关话题可以正常回应，但不主动扩建成新的通用服务，不自动生成 `ContentItem`、`DisplayArtifact`，也不自动写成组织知识、品牌事实或账号公开前情。**交互可以自然外延，协作必须能够收敛；第一版正式能力承诺仍只落在内容生产和经营陈列两个已注册应用。**未来模块通过 D-024 的版本化应用合同注册，不为尚未立项的模块预建空流程。

每段实质协作最终应收敛为四种结果之一：问题已经回答；形成一项判断/方案；形成可执行业务任务；或由用户明确暂存、以后再继续。这只是交互结果，不是四态工作流。系统不得为了收敛而强迫每轮出稿，也不得在已经足够形成结论时继续无休止发散；适当时只提出一个最短下一动作。

#### 3.1.2 混合主动式协作与新账号冷启动

笛语采用**混合主动式交互（mixed-initiative interaction）**：任务既可以由用户明确提出，也可以由系统基于当前授权上下文发现内容空白、潜在线索或连续性机会，主动提出选题、系列、内容组合和生产节奏建议。系统主动性是帮助用户发现“现在适合做什么、为什么、下一步怎样开始”，不是替用户决定经营目标，也不是在后台自行生成、采用或发布正式内容。

系统建议、策划方向和计划草案均属于**待接受的协作候选**，不自动成为 `task_seed`、`ContentItem`、采用事件或发布指令。用户明确接受其中一项、要求把它做成成果，或选择计划中的某项继续执行后，才通过既有 `accepted_conversation_proposal` 或 `selected_input` 形成任务种子；不新增“自主任务”状态或另一套任务系统。

租户开通后的第一个内容业务里程碑，是依据 D-014 通过行业原型预填、少量具体取舍对照和简短对话，形成一份系统起草、企业修改并确认的最低可用品牌或企业/店铺内容战略基线。企业不需要从空白处撰写“我们怎样观察世界”或填写技术表单；确认只表示判断草案准不准。确认基线后，系统才依据真实组织规划最小账号/角色方案、首批账号和账号冷启动方案；真实存在多组织或多账号协同时再形成 IP 矩阵。确认前可以导入资料、试聊和预览样稿，但草案不能静默成为正式品牌或企业/店铺人格。

一个没有内容历史的新 `PublishingAccount` 首次进入内容生产时，不应只面对空白输入框。系统应先在权限内复用已有的品牌定位、`PublishingAccount + ContentRole`、目标人群、区域/门店语境、平台、当前商品/活动、可用素材和既有生产能力；只对会实质改变方案而又无法从现有上下文取得的信息逐项作最小追问，然后给出一个可修改的账号冷启动方案。首版冷启动方案至少表达：

1. 账号当前应优先建立的受众认知或关系目标；
2. 有明确品牌关联路径的少量内容方向、系列或栏目建议，并说明分别调用 P1—P5 中哪类能力；
3. 一个与实际素材、人员和制作能力相称的首个生产周期建议，包括优先次序、建议节奏或发布窗口、所需输入和缺口；
4. 一个可以立即接受并开始执行的最短下一任务。

策划和生产发布计划是内容生产模块对 P1—P5 的**上层编排能力**，不是第六类内容产品。用户明确要求制定计划，或接受系统提出的计划草案后，计划本身可以形成可保存、修改和版本化的应用级规划成果；但计划中的单项建议仍不是已启动任务，只有用户选择执行时才分别形成内容任务。计划中的时间只是建议发布窗口，不等于系统已取得发布权限或承诺自动发布。该规划成果是否需要独立持久化对象及其最小字段由 D-016 决定，本项不预建日历引擎、提醒系统、自动排程器或自治运营 Agent。

主动建议必须来自当前可用的品牌/账号预设、商品与活动、内容连续状态、用户主动提供的评论/观察/素材，以及具备版本和有效期的平台趋势信息；缺少这些依据时可以提出探索问题，不能生成套用所有品牌的“万能三十天计划”。每个建议至少应说明预期价值、品牌关联、为何现在适合、需要什么输入和接受后的下一步，系统也应允许用户忽略、否定或调整，不以未接受建议形成负向责任或强制流程。

#### 3.1.3 模型 API 是语言智能，不是笛语系统

只接入 GPT 或 Claude API，可以获得自然语言理解、对话、推理、生成、工具调用和结构化输出能力，但不会自动获得笛语所需的用户身份、租户隔离、品牌上下文、长期私人记忆、任务边界、品牌/商品资料与人工任务输入的忠实使用、P1—P5 合同、内容版本和留痕。OpenAI 的 Conversations API 可以持久保存会话状态；Claude Messages API 默认无状态，而其 Memory Tool 也明确由应用执行和持有存储。两者都不能成为笛语私人记忆或业务状态的唯一真源。能力依据见 [OpenAI Conversation state](https://developers.openai.com/api/docs/guides/conversation-state)、[Claude API primer](https://platform.claude.com/docs/en/claude_api_primer) 与 [Claude Memory Tool](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)。

具体实现选型尚未裁决。一种待评估的最小候选路线是一个主对话模型、一个轻量对话协调/上下文编译层、现有两个业务执行入口以及既有权限、版本、事件和轻量记忆存储；这不是当前定论。无论最终实现怎样，模型只能负责通用理解、推理和表达，笛语仍须持有权限、作用域、记忆、事实、任务路由、业务合同和结果留痕。商用 SDK 可以在未来作为会话、工具调用、流式响应或运行追踪的复用方案，但本轮不确定供应商、SDK、UI 套件或运行框架，也不授权安装、PoC 或双生产线建设。本段状态为 `implementation_option_deferred`，不影响 D-031 第 1 项业务语义成立。

### 3.2 任务种子与作用域（`SH-02`）

只有对话已经形成可执行任务时，才创建任务种子。它不是用户最后一句话，而是当前成果所需的有效意图和输入引用快照：

```yaml
task_seed:
  seed_origin: direct_user_request | selected_input | organization_task | user_requested_continuity_followup | accepted_conversation_proposal
  target_application:
  payload_or_refs:
  intent_snapshot:
    requested_outcome:
    accepted_constraints:
    rejected_directions:
  supplied_by:
  usage_scope:
```

`seed_origin` 只描述任务如何启动；商品、评论、观察、活动简报、对话片段和旧内容等具体载荷由 `payload_or_refs` 引用，不能把触发方式、素材类型、业务对象和连续关系塞进一个互斥枚举。外部信号、系统发现的内容空白或连续机会只能触发建议，不能单独创建任务；`accepted_conversation_proposal` 只有在用户明确接受共同讨论形成的方向后成立，`user_requested_continuity_followup` 也要求用户明确继续。`target_application` 引用当前真正注册的应用合同，第一版只有内容生产和经营陈列；普通交流没有目标应用，也不创建空任务。`user_seed` 只是在 `seed_origin = direct_user_request` 时对原始用户输入的兼容视图，不再是所有产品的强制原句字段。上述逻辑要素已经由 D-031 第 1 项确认；代码字段名、值域和物理存储仍由实现设计与 D-016 决定，不要求对应一张新表。

同一成果下的缩短、改写、换开头、调整语气、改变拍摄资源或表现方式和继续深化，默认进入同一任务的版本链；目标应用、主要成果或独立交付对象发生实质变化时，才建立新任务。媒体改写从用户当前选中的源版本继承主要产品价值、核心结论、品牌/账号身份、用户前提、商品绑定和承重事实引用，只记录本次真正变化的制作条件、表现偏好、时长或平台包装，并生成一份新的完整版本；不为此新增抽象 `ContentCore`、改写 Agent 或形式矩阵。`intent_snapshot` 保存执行时已经达成的有效理解，不复制全部聊天记录，也不保存模型隐藏推理。用户最新明确修正覆盖当前会话中较早的软性要求，但不能覆盖权限、当前硬事实、品牌硬规则和平台红线。

系统只在缺失信息会改变应用路由、主要成果、不可替代的权限/硬事实，或将导致不可逆/外部动作时追问；其他创作性空白以简短可见的假设继续。追问默认只问当前最关键的问题，不把内部合同字段转成用户表单。

以下是 D-031 第 2 项已经确认的最小作用域规则：

| 信息来源 | 默认作用域 | 不得自动扩大为 |
|---|---|---|
| 日常对话、临时生活/工作陈述和本次内容种子 | 当前任务 | 长期记忆、个人素材、账号人设、系列历史或正式知识 |
| 用户主动保存到个人素材区的内容素材 | 当前自然人的个人素材区，可在其以后任务中按需调用 | 组织知识、品牌事实、其他人的私人记忆或其他工作区素材 |
| 用户显式建立的账号/系列/组织配置 | 明确指定的账号、系列或组织范围 | 从日常聊天自动推断出的更大范围 |
| 明确表达或反复稳定出现的协作偏好 | 当前自然人的私人协作偏好 | 生活事件、人物履历、品牌/商品/经营事实或发布账号公开前情 |

任何内容都不能因为被检索、生成、采用或使用一次就自动进入长期私人记忆。`not_applicable` 与 `unknown` 在创作权限之外的精确运行后果仍属于第 4—5 项候选，不因第 2—3 项确认而提前冻结。

#### 3.2.1 `[SH-03]` 私人交互记忆只作为受控上下文来源

系统可以调用当前自然人使用者的轻量私人协作偏好，以延续其协作方式、明确偏好和反复纠正。它只回答“系统怎样更好地帮助这个使用者生产内容”，不保存日常生活事件、人物关系或本次内容种子，也不等于 `creator_private` 个人素材区、发布账号的 `AccountContinuityMemory`、`SeriesState` 或组织知识。

私人交互记忆默认仅供当前使用者本人调用；组织管理员、同事、总部、区域和其他账号不能查看或借用。它可以影响系统怎样交流、怎样准备候选，但进入品牌成品后仍须服从当前发布账号、内容角色、品牌规则和业务合同，不能让操作人的个人偏好覆盖品牌表达。D-031 第 7 项已经确认：本人可以查看、纠正、删除或关闭私人协作偏好，也可以开启一次完全不读取、不写入私人偏好的临时会话。

第 1—2 项已经确认私人协作偏好可以作为本人专属上下文，并且日常内容不得自动写入；第 7 项进一步确认用户控制语义。“关闭”只停止后续读取与写入，保留偏好以便本人以后恢复；“删除”才移除相应偏好内容。偏好摘要的物理存储、提炼算法、写入时点、冲突合并、删除实现、保留期限和账号停用后处置继续由 D-016 与数据策略决定。特别是，ADR-012 对个人素材原件的退出接管不能自动扩张到私人协作偏好。

#### 3.2.2 两类持久化来源，一种轻量连续投影

“企业知识库 + 每个账号的私人记忆”作为逻辑分层方向成立，但不能实现为两套同等重型的知识工程。还必须把这里的“账号”限定为当前租户内可追溯的**自然人使用者**，而不是可能由多人运营或发生交接的 `PublishingAccount`。企业知识库保存可治理、可共享、可版本化的组织知识；私人交互记忆保存系统怎样与本人协作的非权威摘要。私人记忆改善的是长期对齐、意图命中和交互连续性，不会提高基础模型本身的知识、推理上限或事实正确率。

首版采用“**两类持久化来源 + 一种运行投影**”，不建设第三套记忆平台。企业知识库是组织知识和规则的权威来源；私人记忆只在“如何与本人协作”的范围内有效，不是商品、经营或品牌事实的权威来源：

| 语义部分 | 回答的问题 | 已确认的逻辑边界 |
|---|---|---|
| 企业/品牌知识库 | 组织知道什么、当前允许怎样表达 | 是组织共享、可治理的知识与规则来源，不等于个人聊天记忆 |
| 自然人私人协作偏好 | 系统怎样更好地与这个人协作 | 仅在本人范围内提供非权威协作偏好，不保存日常内容，不成为品牌、商品或经营事实 |
| 发布账号/系列连续投影 | 这个对外账号或系列已经讲过什么 | 从内容版本、事件和经确认的连续状态形成，不复制成第三套知识权威 |

当前会话和“当前工作意图”属于短期运行上下文，不算第四种长期记忆。模型供应商的会话历史、Memory Tool 或产品记忆可以作为实现参考或适配能力，但都不得成为上述数据的唯一真源。

ChatGPT 当前将已保存记忆、历史聊天引用和临时聊天分开；Claude 当前将记忆摘要、项目范围记忆、历史聊天检索和无痕聊天分开。笛语只借鉴这种“精选记忆、作用域隔离、用户可控、临时不记忆”的产品原则，不照搬供应商的内部提炼算法、文件形式或会话编号。依据见 [ChatGPT Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq)、[Claude chat search and memory](https://support.claude.com/en/articles/11817273-use-claude-s-chat-search-and-memory-to-build-on-previous-context) 与 [Claude Memory Tool](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)。

第 2 项确认“可长期保留什么”，第 7 项确认用户怎样控制；以下列表中的提炼、召回和存储路线仍是 D-016 的**实现候选**，第 7 项用户控制语义已经确认：

1. 用户明确表达或反复稳定表现出的协作偏好，可以更新紧凑偏好摘要；只针对本次的修改和单次生活内容不自动长期化；
2. 单次行为不推断为稳定偏好；系统可以继续观察和被用户随时纠正，不为此建立复杂候选记忆状态机或频繁追问；
3. 新会话只取当前任务相关的少量摘要，不回灌全部历史聊天；需要寻找某段旧讨论时，可由用户明确触发历史检索，首版不以全历史自动 RAG 为前提；
4. 用户可以查看、修改、删除、关闭私人记忆，并可发起不读取也不写入长期记忆的临时会话；
5. 当前明确指令覆盖较早的私人偏好；权限、当前硬事实、品牌/平台硬规则以及当前 `PublishingAccount + ContentRole` 合同始终约束最终业务成品；
6. 首版复用现有关系库和访问控制即可，不新增私人向量库、记忆图谱、专职 Memory Agent、全历史夜间归纳任务或双模型记忆同步。只有实际规模和召回评测证明关键词/结构化筛选不足时，才在同一隔离范围内增加向量召回。

当前已冻结私人协作偏好的用途、本人边界、内容素材须主动保存、与组织知识/个人素材/发布账号连续性的分离、不得覆盖权限/硬事实/品牌合同，以及查看、纠正、删除、关闭和临时会话控制。具体提炼、写入、召回、删除实现及存储路线继续作为 D-016 与数据策略的候选，不提前冻结表名、JSON 结构、摘要长度、召回阈值、物理清理方式或账号停用后的生命周期。

#### 3.2.3 D-031 第 1 项正式裁决（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 1 项当前唯一规范口径。

1. **交互形态**：笛语采用自然对话、必要收敛的知识协作方式。每次输入先是对话回合，不自动成为任务；普通交流可以结束于回答、判断/方案或明确暂存，只有形成可执行业务成果时才创建或延续任务。
2. **混合主动性**：任务既可由用户直接提出，也可由系统在已授权上下文内主动发现机会并提出选题、系列、策划和计划建议。系统必须帮助意图不清或不知道从何开始的用户，但不得替用户确定经营目标、采用内容或执行发布。
3. **建议、计划与任务分离**：未被接受的建议不是任务。用户明确接受建议、选择计划项或要求形成成果后，才创建 `task_seed`。内容策划和生产发布计划是 P1—P5 的上层编排，不是 P6；计划项在被选择前不是已启动任务，建议发布窗口也不是自动发布指令。
4. **新账号冷启动**：新 `PublishingAccount` 不以空白输入框或长问卷开始。系统先复用当前品牌、账号/内容角色、目标人群、区域语境、平台、商品/活动、素材和实际生产能力，仅追问会改变方案的关键缺口，再给出受众/关系目标、少量品牌相关方向、首个生产周期建议及一个可立即接受的最短任务。
5. **任务种子合同**：`task_seed` 是执行时的最小有效意图与输入引用快照，而不是用户最后一句话。它在逻辑上至少表达启动来源、目标应用、输入/对象引用、希望形成的成果、已接受约束、已否定方向、提供者和使用范围；普通交流不创建空任务。上述为语义合同，不冻结代码字段、表或 JSON。
6. **任务连续与追问**：同一成果的缩短、改写、调整语气和继续深化进入同一任务版本链；主要成果、目标应用或独立交付对象实质变化时建立新任务。只在缺口会改变路由、主要成果、不可替代的权限/硬事实或不可逆/外部动作时作一个最小追问，不能把内部合同变成长表单。
7. **条件性执行策略**：内容生成默认 `generation_mode = focused`，交付一个主候选；用户明确需要结构不同的多路线时使用 `exploratory`。竞争力是所有输出的共同质量要求，不设置 `competitive` 档。`distribution_strategy = unified | localized | hybrid` 只用于多账号矩阵任务；每个最终成品仍归属一个 `PublishingAccount`。
8. **私人记忆调用边界**：私人协作记忆可以作为任务形成前后的本人专属上下文，但不是组织知识、私人素材箱或发布账号/系列连续状态，也不能覆盖权限、硬事实和品牌合同。用户查看、纠正、删除、关闭和临时会话控制已由第 7 项确认；自动提炼、物理写入/删除、冲突合并、保留期和停用后处置留给 D-016 与数据策略。
9. **首期应用边界**：开放对话不是第三个应用模块，也不增加第六类内容产品。第一版正式成果仍只路由到内容生产和经营陈列；未来业务模块继续通过 D-024 注册，不为未知能力预建空流程。
10. **实现中立**：GPT、Claude 或其他模型 API 只提供通用语言智能，不能替代上述笛语业务合同。商用 SDK、模型供应商、UI 套件、Agent 运行框架和部署组合仅为以后可评估的实现选项，本项不作选型、不授权安装或 PoC，也不要求双供应商生产线。

本项没有提前确认：前提模式与创作权限（第 3 项）、私人记忆的用户控制（后由第 7 项确认）、规划成果和任务种子的物理对象/字段（D-016）、平台媒体合同（D-032），以及任何 SDK/模型/UI/运行框架选型。

#### 3.2.4 D-031 第 2 项正式裁决（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 2 项当前唯一规范口径。

1. **日常内容默认临时**：日常对话、生活/工作陈述、人物关系、内容种子以及本次生成中的创作内容，默认只服务当前任务，不进入长期私人记忆。
2. **内容素材主动保存**：只有用户主动保存到本人个人素材区的内容素材，才允许在其未来任务中按权限再次调用；使用、生成、采用或高表现都不能代替这一步主动保存。
3. **只自动维护协作偏好**：系统可以从用户明确表达和反复稳定的纠正中，自动维护一份轻量私人协作偏好，用于理解怎样更好地交流和生产内容；不得把单次事件、人物履历、故事细节或创作增益塞入该偏好。
4. **以行动代替盘问**：系统应优先结合当前任务、品牌/账号上下文和私人协作偏好直接产出；只有缺失信息会改变主要成果、触碰权限/硬事实或导致不可逆/外部动作时，才追问当前最关键的一个问题。
5. **不同记录不混用**：个人素材区、私人协作偏好、组织知识、发布账号/系列连续状态和 ADR-010 要求的生成留痕是不同语义。第 2 项不取消生成审计或内容连续性，也不让管理员因此取得私人偏好或个人素材。
6. **实现继续后置**：偏好摘要的用户查看、纠正、删除、关闭和临时会话语义后由第 7 项确认；怎样提炼、保存、物理删除及处理账号停用，由 D-016 与数据策略决定。本项不新增记忆图谱、私人向量库或专职 Memory Agent。

### 3.3 `[SH-04]` 运行身份、业务控制与角色适配

ADR-005 已冻结实际操作人、总部组织性 `ContentRole`、`PublishingAccount` 及总部业务控制关系不能混为同一身份；现有租户/组织/工作区决策也要求保留渠道主体隔离。将这些原则推广成通用的 `represented_organization_id`、`business_controller_id` 及首期派生规则，不属于 D-031 已确认内容，继续由 D-016 与试点权限合同验证；不能反向声称全部字段都已由 ADR-005 冻结。

- `actual_operator_id`：谁实际使用系统，只用于身份、安全和审计；
- `workspace_id`：任务在哪个隔离工作区运行；
- `represented_organization_id`：这条内容代表哪个经营主体发声；
- `business_controller_id`：谁在系统内管理、交接该业务内容资产；
- `publishing_account_id`：最终以哪个自媒体账号表达；
- `content_role_id`：以哪个组织性角色/账号人设表达。

本稿建议首版试点不支持代运营或跨组织业务控制，因此 `business_controller_id` 从工作区所属经营主体派生，并要求与 `represented_organization_id` 相同；该首期相等规则不是 D-031 已确认内容，继续由 D-016 与试点权限合同验证。未来若支持代运营，两者可以通过版本化合同分开，不能在语义上永久合并。

还必须分开两种“范围”：

1. **账号操作与组织资产授权**属于硬访问控制，未授权时阻断任务；
2. **内容角色表达适配**描述该人设通常适合怎样表达，通常进入质量判断而非现实资格审查；只有独立 `PolicyRule` 明确规定的禁项才成为硬规则。

因此，店长内容角色偶尔讲家庭生活不因“缺少现实专业资格”自动被阻断；系统判断的是内容是否符合账号配置、品牌规则和当前任务，而不是调查真人权威。

#### 3.3.1 对外人设与对内组织身份的确认口径（2026-07-20）

P1—P5 是对外内容生产的上层成果合同，定义成品必须为受众完成什么价值，不是账号类型、人设模板或权限角色。每条实际对外成品及其黄金任务必须明确代表组织、发布账号和组织性内容人设，并由人设的真实权威、服务对象、稳定语气和身份禁区约束表达；实际操作人只是账号使用者，不能未经配置进入成品人格。

对内零售经营陈列不沿用上述对外 IP 人设。内部成果首先依据总部、区域代理或加盟门店的**组织身份**和当前培训、指导、执行或复核等**业务任务**编译；岗位只在确有必要时说明谁负责制定、执行或复核，不为内部场景另造自媒体人格、粉丝关系或一套“内部 IP 人设库”。因此：

- 对外内容：`发布账号 + 组织性内容人设 + 一个 P1—P5 主合同 + 当前任务`；
- 对内陈列：`组织身份 + 业务任务 + 必要业务条件`，继续进入独立经营陈列合同。

这一区分只明确运行语义，不新增 `InternalContentRole`、两套人格路由器或新的物理字段；最终对象仍由 D-016 按最小充分原则决定。

#### 3.3.2 P1-B 已确认：三级配置分层与知识/人设解耦（2026-07-20）

内容生成采用三个独立配置范围，但只由一次运行编排合成：

1. **系统全局基线**：定义应用模块、P1—P5 合同、领域通用知识、行业角色模板、共享能力、租户隔离和平台能力目录；
2. **品牌租户叠加**：定义品牌 IP 矩阵、品牌表达/受众、账号框架、内容组合重点、品牌专属知识以及特殊能力/数据模型开关；
3. **账号/人设叠加**：定义渠道层级、矩阵角色、默认主题、选材优先级、视角、语气、连续定位及身份/权威边界。

用户本次任务可以改变品牌和人设的软默认选材，但不能突破租户/品牌、原始资料/私人素材、身份/权威等硬边界。领域通用知识默认供所有租户任务调用；品牌专属正式知识默认供该品牌全部有效账号调用，不按总部、区域、终端分权。人设是默认知识编排器，不是知识访问控制器。

租户入驻先按 D-014 形成企业确认的最低可用品牌或企业/店铺内容战略基线，再使用服装行业角色模板按真实组织规划最小账号/角色方案，最后由企业定制、确认和逐步激活首批账号。具有真实三级渠道时可以形成总部—区域—门店矩阵；单店集合店或买手店不被要求补造层级。行业原型与账号角色模板都只是起点；企业确认的基线和人设版本优先，模板升级和一次临时任务都不得静默覆盖。首期不建设三套 Agent/服务、复杂继承 DSL 或逐知识权限引擎，具体配置对象、版本字段和路由快照留给 D-016。

### 3.4 `[SH-05]` 用户前提、前提模式与创作权限

用户输入、选择或配置的人设、经历、人物关系、生活事件和叙事前提，在明确任务、账号或系列作用域内，作为系统可以充分使用的运行时前提。系统不调查其外部真实性，也不以“证明人生经历”作为创作前置审批；但必须忠实保留用户给定的承重信息。

运行时可以用以下内部标签区分表达口径，但不得把它们变成要求普通用户填写的表单：

- `source_presented_as_actual`：任务来源希望按已经发生或现实存在来表达；不表示系统外部核验；
- `hypothetical`：条件性、假设性场景；
- `fictional`：明确虚构或情景创作；
- `not_applicable`：纯商品解释、抽象选择判断等不存在叙事真实性模式的任务。

创作权限可以映射为三档内部语义：

- `faithful`：保持全部承重前提，只做语言、结构和非情节性增益；
- `enriched`：保持承重前提，允许增加不改变因果和结果的表达、内心独白、非事实性动作设计、一般性情绪、场景、节奏和转折；
- `dramatized`：允许完整戏剧化组织，但不得冒充逐字记录、实际门店案例或已核验现场。

不为 P1—P5 分别建设一套创作权限矩阵，统一使用三条默认规则：

1. 用户按真实经历讲述的生活或工作种子，叙事层默认 `enriched`，系统不为普通创作空白反复追问；
2. 所有产品中的身份、关系、事件结果、用户明确给出的时间/地点/数量，以及商品、价格、库存、活动、交易、健康、法律等承重事实始终 `faithful`；
3. 完整虚构或戏剧化改写只有在用户明确要求，或成品明确作为假设、情景演绎、虚构内容呈现时才能使用 `dramatized`，不得冒充逐字记录、真实顾客案例或已核验现场。

`fictional + faithful` 仍可成立，表示忠实于用户给定的虚构设定而不继续扩写；真实经历也可以被改编为情景演绎，但必须满足第 3 条，不能让受众误以为新增对白、人物或结果也真实发生。

系统应从种子中提取用户可修正的 `premise_invariants`，只记录实际出现的身份、关系、事件结果、指定时间/地点和数量。所谓承重信息，是指删除或改变后会改变人物身份、责任归属、事件因果、交易结果、健康/法律含义、商品结论，或公众对“实际发生了什么”的理解。幽默、比喻、内心独白、气氛、节奏、转场和明确属于拍摄设计的非事实性动作不逐项确认；准备改变承重信息或进行完整虚构/戏剧化改写时才需要用户明确要求。

用户生活前提不能覆盖品牌方已经提供的商品资料、品牌禁区和平台硬规则。用户手工输入的“本店当前库存 3 件”“今日活动价”等动态经营信息，只作为当前任务的输入使用：系统不核验现实真值、不进行来源判级，也不把它自动写入品牌/商品知识库。用户在同一任务中明确修正时，以最新明确输入更新任务；系统只负责不擅自改写该输入，最终内容和方案仍由使用者复核。

#### 3.4.1 D-031 第 3 项正式裁决（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 3 项当前唯一规范口径。

1. **真实种子默认充分增益**：用户按真实经历讲述的生活或工作种子，系统默认可以不追问普通创作细节，直接增加幽默、比喻、内心独白、情绪、节奏、转折、画面和拍摄设计。
2. **关键前提不得改写**：系统不得改变人物身份、人物关系、事件结果，以及用户明确给出的时间、地点和数量；也不得借叙事增益改写商品、价格、库存、活动、交易、健康、法律等承重事实。
3. **创作新增不得冒充事实**：系统新增的对白、动作、人物、情节或结果，如果会影响公众对“实际发生了什么”的理解，不得作为真实记录表达。
4. **完整戏剧化需要明确依据**：完整虚构或戏剧化改写，必须由用户明确要求，或在成品中明确作为假设、情景演绎或虚构内容呈现；不得冒充真实顾客故事、逐字对白或已核验现场。
5. **统一规则而非五套矩阵**：上述规则同时适用于 P1—P5。叙事表达可以丰富，承重事实始终忠实；不再为五类产品建设五套重复权限体系。
6. **内部标签不是用户负担**：`source_presented_as_actual/hypothetical/fictional/not_applicable` 与 `faithful/enriched/dramatized` 可以作为运行时语义标签，但不要求用户选择，系统应从任务自然判断；具体代码枚举和存储仍由 D-016 决定。

### 3.6 关键来源的最小内部记录

第 4 项不建设独立“运行时主张谱系”系统。面向用户的标题、脚本、正文、分镜和制作说明均不显示“用户前提、事实、专业判断、创意表达”等标签，也不要求用户理解来源分类。

第一版只复用现有结构完成最低限度的内部防错：

1. `task_seed` 已经保留用户本次提供的前提和输入引用；
2. `required_fact_refs` 只复用成品实际调用的既有品牌、商品或规则资料引用；用户手工输入的库存、价格、活动等经营信息已经随 `task_seed` 留痕，不另建快照对象或来源判级；
3. 用户选用的个人/组织素材继续使用已有素材引用；
4. `GenerationEvent` 继续记录本次运行、模型、版本和使用过的关键输入引用；
5. 系统专业判断使用条件性表达，创意表达服从第 3 项边界；二者不建立逐句记录，也不得自动回写为商品、品牌或经营事实。

因此，第一版不新增 `claim_lineage` 对象、三维主张枚举、逐句标签表、主张图谱或全文证据标注。只有后续真实错误样本证明仅凭任务种子、事实引用和生成事件无法定位问题时，才针对具体高风险主张增加更细映射。

#### 3.6.1 D-031 第 4 项正式裁决（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 4 项当前唯一规范口径。

1. **成品零负担**：任何对外内容和普通用户交付中都不显示“主张谱系”及其内部标签。
2. **不建独立子系统**：第一版取消完整运行时主张谱系、逐句标注、三维枚举、独立表和知识图谱。
3. **只复用必要输入引用**：既有品牌、商品或规则资料沿用已有引用；人工经营信息只保留本次任务输入，不另建事实快照、来源等级或冲突记录；不为幽默、比喻、节奏和一般创作表达建账。
4. **复用现有留痕**：用户前提由 `task_seed` 承载，事实来源由 `required_fact_refs` 承载，素材与生成过程由既有引用和 `GenerationEvent` 承载，不新增重复记录。
5. **不允许反向污染**：系统专业判断使用条件性口吻，创意表达不得自动写回事实知识库；这通过上下文分层和生成/检查规则实现，不要求逐句持久化。
6. **按证据升级**：只有真实错误和评测证明最小记录不足时，才增加目标明确的局部主张映射，不为假设风险预建重型能力。

### 3.7 人工经营信息的使用边界

第一版内容生产和零售经营陈列都以用户手工提供、选择的业务资料为主要任务输入，不接入 ERP/POS，也不把连接外部经营系统作为试点或验收条件。笛语只生成内容、搭配、陈列和策划建议，不创建交易、不扣减库存、不计算账务，也不拥有判断现实库存、价格或活动真值的职责。

运行时只遵循以下最小规则：

1. 用户手工提供的库存、价格、活动、营业状态和门店空间等信息，只在当前任务中按其原意使用；系统不得擅自猜补、改写或扩大到其他门店、区域和时间；
2. 系统不比较总部、区域、门店或未来 ERP 谁更权威，不建立 A/B/C 事实等级、新鲜度评分、冲突状态机、对账机制或库存真值服务；
3. 用户在同一任务中明确更正信息时，视为更新任务输入，不属于系统裁定现实真相；若两项互相矛盾的信息都是完成用户明确要求所必需，系统只询问“本次按哪一个输入生成”，不得自行判真；
4. 库存、价格或活动并非当前成果必需时，直接省略这些信息并继续生成，不弹出补数流程；只有用户明确要求“基于本店现有库存”“写明今日价格/活动”等数据依赖成果且没有提供最低输入时，才追问这一项；
5. 没有经营数据时仍可输出不依赖该数据的内容、通用搭配、概念陈列或培训建议，但不得自行声称“实时库存”“系统核验”或“已执行”；
6. 将来若某租户选择接入 ERP/POS，笛语也只消费外部系统提供的数据作为任务输入；库存核算、交易记录、冲突处理和最终真值仍由外部经营系统负责，连接器能力另行立项。

上述规则不削弱品牌方已经提供的商品资料、品牌规则和平台规则对创作的约束；它只取消对动态经营信息建设交易系统式真值判断。

#### 3.7.1 D-031 第 5 项正式裁决（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 5 项当前唯一规范口径。

> **人工经营信息是任务输入，不是笛语需要裁定的交易真值。第一版不接 ERP、不做来源判级或冲突裁决；非必要信息缺失时省略并继续，只有用户明确要求的数据依赖成果缺少最低输入时才询问用户。**

### 3.8 `[SH-07]` 创作增益、生成模式与成品交付

系统遵循：

> **最大有效增益，最小无依据核心改写。**

系统应主动增加语义理解、受众张力、专业解释、内容结构、开头、节奏、转折、比喻、幽默、互动和视觉叙事价值。只复述种子、没有增加专业、叙事、结构或品牌价值，属于“过度谨慎失败”，即使没有写错事实也不合格。

结构不同的多路线是系统能力，不是每次默认输出：

- `focused`：默认；系统内部选择一条主路线，只交付一个最佳候选；
- `exploratory`：交付 2—3 条结构明显不同的路线，让用户探索方向；

两个模式使用相同的权限、事实、品牌和前提不变量；`focused` 不等于低增益，`exploratory` 不伪造确定冠军。**内容竞争力是所有模式的共同质量目标，不另设一个暗示默认模式质量较低的 `competitive` 档位。**系统可以在 `focused` 内部比较必要候选后只交付一个版本，但不得因此承诺爆款。所有生成尝试继续按 ADR-010 留痕。

系统对用户只提供**与本次任务相匹配的完整成品包**，不建立面向用户的“发布候选 / 制作规格 / 审阅视图”三视图。“成品”不是单指一篇文案，也不限于脚本、分镜和造型说明；它可以按任务需要包含标题、正文、完整台词、口播/解说文本、字幕与屏显文字、镜头/图片顺序、拍摄与表演提示、造型、场景、道具、节奏、封面、互动、CTA，以及背景音乐方向、环境声、转场声和其他音效设计。以上是开放式组成清单，不要求所有媒体机械交付全部项目；具体组合由用户任务和 D-032 的平台/媒体合同决定。用户要可拍摄短视频方案时，完整台词或解说、必要分镜和声音设计都是成品组成部分，而不是后台运行信息。

为了让非专业用户先理解再执行，完整成品默认先用一小段自然语言作**成品导读**：说明这版想让受众得到什么、为什么当前表达方式适合这个品牌账号/平台/现实制作条件，以及创作者最应抓住的一个执行重点，然后再给出完整成品。导读是交付体验的一部分，不是用户要填写的字段，也不是默认对外发布内容；不得出现 P1—P5 编号、路由、规则名称、资料分类、评分、提示词、模型、重试、事件标签或隐藏推理。用户明确偏好“直接给成品”时可以省略，不新增设置页、开关或另一套审阅视图。

D-032-B/C 进一步确认两类媒体的完整成品口径：视频采用[单一可执行观看链](../讨论纪要/2026-07-21-069-D032B视频完整成品合同候选.md)，图文采用[单一可执行图文阅读链](../讨论纪要/2026-07-21-070-D032C图文完整成品合同候选.md)。两者均以“符合本次资源条件的创作者无需补创承重内容或重新决定主要表现方式，即可进入当前制作环节”为完整标准，不以字段数量、固定时长、镜头数或图片数为标准。

图文成品应以自然语言完整交付标题与首图承诺、实际图片/图卡的作用与必要次序、必要的图中文字、可直接发布的完整正文，以及本篇真正需要的拍摄、选图、裁切和排版提示。单图、多图、实拍、图卡或混合形式均可按任务成立；不得退化为视频截图拼盘、长文机械切片、固定九页知识卡或“配一张氛围图”等承重占位符。已有素材不足时优先给出当前条件下可执行的补拍、替代或收窄方案，只有缺少不可替代的真实输入时才作一次最小追问；概念方案不得冒充实拍商品、人物或门店现场。P1—P5 共用这一媒体合同，不复制五条图文工作流；P5 仍必须由视觉让新的穿着可能成立，正文不能替画面补作业。

D-032-D 进一步确认，平台或媒体改写采用[共享语义主干＋目标语境一次重编译](../讨论纪要/2026-07-21-071-D032D三平台原生适配与最小媒体终验候选.md)：从用户指定或当前认可的源版本出发，保持唯一主要产品价值、品牌与账号角色、目标受众关系、关键前提、承重事实、核心结论和已确认前情，允许按目标平台、媒体及真实制作条件重做入口、信息顺序、语言/声画/图文分工、镜头/图序、自然时长、标题封面、配文和互动。每次只生成用户当前需要的目标版本；旧版及其他平台版本不覆盖，新版本不自动表示采用、发布或更新系列。语义主干只是一组运行时不变量，不新增重型 `ContentCore`、平台 Agent、流程矩阵或用户表单。

平台明确规则按来源、版本和生效时间调用；当期趋势和热点只作可选软信号，不得覆盖品牌、账号、事实和产品主价值，搜索失败时生成不依赖热点的完整版本。不得固化“抖音三秒反转、小红书教程化、视频号正能量”等平台套路。抖音、小红书视频、视频号和小红书图文各版本必须独立完整，不假设受众看过另一平台前情；只有对应发布账号与平台实际发布的内容才能形成该平台的公开“上一期”。

**完整成品不等于故事成品。**叙事、剧情和反转只是可按需调用的表现能力，不是内容质量的默认答案。系统先完成当前 P1—P5 主要产品价值，再结合种子与可用素材、发布账号/内容角色、现实制作条件、平台和时长，择优使用直接回答、专业解释、观点表达、纯口播、边讲边演示、问答/对话、旁白、视觉展示或多场景叙事等开放形式；没有真实事件或叙事价值时，不得为了“像自媒体内容”而硬造人物、冲突、情节和反转。制作资源与表现语法相互独立：单人轻制作既可纯口播，也可固定机位边讲边演示；多场景资源充足也不意味着必须故事化。

任务解析、上下文组装、资料引用、提示词、模型与路由、规则名称、质量分数、重试过程、被淘汰候选、来源分类、内部事件标签及模型隐藏推理，均不得混入用户成品。系统内部可以为质量检查生成临时结构，但不形成普通用户、企业管理员或内容运营人员可打开的“审阅视图”。D-032 只负责定义各平台成品由哪些可直接使用的部分构成，不恢复运行报告。

### 3.9 媒体实际性与媒体来源

事件前提、创作权限、媒体实际性和媒体来源是四个正交维度：

```yaml
premise_mode: source_presented_as_actual | hypothetical | fictional | not_applicable
creative_license: faithful | enriched | dramatized
media_actuality_default: concept_plan | staged_capture | actual_capture
media_actuality_overrides: # 仅记录少数例外镜头/素材
```

当默认值与例外实际包含多种状态时，系统派生 `aggregate_media_actuality = mixed`；`mixed` 不作为用户手工输入值，否则系统仍不知道哪部分是什么。真实经历可以使用 `staged_capture`，实际拍摄素材也可以服务虚构故事，二者互不证明。

“媒体来源”本身还应至少拆成创作来源与进入系统的路径。`captured/user_uploaded/synthetic/composite/stock` 不能冻结为一个枚举，因为一张用户上传图片同时可能是实拍、库存图或合成图。本文只保留“媒体来源与实际性不可混为一维”的业务区别；D-032 已冻结视频/图文文字成品合同，但不表示系统生成或编辑媒体文件。私人媒体的主动保存、本人复用和权限边界已由 D-030/ADR-017 极简确认，持久化身份由 D-016 决定。

### 3.10 `[SH-08]` 品牌相关性与反同质化

所有成品必须形成至少一条实际参与观点、判断、结构、证据、视觉秩序或服务动作的品牌相关路径。路径可以来自品牌立场/审美、目标人群关系、真实商品与专业、组织人物视角、本地服务关系、账号连续人格、既有栏目/系列或视觉符号。

账号人格资产可以成为 P3 的合法品牌路径，但不能自我认证：“由这个账号发布”本身不等于品牌相关。它必须实际调用该账号稳定的观察方式、价值立场、目标人群关系、系列前情或品牌审美/服务理解中的至少一项。

反同质化不要求每个观点全球唯一，也不按锚点数量机械通过。采用反事实敏感性测试：替换账号预设、商品、品牌立场、事件种子、本地语境或系列前情后，成品的观点、判断、证据、结构、视觉秩序、服务动作或 CTA 是否应发生实质变化。完全无需变化说明组合专属性不足；一个强锚点可以成立，多个弱标签也可能仍是模板。

反事实敏感性是硬验收原则，不等于每次生产请求都额外生成多篇对照稿：

- **日常生产**只做轻量检查：实际调用了哪些品牌/账号上下文、哪些结构受其影响，以及明显模板化、反重复和模型/规则的低成本自检；疑似模板在后台重写，不能仅凭概率性评委作权限/合规阻断，也不向用户展示内部评分；
- **黄金任务与回归评测**才真实执行同种子换账号、同账号换商品、同母题换地区等单变量变形，比较观点、证据、结构、视觉秩序和 CTA；
- **必要抽样**按 ADR-010 由独立分析消费者对生产事件执行，首版只保留事件与调用边界，不建设全量在线多路生成或完整数据飞轮。

#### 3.10.1 D-031 第 9 项正式裁决：日常轻量自检与离线完整评测（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节是第 9 项当前唯一规范口径。

> **日常生成只做快速必要自检；完整对照测试只用于黄金任务、版本回归和必要抽样，不能让每次内容生产都额外生成多篇对照稿。**

1. **日常生成**复用第 3.13 节已有后台合同门，不另建一套在线评测流程。快速自检只覆盖权限与隔离、明确硬禁项、主任务是否完成、用户关键前提是否保持、品牌/商品资料和本次输入是否被擅自改写、品牌相关路径是否成立、应用/产品路由是否正确，以及明显模板化和不可制作问题；
2. 普通质量问题优先在后台自动改写，不向用户展示分数、规则名、检查报告或淘汰候选。概率性模型评委只能辅助质量改写，不能单独成为账号权限、合规或发布责任的硬阻断；
3. **完整对照测试**才真实执行换品牌、换账号、换商品、换地区或换前提权限等单变量变形，只用于黄金任务、会实质影响生成行为的模型/提示词/检索/规则版本回归，以及 ADR-010 范围内确有诊断目的的必要抽样；
4. 不要求每次生产执行多路反事实生成，不建设用户可见评测面板，不保存模型隐藏推理，也不因抽样扩大企业对具体内容、私有素材或后台轨迹的查看权；
5. 抽样比例、相似度算法、具体阈值、模型评委组合和永久评分权重继续属于 `evaluation_hypothesis`，必须由黄金任务、成本与真实反馈校准，本次不凭空冻结。

#### 3.10.2 D-014 正式运行化口径：品牌内容战略基线与动态创作方向编译（2026-07-21）

状态：`owner_confirmed`。完整口径见[讨论纪要 046 确认稿](../讨论纪要/2026-07-21-046-P3D黄金选择与品牌创作方向编译候选.md)。

品牌不是文案外层的固定风格滤镜，而是内容选择机制。租户以企业确认、可版本化的方式维护目标人群关系、稳定选择与拒绝、观察方式、受众关系姿态、表达幅度、IP 矩阵及少量带理由的正反例；系统在每次任务中结合一个主要 P1—P5 合同、当前发布账号、受众情境、任务种子、可选连续前情、平台和制作条件，在当前上下文下择优形成一条创作方向并交付一个完整成品。

品牌基线决定“为什么值得说、怎样理解、绝不怎样利用受众”，账号决定“谁有资格从什么位置说”，目标受众决定“为什么值得看”，当前任务和媒体条件决定“这一篇具体怎样说”。风格、结构、情绪和形式是动态结果，不建立品牌类型到文风的静态映射；企业采用、高播放、评论或系统评分只可形成观察信号，不能自动改写品牌基线、账号人设或正式表达规则。

企业不被要求从空白处撰写抽象品牌哲学。系统先用可组合的描述性行业原型预填，再用少量带具体正反例的“像/不像、用/不用、A/B/都不对”选择和简短对话收集真实取舍，最后自动生成一份人话版完整草案供企业改几句并确认。真实品牌名称不作为正式模板标签；原型、系统推断和预览样稿不能代替企业确认。该确认是正式品牌内容生产和后续 IP 矩阵规划的前置业务条件，但不是法律签字、长问卷或复杂审批工作流。

本节不新增核心语义类别、独立 Agent、模型微调、风格知识图谱、品牌浓度分或日常多路反事实生成，也不向用户展示内部创作方向和检查轨迹。对象、表/JSON/投影和版本引用由 D-016 决定；D-021/ADR-016 已确认只有 ERP 品牌码时沿用企业表达，企业确认该品牌独立的 D-014 内容战略基线后才取得独立内容身份；平台、趋势和媒体表现由已关闭的 D-032/ADR-015 约束。

状态：`adr_confirmed`（ADR-014）。D-031 第 8 项确认“一个独立成品一个主要合同、共享能力不升格、明确要求独立成果时才分别路由”。为消除与原第 9 章的重复，本节只保留决策来源；唯一规范正文统一放在第 9 章 `[SH-09]`。`primary_content_product`、可选内部参与标签和成果/版本关系的物理表达继续由 D-016 决定，不能在此形成第二套路由规则。

### 3.12 `[SH-10]` 生成事件、连续状态与知识升级

#### 3.12.1 单篇与连续系列（第 7 项正式语义）

用户提出的核心区分成立，但正式术语不采用“即时性 / 连续叙事”。“即时性”在内容运营中还可能表示热点、节日或新闻等时效属性；一篇内容可以既有即时性又属于连续栏目。第一版建议使用两级极简语义：

```yaml
continuity_mode: standalone | series
series_kind: thematic | narrative # 仅 series 时适用
```

| 内容关系 | 含义 | 本次需要的上下文 | 后续默认行为 |
|---|---|---|---|
| `standalone` 独立单篇 | 本次独立成立的日常内容，不承诺下篇续接 | 账号/角色稳定设定 + 本次题材、平台、时长和其他任务要求 | 不创建系列前情；生成草稿不进入下次上下文，已确认/发布历史至多用于账号一致性和反重复，不自动成为故事前提 |
| `series + thematic` 主题系列 | 固定栏目、价值承诺或主题范围连续，但不要求人物剧情 | 上述内容 + 系列定位、主题地图、选题/分集计划、已覆盖内容和下一期方向 | 保持栏目承诺、观点和选题进度；不虚构人物关系或剧情线 |
| `series + narrative` 叙事系列 | 人物、关系、事件或剧情需要跨集延续 | 上述内容 + 现实/虚构口径、人物关系、时间线、剧情大纲、已确认前情、未完线索和下一集计划 | 生成前调用当前系列状态，并检查身份、关系、事件和情节一致性 |

因此，“连续系列”不必都有剧情。比如“每周一个真实穿衣问题”属于主题系列，需要栏目策划和分集计划，但不需要人物关系或剧情大纲；只有连续生活故事、门店人物故事等叙事系列才维护剧情连续性。

风格、人设和品牌表达通常来自账号/内容角色的稳定设定；题材、平台和时长默认属于本次任务或单集要求，只有用户明确固定时才成为系列默认值。系统不把这些前情一次性做成长表单：未选择系列时默认按独立单篇执行；用户选择已有系列、明确要求“下一集/继续上次”，或接受系统提出的系列策划后，才绑定或创建 `series_id`。

第一版只需一份轻量系列卡片，合并保存系列定位、品牌关联、稳定设定、主题地图或叙事大纲、分集计划、当前进度和未完线索；不为大纲、计划、人物关系和当前状态分别建设本体、图谱或多套记忆服务。账号人设一致性继续使用既有账号/角色配置，反重复可以查询已确认内容历史，不因此单独建设重型 `AccountContinuityMemory`。

#### 3.12.2 用户指令优先，系统不得反向绑架（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节与第 3.12.1、3.12.3 节共同构成已关闭的 D-031 第 7 项。

> **限制的是系统未经用户同意自动改写系列前情，不是限制用户怎样编排系列。**

系列状态可以由三类明确用户行为更新：用户直接制定或修改系列安排；用户接受系统提出的系列大纲/分集建议；用户明确把某个生成成品或其中要点纳入系列。用户可以随时要求跳集、插集、改序、指定某一集、跨越时间、放弃某条线、从指定集开始采用新设定，或把未在系统内生成的前情作为当前创作前提。系统必须接受并更新当前有效系列计划，不能要求用户先补齐前集、先生成成品或先证明已经发布。

生成、浏览、选择中间稿和导出本身不自动改变系列前情；但用户在任何这些动作同时明确说明“把这一点写入系列”“下一集按这个继续”时，该明确指令立即生效。已经发布或历史生成的版本继续留痕，用户的新编排从其指定位置起生效；系统不得为了保持旧计划而拒绝用户，也不得静默删除真实历史记录。

发布记录只是帮助系统区分“系统已知的公开历史”和“用户本次指定的创作前提”，不是创作通行证。系统可以在以下任一条件成立时使用“上一期”等承接表达：已有相应发布记录；或者用户明确要求/声明按该前情创作。用户声明即可作为当前内容前提，不要求截图或外部证明，也不自动反写为平台已验证的发布事实。两者都不存在时，系统不得自行编造受众已经看过某一期。

以下三层逻辑区别已经纳入第 7 项，但必须服从上面的“独立单篇 / 连续系列”主开关；其中“账号连续记忆”至多是从已确认内容历史形成的轻量投影，不表示第一版新增独立记忆库。运行语义按三层辨析：

| 连续层 | 保存什么 | 默认作用域 | 不能自动写入什么 |
|---|---|---|---|
| 账号/内容角色配置 | 稳定人设、长期立场、服务对象、表达禁区和经确认的长期经历 | 指定 `PublishingAccount + ContentRole` | 单篇生成新增设定、模型推测 |
| 账号连续记忆 | 最近采用/定稿的主题与观点、已经重复的表达、未回应问题和后续承诺 | 指定发布账号内部；平台公开承接另行判断 | 商品/经营事实、未确认人生履历、其他账号记忆 |
| 系列状态 | 系列内称谓、虚构/现实口径、情节进度、开放问题和下一集候选 | 指定 `series_id` | 整个账号长期设定、人物主数据、CRM 关系图谱 |

用户主动配置的人设、长期立场和账号设定通过显式配置进入第一层，不必等待某篇内容被采用。系列中的“婆婆、女儿、老顾客”等称谓和情节默认只进入第三层；只有用户明确把其中某项提升为账号长期设定时，才写入第一层。虚构内容可以进入明确标为虚构的系列状态，以保持续写一致，但不能升级为人物真实履历、顾客记录或门店历史。

`GenerationRun` 表示一次生成尝试，`ContentVersion` 表示具体输出版本，`GenerationEvent` 表示行为事实。第 7 项不再冻结 `selected/adopted/finalized/remembered/rejected` 的复杂自动记忆后果；首版只需要以下最小语义：

| 行为 | 对系列状态的后果 |
|---|---|
| 用户明确制定/修改系列，或接受系统的系列建议 | 按用户指定内容更新系列计划或当前前情 |
| 用户明确把某个成品/要点纳入系列 | 更新相应集次与后续所需前情 |
| 用户声明已发布，或未来连接器确认发布 | 增加系统已知的发布历史；不成为用户继续、跳集或改写系列的前置条件 |
| 仅生成、浏览、选择中间稿、导出或系统自行评分 | 只留行为记录，不自动改变系列计划与前情 |

具体按钮名称、事件名和物理结构由 D-016 决定，不为这些行为建设互斥生命周期状态机。`exported` 继续只表示导出，不自动等于用户采用；但用户可以在导出的同时明确要求把某些设定写入系列。

`SeriesState` 必须保存系列默认的 `series_premise_mode`。单集可以有自己的 `premise_mode`；系统不得自行把现实口径系列切成虚构系列，或反向把虚构系列改写为真实履历，但用户可以明确要求从某一集起改变叙事口径或重设系列。虚构称谓和情节只进入该系列，不能写入账号现实经历、顾客记录或门店历史。

生成内容、被采用、被定稿、获得高播放或良好反馈，都不能自动升级为商品事实、人物履历、经营事实、`KnowledgeStatement`、`ExpressionProfile` 或正式 `PolicyRule`。上述业务语义已经由 D-031 第 7 项冻结；三层连续性最终是对象、投影、表还是 JSON，继续由 D-016 决定。

#### 3.12.3 D-031 第 7 项正式裁决：私人协作偏好控制（2026-07-19）

状态：`adr_confirmed`（ADR-014）。本节关闭第 7 项最后一个待决语义。

1. **本人完全可控**：当前自然人使用者可以随时查看系统保存的私人协作偏好摘要，纠正其中任何内容，删除单项或全部偏好，也可以整体关闭私人协作偏好。
2. **关闭不等于删除**：“关闭”表示后续会话不再读取，也不再自动写入或更新私人协作偏好；原偏好保留，只有本人重新开启后才可继续使用。“删除”才移除被选中的偏好内容，使其不再被后续任务调用。删除动作的物理清理方式与最小技术记录由 D-016 和数据策略决定，不在 D-031 增加删除状态机。
3. **临时会话**：用户可以开启一次私人偏好临时会话。该会话从开始到结束都不读取、也不写入私人协作偏好，结束后不以本次互动更新偏好摘要。
4. **临时会话不是无规则会话**：不读写的对象只限私人协作偏好。当前租户/工作区权限、品牌与组织知识、用户明确选择的发布账号和系列前情、任务输入、平台规则以及 ADR-010 要求的业务生成与事件留痕仍正常生效；临时会话不能变成绕过权限、品牌合同或业务留痕的入口。
5. **不同资产互不连带**：关闭、删除私人协作偏好或使用临时会话，不删除、不停用也不改写系列计划、账号/角色配置、已经生成的内容与版本、用户主动保存的个人素材、组织知识或业务留痕。反过来，停用某个系列或删除一篇内容也不能静默改写私人协作偏好。
6. **本人边界不变**：私人协作偏好只服务该自然人本人；组织管理员、总部、区域、门店同事和其他账号使用者均不能查看、修改或调用。它不是企业知识、发布账号资产或个人素材箱。
7. **极简实现约束**：首版只需要一份紧凑偏好摘要和查看、纠正、删除、关闭、临时会话五类简单控制，不建设私人本体、记忆图谱、专职 Memory Agent、全历史自动 RAG 或复杂候选记忆工作流。具体表、JSON、摘要格式、提炼算法、保留期和账号停用后处置由 D-016 与数据策略决定。

至此，D-031 第 7 项完整关闭：内容连续性使用“独立单篇 / 连续系列 + 轻量系列卡片”，系列编排以用户明确指令为准；私人协作偏好由本人完全控制，且与系列、内容、素材、组织知识和业务留痕分离。

### 3.13 `[SH-11]` 后台检查、自动改写与最小必要提示

系统遵循：

> **成品优先，后台自检，运行轨迹不可见。**

第一版不建设面向用户的质量状态机、检查报告或候选资格标签。系统在交付前自行检查、自行改写，只把可直接使用的成品交给用户：

1. 普通内容质量问题，包括主任务没有完成、关键前提被改变、品牌关联不足、角色表达失配、内容同质化或不可制作，均先在后台自动改写，不把失败稿连同“需重写、合同不合格、已缩减”等状态交给用户理解；
2. 缺少非必要信息时直接省略并继续；只有缺少用户明确要求成果的不可替代输入时，才用一句话询问这一项，不展示缺失字段表或检查过程；
3. 未授权访问、租户隔离或明确系统硬禁项只能停止受限动作，并用一句话说明最短可行动条件；不得借此展示内部规则、评分或运行轨迹；
4. 只有某项无法通过自动改写或省略解决、且会直接影响用户能否安全或正确使用成品时，才附一条简洁、可行动的必要说明；它属于本次交付说明，不是审阅报告；
5. 用户主动追问“为什么这样写”时，系统可以解释结论依据、关键假设和修改建议，但仍不提供提示词、检索过程、内部来源分类、评分、重试记录、被淘汰候选或模型隐藏推理；
6. ADR-010 的版本与生成事件继续用于机器自评和故障处理，不进入内容交付界面，不赋予企业管理员或运营人员查看后台运行轨迹的入口，也不得保存模型隐藏思维链。

1. **主产品任务完成**：成品确实完成 P1—P5 中声明的主价值任务；
2. **用户前提忠实**：没有无授权改写核心身份、关系或事件结果；
3. **既有资料与任务输入忠实**：未无依据新增或篡改品牌/商品资料，也未擅自改变用户本次手工输入的经营信息；
4. **账号授权与角色语义正确**：账号操作/组织资产授权和明确 `PolicyRule` 必须满足；一般角色表达适配不足进入重写，而不是被误作真人现实资格审查；
5. **品牌实质相关**：品牌/账号路径实际参与内容，不是结尾硬接；
6. **应用边界正确**：没有把内容成品冒充经营陈列执行方案、现场事实或未建设的交易能力；
7. **真实可制作或明确降级**：当前条件可制作，或者已清楚标识概念、假设、摆拍和未知项。

条件门只在相应内容被调用时生效：价格、库存、活动和营业状态只使用用户本次明确提供的输入，系统不负责核验现实真值；品牌/商品资料、平台规则或趋势仍调用其已有版本；“上一期”等公开承接依据已有发布记录或用户本次明确给定的前情，不能由系统自行编造。提出商业 CTA 时只使用 CTA 实际需要的信息，不因“到店/咨询”自动要求价格与库存。

通过上述门后，后台再评估有效增益、内容张力、品牌/账号专属性、专业取舍、非宣讲感、平台原生性、连续性、反重复和真实可制作性。无 Logo 识别、稳定视觉母语、极强城市辨识度和跨期符号一致性属于成熟度指标，不作为新品牌首批内容的绝对阻断门。评分阈值必须由黄金任务和真实反馈校准，不在讨论阶段伪造永久权重，也不显示给普通用户。

### 3.14 `[SH-12]` 多账号发布策略

`distribution_strategy` 只在一个矩阵任务需要派生多个账号成品时适用：

### 4.1 产品定义

P1 不要求用户先提出明确的穿衣或购买问题。内容生产意图成立后，系统可以从一句话、商品、图片、场景、评论或账号前情等弱种子中，结合目标人群、账号、领域知识与品牌/商品知识，主动发现代表性的穿衣选择问题并形成可执行、诚实的选择判断。商品可以成为输入起点、判断证据或候选解法，但目标人群的选择机会才是内容价值锚点。

### 4.2 服务对象与稳定问题

- 品牌总部：把目标人群的长期穿衣难题转化为可搜索、可复用的品牌专业内容；
- 区域/门店：回应当地天气、通勤、场合和现场试穿中的真实选择问题；
- 目标受众：知道怎样选、为什么这样选、什么情况下不该选以及有哪些替代。

P1 回答的是“目标人群面对这类任务时应该怎样判断”，不是“请介绍这件商品”。系统可以创造本篇要回答的内容问题，不能把推断出来的身体、职业、预算、经历或场景冒充用户事实。

### 4.3 可接受种子

- “胯宽，明天见客户，这条裤子能不能穿？”
- “广州突然降温，通勤又不想穿得太厚。”
- “两件外套只能留一件，怎样选？”
- “我总觉得宽松就能遮肉，为什么反而更拖沓？”
- 门店常见选择问题、评论和搜索问句；
- “这条高腰阔腿裤今天到了，帮我做内容”等商品型弱种子；
- 一张图片、一个场景、一句观察、账号前情，或用户接受的系统选题建议。

### 4.4 必需与可选上下文

必须至少具备：

- 已经成立的内容生产意图；
- 目标人群/账号服务对象；
- 一个可观察的任务种子；它可以是明确问题，也可以是足以发现代表性选择机会的弱种子；
- 发布账号所绑定内容角色的表达档案（`role_expression_profile_ref`）；账号操作和组织资产使用另由硬授权引用控制；
- 品牌关联路径。

若推荐具体商品，还必须具备当前商品事实、可用版本和适用边界。预算、当地天气、库存、已有衣橱和视觉素材可选；缺失时不得假装已知。

P1-B 已确认：用户无需填写字段或选择 P1—P5；普通交流不自动建任务，模糊种子获得主动内容机会，明确执行信号成立且信息足够时直接生成一个最佳成品，真正分叉时最多先给三个简短方向，只有缺少不可替代信息时问一个问题。路由按主要成品价值判断；系统全局、品牌租户和账号/人设三层配置由同一次运行编排合成，品牌关联必须实质改变判断或表达。完整口径见[讨论纪要 031 V0.4](../讨论纪要/2026-07-20-031-P1B自由输入与渐进收敛候选.md)。

### 4.5 允许增益

- 把模糊顾虑拆为场合、轮廓、活动量、温度、搭配和心理感受等判断维度；
- 从弱种子中提出目标人群中具有代表性的选择问题，但不得把它写成用户本人已经具备的事实；
- 提供主选择、反选条件，并在确有合理替代空间时提供替代方向；
- 用条件化专业知识解释比例、版型、色彩和搭配；
- 把建议转成问答、清单、短视频、图文或可选 `ContentStylingSpec`；
- 自然调用品牌真实商品，但不要求每条都绑定 SKU。

### 4.6 禁止新增与典型风险

- 不得根据一句话诊断疾病、体型缺陷或制造身体羞辱；
- 不得承诺必然显瘦、显高、减龄、遮肉或适合所有人；
- 不得猜测用户精确身高体重、预算、职业和衣橱；
- 不得虚构商品库存、价格和试穿结果；
- 不得为了推荐品牌商品而忽略更合适的反选和替代方案。

### 4.7 品牌关联与商品角色

品牌关联可来自目标人群理解、品牌版型/审美方法、真实商品组合、门店专业服务或既有问答系列。`product_role` 常用 `evidence/solution`，也可为 `none/scene`；`desired_audience_shift` 与 `primary_commercial_action_intent` 分别表达认知变化和期望行动，不能混入商品角色。成品只有在用户本次明确提供价格、活动或库存时才可使用相应信息；系统不主动核验，单纯邀请咨询或到店也不要求补充这三类信息。

### 4.8 默认交付

P1 只有一个已确认的成品合同：

> **P1 必须围绕一个对目标受众有意义的穿衣选择，形成有依据、有边界、可采取下一步行动的条件性选择结果。表达可以是直接回答、选择方法、生活情境或真实演示，但任何表达都不能替代选择价值。系统可以从弱种子创造本篇问题，不能创造用户或商品事实；最终向创作者交付一份连贯、可直接使用的完整品牌内容成品。**

受众看完应能知道优先怎样选或怎样判断、哪些关键条件会使判断成立或改变，以及下一步怎样选择、比较、试穿、调整或放弃。这些是验收结果，不是用户可见固定三段、字段或篇章标题；替代商品、显式“适合/不适合”段落、具体 SKU、购买 CTA、生活故事和真实演示均不是每篇必填项。

若出现商品或组合，正文只使用当前可成立的依据。系统可以按任务组合知识解释、生活叙事、可执行行动和传播表达，但不形成多个用户模式、流程或交付，也不机械堆满全部能力。用户明确需要造型、演示或拍摄方案时，相应说明作为同一成品组成部分一并交付；后台来源、检查和改写过程不展示。

### 4.9 连续记忆

独立单篇默认不形成系列前情；用户明确把某个问题、判断、未完问题或后续选题纳入系列时，系统按其指令更新。仅生成、浏览或导出不自动写入，但用户可在任何时候要求继续、跳过、插入或重排某一期。单篇判断不得自动升级为稳定表达规则，也不得建立具体顾客画像。用户本次提到的身体感受、预算和衣橱默认只服务当前任务；创作者关于自己的长期设定，只有显式配置后才进入账号/角色配置。“上一期”可以依据已有发布历史，也可以依据用户本次明确指定的前情；系统不得要求用户先补发布证明。

### 4.10 验收与反例

合格：即使用户没有先提出明确问题，系统也能在不创造用户事实的前提下从弱种子发现代表性选择机会；最终受众能够复述选择方向、关键边界和下一步行动，品牌专业或账号立场自然参与，创作者得到可直接使用的完整品牌内容。通过这些基本价值后，成品相对于通用 AI 文案、传统品牌宣传或课堂式说教还应至少具有一个可明确指出的竞争赢点，并且没有致命平庸点。

不合格：标题是“小个子显高秘诀”，正文只有“自信最重要”和商品卖点；不管任何输入都推荐同一件“百搭显瘦”商品；或者内容事实正确、媒体包装完整，却没有观看理由、可以原封不动换成任何品牌、充满授课感或以克制之名进行隐蔽卖货。

日常生成只做快速必要自检并在后台改写；P1-D 已用两个最小黄金任务证明相对普通稿可形成明确竞争赢点。后续回归只验证选择价值、事实/品牌边界和必要敏感性，不以文字或结构相似度模仿黄金稿，也不以未经校准的永久分数或市场结果承诺替代。

## 5. P2 合同：商品真相与专业取舍

> 稳定产品合同：`P2-C01`—`P2-C04`。本章继承 `SH-01`—`SH-13`，以下只规定 P2 专属差异。

### 5.1 产品定义

P2 采用“条件性商品真相与预期管理”主干：围绕品牌的真实商品、系列或设计选择，主动找到本篇最值得解释的一项商品选择或认知问题，把参数和卖点还原为目标受众可感知的价值、相伴的代价/限制/放弃，以及该取舍成立与不成立的条件。

P2 不要求用户先提出完整商品问题，也不强制完整介绍整件商品。系统可以创造本篇解释问题，不能创造商品事实或品牌设计原意；反直觉切口、问答、比较、演示、替代和使用建议均为按需表达能力，不是每篇固定栏目。

### 5.2 服务对象与稳定问题

- 总部、区域和门店的内容运营者：依据各自账号人设和表达身份，为外部受众生产完整内容；知识默认可用不等于区域或门店可以冒充总部设计、开发或选品角色；
- 目标受众：理解这件商品为什么会呈现当前特点，应当期待什么、不应期待什么，以及该取舍在什么需求、目标和使用条件下值得或不值得。

P2 是对外内容产品，不是员工商品培训、内部陈列指导或经营执行方案。面向企业员工、代理和门店的通用陈列培训可以使用领域知识；面向本品牌、本季、本店具体商品的培训、组合和执行建议属于零售经营陈列，须绑定相应商品资料、品牌规则及用户提供的必要业务条件。

### 5.3 可接受种子

- 用户当前选中的商品/系列、商品卡片、照片、视频或一句“讲讲这件”；
- “为什么这件风衣没有做得更修身？”
- “这件针织衫贵在哪里，哪些价值能讲，哪些不能讲？”
- “这个面料容易皱，为什么品牌仍然选择它？”
- “这条裤子到底不适合谁？”
- 商品资料、评论与一线观察、商品会、组织任务、用户接受的系统选题，以及因商品适配、护理或实际表现引发的讨论。用户不必先写成完整问题。

若主要成果是解释当前商品为何出现某种表现，可以进入 P2；若主要成果是判断质量责任、退换资格、期限或售后流程，则进入服务/政策回答，P2 不代为判责。

### 5.4 必需与可选上下文

完整 P2 至少保留一项能够让受众获得新增商品理解的真实商品专属锚点。锚点可以来自品牌专属知识、用户当前选中的真实商品、用户本次明确输入，或素材中清楚可观察的商品特征；不要求 SKU、在售、库存、ERP、完整商品档案或官方设计理由。这里的“真实”表示存在当前资料或输入可指向且不是模型虚构，不表示笛语核验现实世界真值。

当成品对具体商品的成分、结构、工艺、性能、穿着结果、设计/开发/选品理由或真实经历作出承重表达时，只能使用当前可用商品资料、用户本次明确输入、可观察信息，或基于已知特征且不冒充已验证性能的专业解释。资料不足时只删除或收窄相关商品主张，不压缩其他创意表达；只有缺失内容会改变用户明确要求的独立成果时，才作一次最小追问。

商品只是背景、主要价值是品牌/账号生活观点时路由 P3；主要价值是商品组合、视觉欲望和审美世界时路由 P5；主要价值是帮助具体的人作选择时路由 P1；没有商品专属锚点的通用专业问题可以自然回答，但不能冒充完整 P2。

系统内部只复用本篇实际使用的商品对象和承重资料引用，不建设逐句证据链、事实评分、责任人审批或用户审计界面。用户最终只看到成品。

### 5.5 允许增益

- 根据品牌、账号人设、目标受众和内容语境，自由创造选题角度、冲突、比喻、幽默、无厘头表达、明确演绎的场景与对白、节奏、镜头、声音和情绪；
- 把已知商品特征自然融入生活问题、人物观点、动作、视觉和情绪，而不是把商品资料写成讲义；
- 把参数和结构翻译为有边界的穿着、使用或视觉影响，解释同源收益、限制和取舍；
- 品牌没有提供设计、开发或选品理由时，仍可充分创造表现，并基于已知特征作专业解释，但不能替品牌编造动机；
- 根据任务择优使用问答、比较、试穿、评论回应、观点口播、情景演绎或其他内容形式，不冻结固定结构；
- 按需调用造型、镜头、对比和视觉证明等共享能力；调用这些能力不新增 P5 合同。

### 5.6 禁止新增与典型风险

- 不得编造成分、工艺、成本、产地、测试、销量、功能和研发故事；
- 不得编造设计/开发/选品原意，或把明确演绎的对白、顾客反馈和门店事件冒充真实发生；
- 不得把一次试穿、图片观察或条件性专业解释扩大成所有人的真实体验或当前商品已验证性能；
- 不得把审美判断写成客观功能；
- 不得使用“顶级、绝对、零风险、人人适合”等无依据承诺；
- 不得为了诚实而贬损商品，也不得把“不适合谁”做成反向营销套路；
- 不得用泛面料百科替代该商品的真实品牌选择。

### 5.7 品牌关联与商品角色

P2 的品牌关联必须落到至少一项真实商品专属锚点及品牌/账号对它的独特理解，商品不能只是背景陈列。商品在数据库中的角色、对象身份和投影方式留给 D-016，不为 P2 新增枚举。商业行动目标可以为空；P2 可以只完成商品认知，不要求咨询、到店、查看详情或购买，也不因出现购买目标进入交易系统设计。

租户内品牌知识默认可用，但表达必须符合账号身份：总部适当角色只有在存在品牌确认理由时才能使用相应第一人称设计/开发表达；区域和门店可以准确转述或作专业解释，不能冒充总部设计团队；代理、批发、散货和买手/集合店可以解释商品及已确认的选品判断，不能把“我们选择”写成“我们设计”。

### 5.8 默认交付

每个独立 P2 成品只建立一条主要商品认知主线：明确本篇希望受众对该商品新增、修正或重新建立什么预期。系统只调用支撑该主线所需的真实商品信息和专业解释，并把相伴价值、限制或取舍，自然融入符合品牌及账号人设的完整外部表达。

商品事实不得写成资料朗读，真实取舍不得写成固定“劝退段”，品牌关联不得依靠片尾口号、生硬贴标或强制品牌宣言。比较、替代、生活情境、幽默、演绎、口播、视觉呈现、声音和 CTA 均按需使用，只在增强商品认知主线时加入。品牌相关可以由选择哪项商品真相、以什么立场解释以及哪个账号怎样发声而自然成立，不要求每篇念品牌名、Logo、品牌历史或价值观。

形式改写可以重新组织全部表达，但不得丢失商品专属认知主线；具体台词、解说、画面、动作、字幕、声音、时长和媒体组成由当前任务与 D-032 决定。正文不得出现资料审计术语或后台判断。

### 5.9 连续记忆

独立单篇的商品观点不自动进入系列；用户明确建立/修改商品栏目、接受某一集方向或指定下一期时，系统按其指令维护主题地图和分集计划。仅生成、浏览或导出不自动写入，但不限制用户跳集、插集或直接指定某一期。“上一期”可以依据已有发布历史，也可以依据用户明确提供的前情；发布记录不是继续创作的门槛。商品参数始终重新读取当前真源，不能从上篇文案反向成为事实；商品换季、下架或事实变更时，旧内容前情不得覆盖当前资料。

### 5.10 验收与反例

合格：能用一句话复述本篇主要商品认知主线；商品事实、条件性取舍、品牌/账号表达和创意共同服务这条主线，而不是拼成几个段落。替换成相邻商品后核心结论会实质变化；删掉“高级、显瘦、百搭、必入”等形容词后，受众仍能复述一项真实商品价值、相伴限制，以及它在什么条件下成立或不成立。内容至少提供认知反转、误解解除、可观察呈现、新鲜语言、真实共鸣或值得讨论的问题之一。

不合格：逐行朗读商品表后加“高级感拉满”；用泛面料百科冒充该品牌商品真相；虚构品牌为了某类女性历经多年研发的故事；以“千万别买”开头，最后却绕回人人都适合的表演式劝退；情景演绎很好看但受众不知道商品有什么不同；或即使事实全部正确，最终仍像内部培训讲义、商品说明书和课堂包装。

## 6. P3 合同：品牌/账号观点与生活叙事

> 稳定产品合同：`P3-C01`—`P3-C04`。本章继承 `SH-01`—`SH-13`，以下只规定 P3 专属差异。

### 6.1 产品定义

P3 不要求用户提供完整故事，也不要求每篇出现商品、品牌宣言、人生金句、连续剧情或商业行动。当生活、工作、家庭、组织或情绪种子具有内容价值时，系统应结合当前品牌、发布账号、内容人设和目标受众，从其中找到最值得表达的一项具体观察、感受、矛盾或生活反应，并交付一份连贯、可直接使用、具有观看回报的完整对外成品。

受众看完后，应能识别或进一步确认：这个账号面对这类生活会注意什么、怎样理解或判断、怎样对待自己与他人。每篇至少新增或强化一项可归属于该账号的人格信号，并给目标受众带来具体的理解、共鸣、幽默、陪伴、反思或继续认识该账号的理由。单篇不承担塑造完整人格、证明长期一致性、建立系列或写入长期记忆的责任。

这里的“人格信号”不是新字段或分数，只表示本篇对一个可识别账号人格作出了一次有效贡献。P3 的长期竞争原则是“熟悉，但猜不到”：稳定的是观察位置、价值取向和待人方式，变化的是题材、细节、情绪、幽默和表达形式。

### 6.1.1 真诚与反 AI 口水化

真诚是 P3 的创作哲学，不是系统能够保证的受众感受、流量结果或评分字段。系统不得用精致假话、虚构履历、模板共情和宏大升华冒充真诚。

允许自然口语、长短句变化、停顿、自我修正、尚未解决的矛盾和人物反应中的小毛边；但口语化不等于同义反复、口头禅堆积、故意错字、表演式结巴、结构松散或低信息量。禁止用抽象正确、假共情、万能金句、空泛反问、强行升华和模板化“不是……而是……”填充内容。首版不为此新增真诚分数、AI 文风检测服务或机械禁词表，只复用既有后台快速自检完成删减或改写。

### 6.2 服务对象与稳定问题

- 品牌总部：让品牌世界观通过组织中的人和生活持续成立，而不是只靠宣言；
- 区域/门店：以经营者、店主或店员的真实视角建立本地人格；
- 目标受众：获得情绪、理解、陪伴、幽默或人生观察，并逐渐识别“这个账号怎样看生活”。

P3 的首要任务是建立人格和长期关系，不以每篇出现商品或立即成交为条件。

### 6.3 可接受种子

- “今天喝了一直喝的蓝山咖啡，居然是甜的。”
- 一次普通工作失误、被理解、被拒绝或重新选择；
- 夫妻、亲子、父母、朋友、年龄变化和城市生活片段；
- 设计、订货、培训、开店和门店经营中的小事；
- 账号上一期留下的问题或观众追问。

父母、爱人、子女等是关系角色或人物称谓；失败、欢喜、幽默和无厘头等是主题、状态或表达手法，不建立共享本体实体。

### 6.4 必需与可选上下文

必须具备：品牌稳定核心、目标 `publishing_account_id`、所绑定的 `content_role_id` 与 `role_expression_profile_ref`、目标人群、`task_seed` 和品牌关联路径。账号操作与组织资产使用另由硬授权引用控制；实际操作人及其工作区由 `actual_operator_id`、`workspace_id/represented_organization_id` 记录，不能与对外发布账号或内容角色合并。若续写系列，系统应取得当前系列卡片，并优先执行用户本次对跳集、插集、改序、指定前情或重设剧情的明确指令。正文使用“上一期”等表达时，可以依据系统已有发布历史，也可以依据用户明确提供的当前创作前提；没有二者时系统不得自行添加。

商品、门店、组织事实、人物前情和趋势可选；若使用具体事实，必须遵守其当前来源和范围。

### 6.5 允许增益

- 从弱生活、工作、组织或情绪种子中主动选择本篇最值得表达的一项观察、矛盾、感受、反应或未完问题，形成一条占主导的人格反应主线；一条主线表示一个主要观看理由，不限制情绪、场景和转折数量；
- 提炼生活种子的矛盾、惊喜、情绪和可分享观点；
- 选择清晰的 `tone_style` 和 `desired_emotional_effect`；各自默认一个主值，确有必要时最多一个辅助值，不能把表达方式和受众感受重新合并成“效果”大枚举；
- 使用比喻、幽默、无厘头、克制共鸣、观察或轻励志等结构；
- 按当前任务的 `creative_license` 安排节奏、视角、对白、动作和表现；无论何种许可都不能擅自改变核心身份、核心关系和事件结果；
- 由 `product_role = none/scene/evidence/solution` 描述商品如何参与，由 `desired_audience_shift` 与商业行动意图独立描述认知变化和后续动作；
- 进入夫妻、育儿、公婆、同事、金钱、疲惫、失败、欢喜、狼狈和普通生活矛盾等人设生活领域；未提供真人经历时使用观察、一般情境或明确演绎，不冒充账号真实履历；
- 仅在自然有价值或用户明确需要时，形成可连续发展的栏目、开放问题和下一集候选；
- 在用户明确发起矩阵任务时，基于共同事实和多个目标账号的合法位置形成独立子成品；共同母题不强迫各子成品使用同一个主要产品合同。

### 6.6 禁止新增与典型风险

- 不得无依据改变或新增会重写人物核心身份、核心关系、事件结果的婚姻、亲子、职业、创业、疾病、冲突和重大人生经历；
- 在 `enriched/dramatized` 下可以增加不改变事实走向的对白、动作、情绪和场景，但不得冒充逐字记录或实际门店案例，也不得自动写入长期履历；
- 不得用婆媳冲突、性别对立或苦难制造与品牌无关的泛情绪流量；
- 不得为了“完整人设”替账号补造丈夫、孩子、公婆、顾客、同事、婚姻状态或人生失败，也不得强迫所有账号覆盖所有生活题材；
- 不得每篇都从小事上升为宏大励志和 AI 金句；
- 不得反复改写同一个意思，堆叠抽象生活词、假共情和万能祝福，以口语化为名保留没有观察、人格或受众回报的 AI 口水；
- 不得为了品牌相关而在结尾硬接商品、活动或购买链接。
- 不得把矩阵联动做成只换账号名、城市名、方言和片尾的假差异，也不得借共同母题跨工作区读取私人素材、私人偏好或系列前情。

### 6.7 品牌关联与商品角色

P3 不以出现品牌名、Logo、商品、品牌历史、价值观宣言或商业行动作为品牌关联成立条件。成品首先必须具有独立观看价值；商品可以承担真实生活场景、人格表达或观察载体，但不能仅凭出现就证明品牌关联。品牌对目标人群的长期理解、稳定价值取舍、专业或审美观察、真实组织经验，必须对本篇的核心观察、判断、待人方式或表达取舍产生承重影响。

发布账号必须从企业 IP 矩阵中的合法位置发声。保持相近账号职能和目标人群而换成价值取舍明显不同的品牌时，核心观察、判断或关系姿态应实质变化；保持同一品牌而改用不同组织层级或内容角色时，可信细节与发声资格必须重新成立。账号专属、本地化或混合表达应随账号位置实质变化；显式 `unified` 统一发布可以受控复用品牌核心成稿，但不得把复用稿说成区域或门店的独立人格内容。只替换品牌名、城市、商品、语气词、片尾口号或 CTA，不构成专属性。

找不到自然关联时，系统不得硬插商品、复述品牌口号或强行升华人生道理。它可以继续普通交流、保留为用户主动保存的私人素材，或提出一个真正与品牌有关的创作角度；不能把无品牌落点的生活段子冒充正式 P3 成品。单篇始终验收品牌上下文真实参与和账号位置合法；去 Logo 可识别属于多个成品累积后的 IP 成熟目标，不作为新账号每篇内容的硬门。

### 6.8 默认交付

默认围绕一条占主导的人格反应主线，交付一个与当前媒体任务匹配、可以直接使用的完整成品；视频应包含完整台词、对白、口播、解说或字幕文本，并按形式补足实际需要的画面、动作、节奏、背景音乐或音效建议，图文应包含可直接使用的正文和必要视觉组织。互动、连续入口和商业动作只在本篇确有价值或用户明确需要时加入，不是 P3 固定组成。`exploratory` 才交付 2—3 条结构真正不同的成品路线；`focused` 可以在内部作必要比较，但对用户仍只交付一个主版本。来源分类、创作新增清单、检查过程和生成轨迹均不展示。

同一主题可以按需改写为单人口播、多场景叙事、对话、旁白、动作演示、图文或其他形式，作为同一任务的新完整版本；关键前提、品牌/账号身份、人格主线、承重商品事实和商业表达诚实性必须保持。矩阵任务可以生成多个账号子成品，但每个子成品仍独立绑定发布账号、合法人设和一个主要产品合同；没有真实视角差异时减少版本或采用 `unified`，不机械扩写。

### 6.9 连续记忆

用户可以直接制定或修改系列前情，也可以明确把某个成品、观点、称谓、开放问题或后续承诺纳入系列；系统生成本身不自动写入。用户可以跳集、插集、改序、跨越时间或从指定集起重设剧情，旧发布记录继续留痕但不反向限制新编排。系列内称谓、关系别名、情节前情和进度只进入当前系列，不自动进入账号配置，也不建立家庭成员人物主数据、CRM 或共享关系图谱。虚构前情可以继续存在于明确标记的虚构系列中；新核心人生设定和账号长期立场仍须由用户显式配置。“上一期”等公开承接可以来自系统已知发布历史，也可以来自用户本次明确指定的创作前提，后者不自动被记成平台已验证发布。

矩阵任务中的共同母题不自动建立跨账号共享系列；每个账号分别决定是否将自己的成品纳入本账号系列。总部、区域和门店不得借联动读取或继承其他账号的家庭称谓、人物前情、未完剧情和连续记忆。

### 6.10 验收与反例

合格：删除品牌名、Logo、品牌口号、CTA 和纯促销插入后，内容仍有一个具体且值得观看的生活观察，使受众进一步认识这个账号会注意什么、怎样理解或判断、怎样对待人，并获得理解、共鸣、幽默、陪伴、反思或继续认识它的理由；商品若真实承担生活场景、人格表达或观察载体可以保留。保持相近账号职能和目标人群而更换价值取舍明显不同的品牌时，核心观察、观点或关系姿态应实质变化；账号位置始终必须合法，只有账号专属、本地化或混合表达要求随账号实质变化，显式 `unified` 可以受控复用品牌核心成稿。表达可以自然口语和不完美，但实质段落必须推进具体观察、人格贡献、受众回报或必要节奏，不能靠同义反复、假共情和强行升华注水。

不合格：“不是咖啡变甜，是你的心变甜了；成年人要学会发现小确幸。”文句顺滑但任何账号都可发布，没有具体观察和账号人格；“在快节奏的时代停下脚步，珍惜每一份小美好，愿努力生活的你找到幸福”即使改得更口语，也仍是重复、空泛且可随意替换的 AI 口水；为制造烟火气突然给总部专业账号编造婆媳矛盾；或把同一事件只替换账号名称和城市便宣称完成矩阵多视角。

## 7. P4 合同：一线问题回应与本地生活

> 稳定产品合同：`P4-C01`—`P4-C04`。本章继承 `SH-01`—`SH-13`，以下只规定 P4 专属差异。

### 7.1 产品定义

P4 是品牌本土化能力中的关系型内容产品。系统不等待本地受众先评论，而是在保持品牌战略内核和账号身份不变的前提下，结合地域目标受众的生活结构与合法账号的当地位置，主动发现该品牌在当地值得关注、表达或共同经历的关系命题；存在真实评论、一线观察或服务事件时，再形成针对性回应。

品牌本土化继续作为 P1—P5 的共享编译能力：地域条件主要改变选择时路由 P1，主要解释商品时路由 P2，主要建立账号人格时路由 P3，主要创造视觉欲望时路由 P5；只有主要成果是让品牌或账号进入当地人的日常、建立或推进当地关系时，才以 P4 为主。

### 7.2 服务对象与稳定问题

- 区域代理：把全国品牌核心转成当地气候、城市结构和经营语境；
- 加盟门店：建立熟悉感、服务信任、互动和真实到店理由；
- 本地受众：得到与自己所在城市、生活节奏和门店体验有关的回应。

P4 首要经营的是关系。D-006 已关闭：第一阶段本地生活只建设在地内容、用户明确提供的门店事实和可选自然到店指引，不建设团购、预约、下单、支付、核销、评价运营、广告归因或即时零售；该能力边界不把 P4 的关系价值缩窄成门店事实清单。

### 7.3 可接受种子

- 品牌目标受众、区域生活与合法账号位置共同形成的主动在地内容机会；
- 评论或搜索问句；
- “今天进店的人都在问同一个问题”；
- 当地突然降温、梅雨、通勤、商圈活动和节日生活；
- 店主/店员的一句观察、一次服务感受或门店小事；
- 总部母题需要在本地重新理解的任务。

### 7.4 必需与可选上下文

必须具备：合法发布账号及其组织/地域位置、内容角色、品牌核心，以及足以使当前在地关系命题成立的地域上下文。依据 P4-B，品牌无需另行填写或确认独立地域基线；系统使用已有经营区域、D-014 品牌战略基线、账号身份与历史、用户已保存的区域设置，以及有来源、有地域范围和检索时间的按需公开搜索结果。第一阶段不承诺不受限实时本地搜索、持续平台抓取、POI 数据流、ERP/POS 或每次任务必搜。用户自然纠正在当前任务即时生效，只有有权限者主动保存后才成为区域长期设置。本文不把这些语义冻结成字段、表单、数据库对象或工作流节点。

公开可检索且与成品直接相关的天气、公共活动和地域背景可以按需搜索；动态事实实际写入成品时，只在后台保留所需的最小来源、地域范围和检索时间，不向普通用户展示运行轨迹。低质量、过期、冲突或搜索失败的结果直接省略，不阻断仍能成立的成品。门店库存、价格、促销、营业状态、服务能力、实际顾客和真实一线经历仍只能来自用户本次明确输入，搜索不能代替经营主体作出这些断言，也不能自动升级为长期知识。概念/摆拍画面不得反向证明现实天气或门店状态；第一版不接入 ERP/POS，也不得声称数据经过系统实时核验。

### 7.5 允许增益

- 在没有评论或完整本地选题时，主动发现品牌在当地值得表达的关系命题；
- 把一句评论或观察拆成当地人真正关心的问题；
- 结合气候、通勤、商圈和门店服务形成区域转译；
- 生成回应、问答、续集、到店理由和温和互动；
- 使用真实商品、店内场景和 P5 视觉能力；
- 从总部母题生成结构不同的本地版本，而非机械改写。
- 在确有必要时使用搜索发现公开在地线索，再由品牌战略和账号身份决定内容判断；不得把搜索摘要直接交付为成品。

### 7.6 禁止新增与典型风险

- 不得把总部文案只增加城市名、方言和天气词；
- 不得依据一二线/三四线、南方/北方或行政城市名称直接生成当地受众人格、审美与价值观；城市层级只能帮助发现问题，不能自动成为人群事实；
- 方言、地标、民俗、地方音乐和地域梗不能单独证明在地化；只有符合账号表达能力并实质改变生活处境、关系或创作表达时才按需使用；
- 不得把系统虚构、合成或戏剧化的当地习惯、热门商圈事件、顾客故事和店内当天情况冒充实际发生；明确标记为假设、情景演绎或虚构的内容可以生成，但不得写回实际顾客记录、门店事实或本地知识；
- 不得凭空声称现货、最低价、活动、营业时间和销售表现；
- 不得将评论者建立为顾客实体、画像或 CRM 记录；
- 不得把本地生活扩大为首期团购、预约、下单、支付、核销、评价运营、广告归因或即时零售能力；
- 不得增加总部或区域逐条内容审批门。
- 不得把搜索排名、热词或重复出现的网络说法直接升级为品牌关系判断、区域受众事实或长期知识。

明确虚构的顾客角色、模拟问答或情景剧可以作为表达结构，但不能单独证明 P4 的真实近场关系或本地问题成立；任务仍须具有门店实际服务关系、真实评论/观察，或带适用范围的稳定本地生活背景。虚构天气、商圈事件和地方习惯即使标为虚构，也不能冒充本地事实或成为现实到店理由；若内容首要任务只是用虚构人物表达账号观点，应改路由 P3。

### 7.7 品牌关联与商品角色

品牌关联可落到本地化品牌承诺、门店人物/服务、真实商品、当地穿着问题、商圈关系和连续栏目。`product_role` 可为 `none/scene/evidence/solution`；`desired_audience_shift` 与 `primary_commercial_action_intent` 分别表达关系变化和咨询、到店、查看详情或购买等行动。CTA 只有在用户本次明确提供价格、活动、库存或确定服务能力时才可使用相应信息，不能因存在转化目标就强迫内容出现或补全这些信息。

### 7.8 默认交付

P4-C 采用“在地关系命题—品牌账号回应—受众关系回报”的单一主干；主动发现与现实信号只是两种输入入口，不建立两套产品或工作流。后台只过三关和一道成品闸：在地上下文必须实质改变处境、判断、行动或关系姿态；当前品牌账号必须从合法位置作出具体回应；受众必须获得具体理解、帮助、服务边界、熟悉感、信任或继续关系的理由；最后交付一份符合共享成品定义、可直接使用的完整作品。若只是换城市名/方言/地标、账号可随意互换、只有“我们懂你”、或仍是选题/提纲/搜索摘要/客服话术，分别判定相应关卡失败。媒体组成沿用共享成品定义，具体组合归 D-032，不在 P4 重复枚举。商品、故事、自然互动、下一期、到店/咨询动作和 CTA 均按需，不是成立条件；后台核对和运行信息不展示。

### 7.9 连续记忆

用户明确建立/修改本地栏目、指定某一期、接受某个成品方向或要求把已回应问题、当地母题和未完追问纳入系列时，系统按其指令更新；单纯生成、浏览或导出不自动写入。用户可以跳集、插集和重排，不必先补齐内容或发布证明。具体系列的称谓、进度和开放问题只保存在当前系列，不自动与同店其他账号、区域或总部分支共享；总部不能借租户管理权读取加盟店系列前情。系统已有发布历史和用户明确指定的当前前情都可以支持公开承接，但不得借任一方式后台串用其他账号或工作区记忆。

### 7.10 验收与反例

P4-A 已确认的最低基线是：没有评论、搜索词或门店观察时，系统仍能依据品牌与地域上下文主动发现合适的当地关系命题；地域生活应实质改变问题、处境、判断、关系姿态或表达，不能只换地名、方言、地标、门店地址或 CTA；不同地域表达必须保持品牌同源，且系统创造的内容问题和假设情境不得冒充当地人群事实或真实事件。评论和现实信号是增强路径，不是 P4 前置条件。P4-B 进一步确认零表单搜索启动与使用中渐进校准：搜索只补充公开地域背景和当次信号，不能代替品牌关系判断或创造目标受众、顾客和门店经营事实；搜索失败安全降级。P4-C 进一步确认三关加一道共享成品闸，不重复建设媒体清单或用户可见分析报告。

主动在地型与现实信号型任务共用 P4-C 单一成品主干；P4-D 已证明二者需要不同的最小验证入口，但不建立两套生产流程，也不机械统一为“每篇都换城市”。

P4-D 已用[两个冻结任务](../讨论纪要/2026-07-21-053-P4D两个最小黄金任务包待确认稿.md)及[完整黄金证据](../讨论纪要/2026-07-21-054-P4D两份候选黄金成品待确认稿.md)关闭：包一只作为零近场信号下的 P4/P3 路由边界和安全下限特例，其 P4 性来自地域空间不足以推出同一身份/路线的具体关系命题，不来自账号“克制、不装懂”的人格；抽掉地域承重只剩克制态度时必须转 P3。一旦存在真实评论、一线观察、服务行动或可靠当次信号，系统不得继续套用该下限话术，必须形成更具体的关系回报。包二作为现实信号正面基线，把本店事实转成未到店者也能带走的关系许可。地域快照替换与接待动作移除检查均通过，两份平庸近邻分别隔离“是否虚构人群路线”和“是否具有公共观看价值”。自然口语修订不新增文风评分、禁词表或检测服务。

不合格：总部同一脚本分别改成“杭州女生、成都女生、武汉女生”，其余内容完全相同。

## 8. P5 合同：造型灵感与视觉叙事

> 稳定产品合同：`P5-C01`—`P5-C04`。本章继承 `SH-01`—`SH-13`，以下只规定 P5 专属差异。

### 8.1 产品定义

P5-A 正式采用“真实商品锚定的单一可见造型命题”。内容生产意图成立后，系统不要求用户先写出完整搭配方案或抽象视觉命题，而应以至少一件当前真实品牌商品为不可替代锚点，结合 D-014 品牌内容战略、合法发布账号、目标受众、任务种子和用户当前真实制作条件，主动形成一条清楚的造型命题，并交付一份可直接使用或制作的完整对外品牌内容成品。

该命题应从当前媒介的第一个主要视觉入口开始成立：封面、缩略图、首图或开场画面应使受众感知一项与本篇同源的具体造型关系、视觉张力或观看承诺；完整命题可以随后通过动作或视觉序列展开，不要求入口提前讲完整个结论，也不得用与成品无关的流行钩子代替产品价值。具体入口、时长和呈现规格由 D-032 决定。

P5 的成功锚点，是让受众从画面中真正看见一种此前没有被充分显现的穿着可能或生活状态，为穿着欲望和品牌视觉记忆提供具体理由；不是特定播放量、制作预算、场景数量或表面精致度。发布后的播放、保存、评论和转化可以作为观察信号，但不能替代产品合同，也不能自动改写品牌视觉基线。保存、模仿、讨论、转发等竞争性回报按任务择优，不要求每篇同时实现；稳定视觉母语和去 Logo 辨认度由多篇内容累积。

P5 必须按本次真实制作条件兑现。当加盟门店只有一名创作者、一部手机、普通空间和当前真实商品时，系统仍须交付本身能够执行的完整版本，优先依靠商品自身、商品之间及商品与身体、动作、自然光的视觉关系成立，而不是依赖多人、昂贵场景和复杂后期。更高制作规格可以按需采用，但不是 P5 成立的前提；门店下限也不是总部内容的制作上限。不为两者建设两套 P5 产品或流程。

P5 的任务不是回答“我应该选哪件”或“这件商品为什么这样设计”，而是让受众看见“这个品牌怎样通过真实商品显现一种想成为、想进入或想尝试的穿着状态”。视觉叙事不等于故事情节；单人实穿、一个动作、穿着过程、静物与上身关系、图片组、纯构图与声音或有情节短片均可成立，只要视觉关系真正建立、展开或发生变化。

普通用户可以只选择或展示真实商品，也可以只给一个弱种子；系统先使用现有上下文起草一个当前最优的完整成品，必要时用少量具体例子或正反对照帮助其判断。用户只需用“像不像、拍不拍得了、哪里不对、怎样改”等自然语言校准，不负责填写视觉命题、审美母语、材质节奏或空间语法等抽象表单。本项不新增多步搭配向导、强制风格问卷、配置器或日常多候选流程；具体交互实现后置。

### 8.2 服务对象与稳定问题

- 品牌总部：形成可持续的品牌视觉识别和造型母语；
- 区域/门店：在品牌审美边界内生成适合本地场景、真实人物和可用素材的视觉表达；
- 目标受众：获得造型启发、穿着想象、审美愉悦和可照着尝试的模仿线索。

### 8.3 可接受种子

以下视觉种子都必须与至少一个真实品牌商品锚点共同出现，不能单独创建 P5：

- 至少一个真实品牌商品、真实商品组合或本季代表系列；
- “把雨天通勤做成克制但不灰暗的视觉故事”；
- 一种颜色、轮廓、材质关系或情绪；
- 一次内容拍摄、图文组、Fitcheck、静物或以橱窗/模特为拍摄背景的内容摆拍需求；
- 用户与其他内容任务同时提出、但可独立采用的真实商品造型与视觉叙事成果。

以下 P5-B“真实商品最低成立条件”**仅适用于用户当前顶层选择为“内容生产（对外）”、且本次主要内容产品路由为 P5 的任务**。在这个作用域内，用户从品牌专属知识中选择商品，或者在当前任务中明确把实物、图片或视频中的服装作为本品牌真实商品提交，已经足以启动 P5。第一版不要求该商品同时具有 SKU、ERP 记录、库存、价格、成分表、工艺说明或完整商品档案。系统只使用当前已知或可观察、且与本篇造型命题有关的商品特征；缺少的信息不猜补、不宣称。资料不足只收窄商品断言，不压缩仍能成立的视觉创作。

用户明确提交的商品身份是当前任务有效前提，不表示系统已经外部核验，也不自动把图片、口述、系统解释或生成成品写入品牌长期商品知识。若只有来源不明的衣服图片，现有品牌知识也无法建立本品牌商品关系，系统可以继续普通交流或只问一次最小问题；身份未成立前，不把它冒充为完整品牌 P5 的真实商品锚点。

本项不建立“最小商品字段表”。运行时只要求：商品身份具有品牌知识或当前用户前提依据；至少一项颜色、轮廓、比例、图案、结构、穿着状态或商品关系能够真正支撑本篇造型命题；未知的非视觉事实和不可可靠观察特征保持未知。SKU 必填、商品完整度评分、识别置信度阈值、逐属性审批和新的商品录入流程均未被确认。

### 8.4 必需与可选上下文

必须具备：品牌审美/视觉边界、目标人群、目标发布账号、按第 8.3 节最低条件成立的至少一个真实品牌商品、足以承重当前造型命题的已知/可观察视觉特征，以及已经知道的素材/制作条件。`ProductComposition`、`media_actuality` 及其物理身份仍是 P5-C/D-016 候选，不是启动 P5 的用户必填字段。制作条件尚不完整时是否及怎样交付概念拍摄方案继续由 P5-C 收敛，不能冒充当前可直接实拍。

场景、人物、情绪、平台趋势、库存和门店空间可选。库存和真实空间只有在内容声称“当前可售”或“当前门店真实现场”时才成为硬事实。顶层选择为“陈列搭配（对内）”时，即使使用同一件商品、同一套造型知识，也不进入 P5；若任务要生成门店布置、培训、执行、巡检或整改方案，同样必须使用 `display-merchandising` 的独立合同。该应用所需商品、空间、载体、数量和其他业务条件是否充分，由后续经营陈列合同判断，不能套用 P5 的最低启动条件。非真实现场演绎按本稿候选词表标为 `concept_plan` 或 `staged_capture`。

第一版不接 ERP，库存、价格和当前可售状态不是启动 P5 的前提；系统不得因缺少这些信息停止仍能成立的视觉创作，也不得反向声称当前有货或可售。但连一个真实品牌商品身份或足以承重造型命题的视觉特征都无法建立时，不得创建以 P5 为主要产品合同的成品，也不得生成用户另行要求的独立 P5 成果：可以继续普通交流、提出一次最小选品/确认问题，或按主要价值改路由。P3 使用普通构图、镜头和画面节奏等共享能力时，不等于创建 P5；相应输入不足时可以省略或降级该部分，不得阻断 P3。

### 8.5 允许增益

P5-B 第 2 小项正式采用“**商品本体忠实、表现层自由**”：P5 可以自由改变商品的组合、穿着动作和视觉呈现，但不得无依据改变或补造商品身份，以及受众据以判断实物的特征。资料不足时，系统应调整表现方式或收窄商品主张，不得以降低创意为默认答案，也不得用创意掩盖商品失真。

这意味着系统可以改变受众怎样注意、理解和想象这件商品，不能改变商品本身是什么。商品与其他服装、鞋包、配饰的关系，现实可行的穿着动作，以及人物、空间、背景、道具、光线、构图、镜头、节奏、台词、声音、情绪和明确演绎均可充分创作；这些能力仍受当前真实制作条件约束，但不因商品资料不完整而退化成商品档案展示。

- 创意商品组合、主次角色、层次、替代方案和造型节奏；
- 色彩、轮廓、材质、动作、背景、道具、光线和镜头叙事；
- 明确标识的理想化场景、摆拍和概念表达；
- 视觉主题对应的标题、字幕、口播和图文结构；
- 同一组合针对不同账号/平台生成不同视觉方案。

### 8.6 禁止新增与典型风险

“受众据以判断实物的特征”包括但不限于会改变商品识别或购买预期的颜色、图案、长短、廓形、领袖结构、开合方式、口袋、标识、已知面料/功能，以及输入中没有展示或确认的背面、内里和结构细节。系统不能为了完成镜头而猜造未知部位；应优先换镜头、换动作或收窄表达，只有该未知项对当前造型命题不可替代时才作一次最小追问。灯光、滤镜和概念化演绎可以使用，但不能让受众把明显失真的颜色、比例或结构当成真实商品外观。本项不建立属性级忠实度矩阵、逐帧审批或像素相似度检测。

- 不得修改商品颜色、面料、结构和功能后仍声称为真实商品；
- 不得把概念效果图写成门店已执行、当前现货或真实顾客穿着；
- 不得输出为内部经营陈列的可执行方案、巡检证据或整改结论；
- 不得只套流行滤镜、热门色和万能“高级感”，失去品牌视觉识别；
- 不得提出超出人物、场地、商品、素材和拍摄条件且未标明假设的方案；
- 当任务目标明确要求本地化或账号差异化时，不得让总部、区域和门店只复制完全相同的模特动作、构图和脚本；若品牌明确采用统一发布策略，复用同一成品可以成立，但不得伪称已经本地化。

### 8.7 品牌关联与商品角色

P5 必须积累品牌视觉资产，并至少绑定一个真实品牌商品；`product_role` 不能为 `none`，通常为 `scene/evidence/solution`。`desired_audience_shift` 与 `primary_commercial_action_intent` 根据内容希望创造的穿着想象和后续咨询、到店、查看详情或购买动作独立设置。概念化的是场景、构图和演绎方式，不是被冒充为真实商品的服装。

### 8.8 默认交付

P5-C 正式采用“**单一可见造型命题的成品化编译**”：

> **每份 P5 只围绕一条可见造型命题，组织必要的真实商品关系，并在当前媒介和制作条件下编译成一份完整、可使用或可制作的品牌内容成品；画面必须承担主要价值，媒体组成按需，不强制故事或固定清单。**

这里的“一条命题”不等于只能出现一件商品、一张图、一个镜头或一种二元对比，也不要求情节故事；它只要求所有商品、动作与画面共同服务同一项新的穿着可能，不能拼成互不相关的造型清单。P5 成品在语义上只需完成三件事：

1. 让受众看见一条由真实商品和品牌审美共同成立的造型命题；
2. 使该命题依赖的商品与身体、其他单品、穿着动作或环境关系真实成立；单件商品已经足够时，不强迫增加搭配成员；
3. 按当前媒介与真实制作条件，择优提供足以直接使用或制作的画面、动作、次序、文字、口播、声音或制作提示；不适用项不出现。

视觉可以由单帧直接成立，也可以经动作、穿着过程或图片/镜头序列展开；纯视觉、边讲边穿、动作演示、Fitcheck、图文组、静物、人物拍摄和视觉故事均可按需选择。文字、口播和声音可以放大造型命题，但不能替代画面承担 P5 主价值。用户明确要求概念方案时可以交付完整概念成品，但不能冒充已经实拍、当前门店状态或真实商品外观；只有这种区别会影响用户理解或执行时，才在成品中用自然语言说明，不展示后台状态字段、未知项清单或风险报告。

当前制作条件直接塑造唯一主版本，不默认同时附低配/高配和多套候选；条件完全未知时优先复用账号已有制作能力，仍未知则选择不依赖多人、昂贵设备和复杂场景的可执行表达，用户随后可以自然要求升级或改写。镜头数、图片数、平台、时长、是否需要标题/口播/字幕/互动/CTA/音效及具体媒体组成继续由任务与 D-032 决定；`ContentStylingSpec` 是否存在、是表、JSON、投影还是一等对象继续由 D-016 决定，本合同不提前冻结。

### 8.9 连续记忆

用户明确建立/修改视觉系列、接受某一集视觉方向或要求把视觉命题、色彩/构图组合、重复符号和近期疲劳纳入后续计划时，系统按其指令更新；单纯生成、浏览或导出不自动写入。用户可以跳集、插集或重排视觉主题。系统高分视觉方案不能自动升级为品牌正式视觉规则；只有品牌业务确认后才能进入 `ExpressionProfile/PolicyRule`。

### 8.10 验收与反例

P5-A 已确认以下验收口径：

1. **入口处开始可见**：当前媒介的第一个主要视觉入口已经能感知一项具体造型关系、视觉张力或观看承诺；不能只有“高级、松弛、今日穿搭”等泛化文字和普通站姿。完整命题可以后续展开。
2. **新的穿着可能被看见**：审阅者能具体说出受众看见了什么新的组合、比例、状态、动作关系或生活进入方式，不能只复述抽象形容词。
3. **视觉承担 P5 主价值**：移除承担造型命题的商品关系、动作、组合、构图或视觉展开后，“新的穿着可能被看见”这一价值应随之消失。剩余台词、商品解释或生活观点仍有价值，不代表 P5 仍成立；测试不要求整篇变成废稿。
4. **当前条件下可执行**：成品必须适合本次真实资源；一人一手机的门店任务必须得到该条件下能够直接完成的版本，不能只给团队方案后附一句“也可简化”。不要求所有 P5 同时提供低配和高配两个版本。
5. **流量与精致度不替代价值**：高播放、流行转场、昂贵布景和复杂运镜不能挽救不可见、商品可替换的造型命题；低成本手机内容也不能因“不像大片”被判失败。

P5-B 已正式关闭：真实商品最低成立条件、顶层内容生产作用域、商品本体忠实/表现层自由，以及完整 P5 与共享视觉能力的路由边界均已确认。P5-C 采用“单一可见造型命题的成品化编译”。P5-D 使用一个黄金主任务和六项最小检查；用户选择 A《人没换，画面换了重音》为正式黄金候选，并确认其唯一平庸近邻及 T1—T6 判定。P5-D/P5 已关闭，五类专属验收达到 5/5。媒体实际性检查继承共享 M09，只作同一任务的一次低成本状态切换，不重复建立第二个概念/实拍黄金包。完整单变量对照只用于黄金任务、实质性版本回归和必要诊断抽样，日常生成仍做快速必要自检。

“不依赖品牌 Logo 仍可辨认”“形成稳定视觉母语”“跨期符号一致”是需要长期内容积累才能验证的成熟度指标，不作为新品牌首批内容的绝对阻断门。

不合格：首个主要视觉入口只有普通站姿和“松弛感穿搭”标题，真正命题看完整篇后仍只能靠口播解释；七个场景和复杂运镜都很精致，但换成任意相似外套仍成立；当前门店只有一人一手机，系统仍交付摄影师跟拍、灯光组和多个外景方案，只附“可以自行简化”；或把漂亮概念图直接当作门店经营陈列方案。

## 9. `[SH-09]` 主价值路由、共享能力与独立成果

本章是该规则簇的唯一规范正文；第 3.11 节只记录 D-031 第 8 项的决策来源。

1. 每轮输入先按 `SH-01/SH-02` 判断是否已经形成内容任务。普通交流可以正确结束于“不路由”；顶层选择经营陈列时退出 P1—P5。
2. 内容任务不能按“商品、门店、生活、搭配、镜头”等孤立关键词路由。系统先判断成品希望给受众带来的主要变化，再做主价值损失测试：

| 稳定产品合同 | 主要受众变化 | 拿掉后首先失去的成立理由 |
|---|---|---|
| `P1-C01`—`P1-C04` | 帮助选择 | 受众无法知道优先怎样选、何时改变判断和下一步怎样验证 |
| `P2-C01`—`P2-C04` | 解释商品 | 受众失去商品专属新增理解、相伴限制和成立条件 |
| `P3-C01`—`P3-C04` | 建立人格 | 受众没有进一步认识这个账号怎样观察、判断和待人 |
| `P4-C01`—`P4-C04` | 经营关系 | 近场共同处境/真实信号、合法账号回应或可迁移关系回报不再成立 |
| `P5-C01`—`P5-C04` | 创造欲望 | 受众不能从画面看见由真实商品和品牌审美共同成立的新穿着可能 |

3. 一个独立成品只执行一个主要产品合同。商品解释、构图、镜头、造型、动作、声音、本地语境、平台适配和账号观点可以按需作为共享能力参与，但不因被调用就产生第二产品或第二成品。可选内部参与标签若保留，只表示知识曾参与，不成为用户选项、第二合同或独立成果。
4. 某项共享能力条件不足时，优先省略、替代或降级，不阻断仍能成立的主要成品；只有用户明确要求且该独立成果缺少不可替代输入时，才作一次最小追问。
5. 用户明确要求另一项可以脱离原成品单独采用的完整成果时，才新增独立成品并分别路由、分别通过相应产品合同。多个成果可以属于同一任务或系列，但不能嵌套成双主合同；其物理关系由 D-016 决定。
6. 完整 P5 无论作为主成果还是另行要求的独立成果，都必须满足真实品牌商品边界；P1—P4 使用视觉共享能力不自动成为 P5，也不能因没有商品而被错误阻断。
7. 路由与共享能力属于后台合同，普通用户不需要选择或理解 P1—P5 标签。系统最终仍只交付与任务匹配的成品；标签、损失测试和内部参与信息不进入成品。

X-01 已用一张空路由卡、五张带邻类干扰的产品卡和一张独立第二成果阳性卡完成终验：重视觉 P3 仍只形成 P3；只有明确要求“另外、单独、可脱离采用”后才新增独立 P5。完整证据见[讨论纪要 064](../讨论纪要/2026-07-21-064-X01七张实际验收卡与关闭结论.md)。测试成品只证明本条性质，不成为日常固定模板、额外路由 Agent 或每次生产都要执行的七卡流程。
