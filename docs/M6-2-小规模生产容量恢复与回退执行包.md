## 四、规模语义必须保持准确

### 1. `5 租户 / 200 账号`是什么

- 它是第一阶段的**容量验收包络和运维预期**；
- 200 指全平台合计已启用的自然人账号量级，不是 200 个并发请求；
- 首期预计仍是 2—3 个租户，最多按 5 个租户留安全冗余；
- 它不是许可证上限，不在业务代码中硬编码“第 6 个租户”或“第 201 个账号”禁止创建。

### 2. 当前运行形态

- 保持一个 FastAPI 模块化单体应用实例、一个 Uvicorn worker；
- 继续使用当前进程内 `ModelRequestLimiter`；
- 当前默认候选值 `全局并发 4 / 单租户并发 2 / 单租户每分钟 12` 只作为现场验证起点，
  可依据真实供应商和 ECS 实测向下或有限向上调整；
- 配置调整属于可回退运行参数，不是新的业务裁决；
- 若将来增加第二个应用副本，进程内限流不再代表全局限流，届时另行裁决；本轮不得预建
  Redis、分布式信号量、队列或调度服务。

### 3. SLA 边界

本轮只记录：

- 有界测试中的实际响应、模型耗时、拒绝和恢复；
- 一次应用重启的实际恢复时间；
- 一次隔离恢复的实际耗时；
- 一次版本回退及恢复候选版本的实际耗时；
- 当前每日备份频率和保留数量。

这些是当前环境的实测基线，不得写成对所有企业承诺的 SLA、RTO 或 RPO。

## 六、核心验收数字

### A. 隔离容量夹具

- 隔离测试租户：`5`；
- 合计已启用自然人账号：`200`；
- 每个测试账号都通过当前正式数据关系关联到本租户的品牌、发布账号、`ContentRole`
  和操作授权；可以程序生成，不保存 200 份静态夹具文件；
- 跨租户、错误品牌、错误账号和未授权自然人反证：至少各 `1`；
- 生产数据库新增容量测试账号：`0`。

### B. 模型限流

至少证明：

1. 全局上限以内的请求可以进入；
2. 同一租户超过并发上限时快速返回 `429`；
3. 不同租户合计超过全局上限时快速返回 `429`；
4. 单用户短时重复和单租户分钟速率超限可见；
5. 被拒绝请求不产生任务、运行或版本；
6. 正常完成、模型失败和取消路径均会释放槽位，随后请求可以成功；
7. 供应商 `429/Retry-After`、超时和最终失败继续是有界重试，且不产生半版本。

### C. 有界生产运行

- 认证/健康/授权读取小突发：一组，最多 `20` 个并发请求；不做 200 并发压测；
- 真实 DeepSeek 并发探针：一组，最多使用当前全局并发配置值，输入必须无敏感信息且
  不写入正式品牌知识；
- 应用异常退出并自动恢复：`1/1`；
- 异常恢复后，以现有合法身份完成一份非商品承重 P3 或读取已有 P3：`1/1`。

真实并发探针可以由生产容器直接调用当前适配器，以验证 ECS 到供应商的并发能力；
应用限流本身必须由正式 API 路径的隔离测试证明。不要为了凑并发在生产创建永久租户或
永久用户。

### D. 备份、恢复和回退

- 包含 M6-1 真实租户数据的新备份：`1`；
- 独立临时数据库与应用恢复：`1/1`；
- 恢复后的未设租户上下文 RLS 失败：`1/1`；
- 恢复后的“确认品牌—账号—角色—授权—成功内容版本”关系链：`1/1`；

## 七、最小实现方案

### 1. 数据容量只用隔离测试库

增加一组可重复的容量测试，程序化创建 5 个租户和合计 200 个已启用自然人账号。
测试必须走当前开户、账号、角色和授权的正式服务/仓储边界，不直接用无约束 SQL 伪造
最终状态。

只验证：

- 创建和读取可以在现有超时边界内完成；
- 每个租户只能解析自己的账号、角色、授权和内容目标；
- 200 个账号不会进入别的租户运维摘要或上下文；
- 重复执行幂等或在测试事务/清理后不留孤儿。

记录实际耗时作为观察值，不设置脱离现场依据的毫秒级 SLA。不得把 200 条记录提交为
JSON、SQL seed 或生产数据。

### 2. 限流必须测真实请求路径

使用一个可控制阻塞、成功、失败和 `429` 的测试生成器，通过当前正式 API 路由并发发起
请求，证明本包第六节 B 的七项行为。不要只直接调用 `ModelRequestLimiter` 后宣称 API
已成立。

若发现 React 使用的正式 JSON API 能并发，而 HTML 降级表单因同步模型调用阻塞事件循环，
只做让两条入口共享同一受限执行路径的最小修复；不得增加 worker 数量来绕过，因为多个
worker 会使进程内全局限流失真。

### 3. 备份清单替代 `41` 魔数

对现有 `deploy/backup.sh` 与 `deploy/restore_verify.sh` 做最小升级：

- 备份时生成一个不含名称、正文、Prompt、标识符和密钥的校验清单；
- 清单只记录恢复直接需要的 schema 版本及关键对象数量，例如租户、品牌、用户、发布
  账号、角色、授权、确认品牌版本、成功内容版本、激活资产和对象数量；
- 清单纳入现有校验和；
- 恢复时按该次备份自己的清单比较，不再假定激活资产永远等于 `41`；
- 额外验证恢复库中至少存在一条完整的
  “已确认品牌版本 → 发布账号 → 当前 ContentRole → 操作授权 → 成功内容版本”关系；
- 保留未设租户上下文读取失败、应用 readiness 和对象恢复检查。

这不是建立备份元数据平台。只允许一个随快照保存的最小清单，不新增数据库表、管理页或
长期报告服务。

### 4. 版本回退不降级数据

部署 M6-2 候选前先完成新备份。候选健康后：

1. 记录当前候选 SHA、M6-1 已知可用 SHA `f4697a82d388c91900dcdde9b5fefe21a5320549`
   和当前 Alembic head；
2. 使用现有回退脚本切回 M6-1 SHA；
3. 验证 readiness、正式登录入口和已有 P3 可读取；
4. 再部署 M6-2 候选 SHA；
5. 验证 readiness、M6-1 真实租户关系和现有 P3 仍存在。

不得执行生产数据库 downgrade。若候选引入不可向后兼容迁移，必须在切公网前改成
expand/contract 或其他可回退的最小兼容迁移；无法做到则停止并请求主控裁决。

现有 `diyu-m5-4` 容器、备份目录和 systemd 名称属于稳定生产资源标识，不因里程碑编号
变化而重命名。

当前唯一 GitHub Actions 已绿色，但存在 Actions Node 运行时弃用警告。若官方当前稳定
