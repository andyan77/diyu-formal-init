## 三、用户交互与业务合同

### 1. 明确而轻量的对内入口

- 增加一个可直接访问的“陈列搭配”工作台，例如 `/display`；可以从现有页面提供一个清楚的文字链接。
- M4-1 不建设统一首页、复杂导航、意图分类器或在同一输入框自动判断内容/陈列；这些属于 M4-2。
- 当前入口只执行 `display-merchandising / DM01_wall_hanging_rail_execution`，不出现 P1—P5、发布账号、内容人设、平台、CTA 或拍摄脚本。
- 页面只要求用户自然说明本次库存及必要的临时变化，不显示租户 ID、品牌 ID、组织 ID、合同 ID、资产 ID、Prompt、评分或运行轨迹。
- 当前南城店挂杆档案完整时直接生成，不重复询问容量、上下杆、正挂点或来客方向。
- 如果换成没有最低挂杆条件的门店，只返回一个当前最关键的自然问题；不得猜造空间条件，也不得建设复杂建档表单。

### 2. 完整文字执行成品

用户看到的是一份连贯、可直接照做的文字成品，至少自然回答：

- 这版选择多少件上墙、多少件不上墙，以及为什么；
- 商品怎样组成上下联动的搭配区；
- 顾客面对墙时的左右阅读顺序；
- 上杆主焦点、较弱回应和正挂商品；
- 上下杆侧挂节奏、数量与间距；
- 缺货、厚度或现场变化时的最短替代办法；
- 门店人员实际按什么顺序执行；
- 这是一份内部执行建议，不表示总部批准、系统核验或门店已经完成。

用户不得看到结构化校验载荷、规则名称、知识 ID、生成轨迹或模型推理。系统可以在内部保留最小结构，以核对商品、数量、位置和版本；内部结构不是第二份用户成品。

### 3. 数量、商品和现实边界

- 人工库存只在当前任务内成立，不写成 ERP 事实，不回写库存，不判断交易或销售结果。
- 方案只能使用本次清单中的商品，任何 SKU 的上墙数量不得超过用户输入。
- 上墙与未上墙数量必须能对回本次清单；容量是上限，不是全部挂满的目标。
- 商品颜色、图案、品类、双面关系等承重信息只能来自当前品牌商品资料或用户本次明确更正，模型不得补造。
- `DisplayArtifact` 始终是拟议的内部执行建议；用户说“已经照这版挂了”只构成反馈前提，不升级为系统巡检、评分、验收、批准或整改结论。

### 4. 自然反馈与不可覆盖版本

- 首次生成形成 V1；
- 用户自然反馈后形成 V2，V1 不被覆盖，服务重启后仍可读取；
- 冻结反馈只影响 C 区上杆：ZX-V113 由两件减为一件，总上墙数由 15 变为 14；A、B 区、全部下杆、主焦点、回应点和动线保持不变；
- 页面可以先给简短“本次只改了什么”，随后仍交付一份完整可执行新版；不能只显示差异片段；
- 关键商品、动线、载体或多个区域发生承重变化时允许重编译，但本里程碑不扩展其他载体合同。

## 四、最小工程落点

### 1. 保持模块化单体，不复制第二套系统

在现有 FastAPI、应用服务、端口、PostgreSQL、Alembic、生成适配器和事件机制上增量实现。允许为陈列增加清楚的模块内类型、服务和端口，但不得复制认证、数据库连接、模型 HTTP 客户端、错误处理、CI 或版本基础设施。

内容生产与陈列成果必须保持不同根：

- 内容任务继续生成 `ContentItem`；
- 当前陈列任务只生成 `DisplayArtifact`；
- 不把陈列方案塞进 `content_items/content_versions` 冒充内容；
- 可以复用共同的 `BusinessTask`、`GenerationRun`、`ActivityEvent` 和版本实现方式，但不得让旧 P1 数据改变含义。

物理表和字段是可回退实现选择，不需要新增 ADR。只实现当前闭环直接消费的最少关系，准确表达：

- 陈列任务的租户、品牌、控制/执行组织和实际操作人；
- 南城店当前挂杆轻量档案及版本；
- 当前品牌陈列标准及版本；
- 六款合成商品的陈列相关事实；
- 用户本次原始库存输入和解析后的商品数量；
- `DisplayArtifact` 稳定身份、不可覆盖版本、生成运行和必要事件；
- 当前反馈、受影响范围和 V1→V2 关系。

挂杆、库存输入、内部搭配区和反馈不因字段较多自动升级为新的核心本体类别。不要建立 25 张表、通用空间数字孪生、万能工作流载荷或审批状态机。

### 2. 对内可信作用域

建立最小的对内陈列可信作用域或等价实现，至少包含服务端派生的：

- `tenant_id`
- 当前实际操作人
- 当前品牌
- 南城店控制/执行组织

不得要求或允许客户端通过表单、JSON、URL 参数切换这些作用域。陈列不依赖发布账号和 `ContentRole`；不得为了复用现有 `TrustedScope.account_id` 而伪造一个内容账号。

新建的所有租户拥有表继续使用：

- `tenant_id NOT NULL`
- 应用查询显式收窄品牌、组织、任务和操作人
- PostgreSQL `ENABLE/FORCE ROW LEVEL SECURITY`
- 请求角色无表所有权、无 `BYPASSRLS`
- `USING` 与 `WITH CHECK`
- 事务级可信租户上下文，缺失时失败关闭

只为新增陈列表做一组直接的租户根与品牌/组织反证，不重新复制 M3-2 的整个攻击面。

### 4. 生成器与必要自检

复用现有 DeepSeek HTTP 适配层的超时、有界重试、用量和失败记录；可以为陈列定义独立的小型输入/输出合同，但不建设第二个供应商客户端、多模型路由或 DIFY 工作流。

模型输入只能包含当前合法的：

- 品牌陈列标准及版本；
- 南城店挂杆档案及版本；
- 当前六款商品的陈列相关事实；
- 用户本次库存与反馈；
- 当前适用的最小陈列资产；
- 修订时的上一版本必要结构和正文。

模型输出落库前只做直接服务当前合同的确定性检查：

- 没有清单外商品；
- 任一商品使用量不超过输入；
- 上墙、未上墙和总量可对账；
- 同时存在上下、左右、主/次焦点、正/侧挂、数量/间距、替代和步骤；
- 建议与已执行/核验事实没有混淆；
- 局部反馈时未受影响关系得到继承。

检查失败时可以进行一次有界修复生成，仍不合格则明确失败；不得写半成品版本，不建设 27 个在线评分器、模型自评平台或循环重生成。

离线确定性替身必须基于结构化输入产生结果，不得固定返回黄金全文。真实 DeepSeek 在进入 `REVIEW` 前至少完成一次同页 V1 和同任务反馈 V2，并由执行侧人工按 D-005 合同审阅。

如本机到 `api.deepseek.com:443` 再次直连失败，可复用 M3-1 已验证过的、仅限该目标的临时本机 SSH 出站通道完成本里程碑真实验收；不得因此修改 ECS、Nginx、DNS、域名或历史 DIFY。验收后停止并清理临时进程。网络问题只阻塞真实模型终验，不得阻塞全部本地实现。
