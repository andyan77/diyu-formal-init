## 三、冻结的用户路径

### 1. 中立首页

将 `/` 收敛为笛语中立首页：

- 使用正式 Logo、favicon、墨黑、朱砂红、石灰灰和呼吸白；
- 只显示两个已经实现的入口：
  - **内容生产（对外）**：面向品牌受众生产内容；
  - **陈列搭配（对内）**：面向组织和门店生成经营执行建议；
- 每个入口只用一句人话说明“谁使用、得到什么”，不显示模块 ID、合同、数据库、Prompt、知识资产或技术状态；
- 不显示商品管理、CRM、ERP、素材中心、管理后台等未实现空菜单；
- 首页本身不自动创建任务，也不因打开页面自动获得某个业务身份。

用户必须显式选择入口。第一版不使用一个输入框猜测用户属于哪个应用，不建设意图分类器、LLM 路由、置信度阈值或跨应用状态机。

### 2. 两个现有工作台

- 内容工作台迁移到稳定路径 `/content`，保留现有 P1 自然输入、友好交流、V1/V2、读取旧版本、主动保存和明确复用能力；
- 陈列工作台继续使用 `/display`，保留现有库存清单、V1、明确反馈 V2、读取旧版本、缺口提示和无关反馈追问；
- 两个工作台均提供清楚的“返回应用首页”入口；
- 已进入某个工作台后仍以自然语言交互，不要求用户选择 P1—P5、填写技术表单或输入内部 ID；
- 为避免旧本地链接失效，`/?task=...&version=...` 可以最小重定向到 `/content?task=...&version=...`；不得为此建设路由兼容框架。

### 3. 合成演示会话

M4-2 只需要一个诚实的本地演示会话适配器，不实现生产认证。

首页的两个入口分别启动一个**服务器固定映射、应用绑定、不可由客户端改写作用域**的合成演示会话：

| 入口 | 实际操作人 | 组织与业务身份 |
|---|---|---|
| 内容生产（对外） | 总部内容运营甲 | 折线之间总部；折线之间品牌母账号·抖音；总部零售/服务专家 |
| 陈列搭配（对内） | 南城店陈列执行甲 | 折线之间·南城店执行组织；折线之间·南城店 |

最小实现要求：

- Cookie 只保存或表示一个服务器可验证的合成会话标识，不接收租户、品牌、组织、用户、账号或角色 ID；
- 内容会话只能解析内容作用域，陈列会话只能解析陈列作用域；
- 缺少、伪造或错误应用的会话必须失败关闭；内容会话不能直接调用陈列写接口，陈列会话不能直接调用内容写接口；
- 从首页切换应用时允许替换当前合成会话；再次进入原应用后，原任务和版本仍在正确作用域内，不因切换被删除或改名；
- 该机制必须明确标注为“折线之间合成演示”，不得在 README、页面或交付报告中冒充真实登录、真实授权或生产 RBAC；
- M5-4 的生产认证将替换这个最外层演示适配器，业务服务和可信作用域合同不得依赖演示按钮或固定人名。

可以使用两个固定签名标记、一个极小的服务器映射或同等简单实现；不得引入 JWT 平台、OAuth/OIDC、Redis 会话、权限管理后台、用户邀请、组织切换器或通用策略引擎。

这里“选择应用后进入对应演示身份”只是为了让两个已有合成角色分别证明两条路径，不是产品规则。真实上线后，同一个自然人如果同时获得内容和陈列授权，应在保持本人身份不变的情况下看到两个获授权入口；本里程碑既不实现也不冒充已经证明了这项生产认证能力。

## 四、身份呈现与编译边界

### 1. 内容生产

内容工作台用一个简短、不可编辑的自然语言身份摘要，让用户知道当前“代表谁、以什么账号和人设表达”：

- 品牌：折线之间；
- 实际操作人：总部内容运营甲；
- 代表组织：折线之间总部；
- 发布账号：折线之间品牌母账号·抖音；
- 内容角色：总部零售/服务专家。

生成时继续由服务端编译已有 `BrandContext`。实际操作人、组织、发布账号、内容角色和受众不得合并成一个“用户画像”，也不得从页面字段回传后再信任。

### 2. 陈列搭配

陈列工作台只显示与内部执行有关的摘要：

- 品牌：折线之间；
- 实际操作人：南城店陈列执行甲；
- 执行组织：折线之间·南城店；
- 当前门店：折线之间·南城店；
- 当前能力：墙面双层挂杆执行方案。

陈列上下文、运行留痕和用户页面均不得出现或依赖：

- `PublishingAccount`
- `ContentRole`
- 平台
- CTA
- 对外受众
- P1—P5 产品标签。

不得为了统一页面而增加一个混合 `UniversalPersona`、万能 `WorkspaceContext` 或把两种作用域塞进同一个可选字段对象。共享的只是会话校验、页面骨架和导航；两种业务身份继续使用已经存在的独立类型。

### 3. 只读身份摘要的落点

若页面在生成前需要读取当前身份，优先通过现有应用服务/仓储形成最小只读投影；不要在 HTML 渲染函数中直接写 SQL，也不要复制第三套身份查询。

身份摘要服务只为当前两个页面消费，不建设通用用户中心、账号矩阵管理、组织树浏览或身份缓存。

## 五、明显错入口时的最短处理

顶层选择决定默认业务合同。系统不按“搭配、商品、脚本”等单个关键词自动换应用。

只为两个**成果要求明确冲突**的高信号场景增加最小保护：

1. 用户已在内容生产入口，却明确要求“按门店本次库存给出双层挂杆的上下杆、数量和执行步骤”：
   - 不创建 `ContentItem`；
   - 返回一句：“这是给门店内部执行的陈列任务，请切换到陈列搭配。”
   - 页面提供返回首页或进入陈列入口的自然动作。

2. 用户已在陈列搭配入口，却明确要求“生成一条面向顾客发布的抖音口播/拍摄脚本”：
   - 不创建 `DisplayArtifact`；
   - 返回一句：“这是面向外部受众的内容任务，请切换到内容生产。”
   - 页面提供返回首页或进入内容入口的自然动作。

保护规则判断的是用户明确要求的**最终成果与使用对象**，不是词频路由。普通含混输入继续停留在当前应用，由现有能力生成或只问一个必要问题；系统不得静默切换、同时生成两份成果或把 P5 当经营陈列。

可以使用两段很小、可读的确定性冲突判断或等价实现。本里程碑不调用 LLM 判断应用，不实现完整 X-01；P2—P5 价值路由和 null-route 继续属于 M5-1。

错入口提示不得产生任务、运行、版本、保存记录或失败生成记录；它只是交互提示，不是第三类业务成果。

## 六、最小工程落点

### 1. 复用现有模块化单体

继续使用现有 FastAPI、两套应用服务、PostgreSQL、RLS、版本与 CI。允许：

- 重构 `SessionAuthority`，消除 `display_scope()` 静默换人的捷径；
- 增加两个固定演示会话映射及应用专用的 `require_content` / `require_display` 或等价入口；
- 增加中立首页和两个工作台的最小共享页面骨架；
- 增加只读身份摘要；
- 增加最小跨应用冲突提示；
- 按真实路由变化更新 OpenAPI、README 和现有测试。

不需要新建通用编排层、应用插件系统或运行时数据库注册表。代码中可以保留两个稳定模块标识：

- `content-production`
- `display-merchandising`

第三个未来应用真正立项时再按 ADR-003 增加，不预建空模块。

### 2. 数据与迁移

本里程碑已有全部身份、组织、账号、角色和门店合成数据，原则上不需要新增业务表或数据库迁移。

如果执行端认为必须迁移，只有在它直接补齐“一个会话不能静默更换实际操作人”这条验收、且无更轻实现时才允许；先在范围内独立判断，不把普通代码结构包装成用户裁决。

不新增知识资产、不改变 243 条登记、不激活新资产。M4-2 只组织现有两条软件能力，不建设知识检索。

### 3. API 与错误边界

- 现有内容和陈列业务 API 路径尽量保持；
- 内容 API 要求内容演示会话，陈列 API 要求陈列演示会话；
- 缺失或无效会话返回 `401`；合法但应用不匹配的会话返回 `403` 或现有契约中等义且清楚的拒绝，不得回退成成功；
- 客户端提交 `tenant_id`、`brand_id`、`organization_id`、`user_id`、`account_id`、`content_role_id` 等字段不能改变作用域；优先由请求模型拒绝额外字段，或明确忽略并以服务端作用域为准；
- 演示会话启动路由是本地 UI 适配器，可不进入正式 OpenAPI；业务 API、错误返回和现有实际路由必须与 `openapi.json` 一致。

### 4. 页面与视觉

直接消费 `assets/brand/diyu-vi/`。只做当前三个页面的清楚统一：

- 中立首页；
- 内容工作台；
- 陈列工作台。

共享 Logo、颜色、基础排版、按钮和身份摘要即可。不得建设：

- Figma 交付；
- 组件库或 Storybook；
- 独立设计系统；
- 动画系统；
- 多主题；
- 复杂响应式矩阵；
- 管理后台；
- 全量前端框架迁移。

保持服务端渲染或现有轻量实现，不为三个页面引入 SPA。

## 七、必须通过的九项验收

1. 打开 `/` 只看到“内容生产（对外）”与“陈列搭配（对内）”两个可用入口，不创建任务，不出现未来空模块。

3. 选择陈列入口后，页面明确显示南城店陈列执行甲、南城店执行组织和当前门店；完成现有 DM01 的 20→15 V1 与反馈后的 14 件 V2，V1 保留。
4. 捕获或检查两条实际应用输入：内容输入包含当前合法发布账号与 `ContentRole`，不含陈列门店档案；陈列输入/运行回执包含执行组织、门店和陈列资料，不含发布账号、`ContentRole`、平台、CTA 或 P1—P5。

6. 客户端通过请求体、查询参数或 URL 注入另一租户、品牌、组织、用户、账号或角色 ID，均不能改变服务端可信作用域。

验收比较身份、应用合同、可观察行为和副作用，不比较固定 HTML 字符串、像素截图或测试数量。

## 九、执行自主权与停止边界

执行端在本里程碑范围内获得充分授权，可自主：

- 选择最小的签名演示会话实现；
- 重构现有 session、路由、HTML、服务只读投影和定向测试；
- 修复范围内的 Cookie、重定向、错误返回、OpenAPI 和回归问题；
- 提交、推送 `main`、读取 CI、在 CI 失败时定位并修复；
- 在不改变冻结结果的前提下采用比本包建议更简洁、更稳健的实现。

以下情况才停止并提交一张自然语言决策卡给主控：

- 必须改变“用户先选顶层应用”的既有裁决；
- 必须把一个实际操作人静默替换成另一个人才能继续；
- 必须合并 `PublishingAccount`、`ContentRole`、操作人或执行组织的语义；
- 必须新增生产认证、外部服务、密钥、费用、生产写入或破坏性迁移；
- 只能降低租户隔离、RLS、版本保留或既有 P1/DM01 用户结果才能继续。

普通文件组织、函数命名、Cookie 编码、CSS、HTTP 状态细节和测试安排不得上升为用户裁决，不得为它们停工。

## 十、验证与审查上限

1. 开发期间只运行受影响的 session、UI、路由与双应用测试；
2. 新增测试只覆盖九项验收中现有套件缺失的直接证据，优先扩展现有测试文件，不复制测试框架；
3. 不运行真实模型、不做浏览器矩阵、像素截图回归、性能压测或安全扫描平台；
4. 稳定候选只统一运行一次 `git diff --check`、`make lint`、`make typecheck`、`make golden`；
5. 执行侧进行一次范围内自审；推送候选并取得一次远程绿色 CI；
6. 修复后只复核原失败和受影响回归，不重新开启无边界找问题；
7. 九项验收成立后立即把 `MILESTONE.md` 置为 `REVIEW`，提交并推送，停止工作；不得自行写 `CLOSED`，不得启动 M5-1。

执行侧最终只需报告：

1. 实现提交、状态提交、远程分支和 CI 链接；
2. 用户现在如何从首页进入两个应用；
3. 两个应用分别绑定的实际操作人和业务身份；
4. 同 Cookie 静默换人缺口如何消除，跨应用会话如何失败关闭；
5. P1 与 DM01 哪些既有能力完成实测且未回退；
6. 两个错入口场景是否只提示切换且没有产生业务副作用；
7. 本地统一检查与远程 CI 结果；
8. 明确尚未证明生产认证、真实授权、真实租户、真实门店或市场效果；
9. 唯一下一动作：主控终审 M4-2。
