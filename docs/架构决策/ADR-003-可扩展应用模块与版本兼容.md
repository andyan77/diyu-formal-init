# ADR-003：可扩展应用模块与版本兼容

- 状态：已接受
- 接受日期：2026-07-17
- 对应决策：D-024

## 背景

内容生产和陈列搭配是第一版计划实现的应用，不是系统永久上限。未来可能增加商品管理等应用，但为未知需求预建万能对象、空表、空接口和空菜单，会让共享核心膨胀并制造假兼容。

## 决策

1. 顶层采用：**稳定共享核心 + 可注册应用模块 + 版本化输入/输出合同**。
2. 第一版只计划实现：
   - `content-production`；
   - `display-merchandising`。
3. 每个已实现模块必须声明：稳定模块标识、合同版本、输入/输出类型、依赖的共享对象、权限动作、状态机、发布事件与迁移规则。
4. 运行时注册表只登记已完成、已迁移并通过验收的模块。已实现模块可按租户授权启停；未实现模块不进入构建产物。
5. 商品管理等未来应用在未立项和开发前只存在于路线图与保留命名空间中；不创建空表、空接口、空菜单、空后台任务或万能 JSON 载荷。
6. 第一版的商品与库存只是当前两个模块的权威外部输入/只读投影，不等于已开发“商品管理应用”。
7. 模块私有对象、表、状态机与业务规则归模块所有；其他模块只能通过公开服务合同、稳定标识或已发布业务事件访问，不直接改写对方私有数据。
8. 兼容规则：
   - 次版本只允许增加可选字段或不改变旧语义的子类型；
   - 删除字段、改变字段含义或状态机语义必须发布新主版本；
   - 外部连接器字段映射单独版本化，ERP/POS 字段名不得泄漏为领域合同；
   - 对未知可选字段可安全忽略，未知状态或枚举不得静默当作默认值，必须拒绝或降级；
   - 数据迁移采用先扩展、再迁移、后收缩的方式，在承诺期内支持新旧合同共存。
9. 可以承诺稳定标识、模块隔离、加法优先、显式迁移与主版本共存；不承诺尚未知道的未来应用无需任何改造就能天然兼容。

## 影响

- 当前两个应用可独立演进和按租户启停，而不破坏共享核心含义。
- 新应用真正立项时才根据已验证需求建模和接入，不为假设的未来复杂度付成本。
- “顶层可扩展”与“所有未知需求向后兼容”被明确区分，避免无法验收的承诺。
