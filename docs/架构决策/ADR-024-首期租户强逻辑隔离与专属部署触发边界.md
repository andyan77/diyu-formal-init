# ADR-024：首期租户强逻辑隔离与专属部署触发边界

- 状态：已接受
- 接受日期：2026-07-22
- 实施补充：2026-07-22（PostgreSQL RLS 双保险）
- 对应决策：D-008
- 上游边界：[ADR-019](ADR-019-第一版25类最小语义模型与兼容扩展边界.md)、[ADR-020](ADR-020-多品牌共享隔离与经营关系.md)、[ADR-021](ADR-021-组织官方素材工作区与持续供给.md)
- 后续决策：D-010、D-011 已分别由 [ADR-025](ADR-025-首期禁止跨品牌跨租户内容学习与复用.md)、[ADR-026](ADR-026-内容前情复用与正式知识升级分离.md) 关闭

## 背景

笛语第一版面向多个独立采购企业提供共享 SaaS 服务。系统必须防止一个签约企业的数据被另一个企业看见、检索、导出或组装进模型上下文；同一企业经营多个品牌、组织和门店时，也不能因为同属一个租户就自动混用品牌专属知识、组织本地资料或账号私人素材。

隔离要求不等于为每个品牌、区域或门店建设独立数据库。若第一版同时维护共享库、租户独立库和专属实例三套架构，会直接增加迁移、升级、备份恢复和故障排查成本，而当前没有真实合同要求证明这种复杂度必要。

## 决策

### 1. Tenant 是生产数据隔离根

第一版采用共享 SaaS 下的租户级强逻辑隔离。`Tenant` 是签约和生产数据隔离根；`Brand`、`OrganizationalParty` 和账号私人范围是租户内部的适用与可见范围，不替代租户边界。

所谓“强逻辑隔离”，是指其他租户的数据不仅不能在界面中显示，也不能通过普通查询、全文或向量检索、文件访问、后台任务、缓存、生成与行为事件、运行日志、导出、备份恢复或模型上下文组装被当前租户调用。租户身份必须来自已经验证的系统会话或服务端执行上下文，不能由用户输入任意指定。

### 2. 租户内部继续按既有范围隔离

共享 SaaS 不改变已经确认的内部边界：

- 品牌专属知识只在当前任务明确适用的品牌或品牌集合中调用，不能因为同属一个租户而自动串用其他品牌人格、商品故事或连续记忆；
- 组织本地资料继续按真实组织和业务关系使用，经营关系本身不产生数据读取权；
- 私人素材和私人协作偏好继续按账号私人范围处理，不因总部、区域或管理员身份向上穿透；
- 领域通用知识可以跨租户复用，但不得混入任何租户专属事实、素材、记忆或派生内容。

本决策不新增逐账号、逐知识的复杂 ACL，也不改变 ADR-020、ADR-021 已冻结的品牌、组织和素材语义。

### 3. 第一版不按品牌、区域或门店分库

第一版不为每个品牌、区域代理、门店、发布账号或私人素材箱建设独立数据库，也不复制一套平行知识库。物理存储可以共享，但任何生产读取和检索都必须先确定当前租户，再应用组织、品牌和私人作用域。

全文索引、向量索引和缓存属于可重建的派生层，不能成为绕开权威数据范围的第二入口。模型只接收已经完成作用域过滤的上下文。

### 4. 跨品牌和跨租户学习默认关闭

第一版不因共享基础设施而自动获得跨品牌或跨租户复用生成内容、私人资料、派生特征或训练模型的权利。跨品牌和跨租户学习默认关闭。

这一条是 D-008 的隔离底线，不等于已经批准任何匿名汇总、特征复用或训练例外。D-010 后续已由 ADR-025 关闭：第一版不开放上述能力，未来若有真实收益和明确授权，必须重新立项。

### 5. 专属部署只由真实合同触发

每租户独立数据库、专属密钥、独立网络或专有实例不作为第一版公共架构的预建能力。只有具体租户的真实合同因法律合规、数据驻留、客户安全审计或可验证的性能隔离而提出明确要求时，才针对该租户形成版本化部署方案。

触发专属部署不改变业务语义模型。25 类逻辑对象、租户/组织/品牌作用域和应用合同继续保持一致，避免以后为了部署档位复制业务系统。

## 最小实施与验收要求

D-008 关闭的是架构档位，不证明隔离已经实现。进入软件实现后，至少必须证明：

1. 当前租户由可信会话或服务端上下文确定，用户不能通过请求参数切换租户；
2. 首期权威关系存储采用当前项目独立 PostgreSQL 数据库和共享 schema；所有租户拥有表必须有不可为空的 `tenant_id`，应用查询继续显式使用服务端作用域过滤，同时对这些表启用并强制执行 RLS；
3. 请求应用角色不是表所有者、没有 `BYPASSRLS`；迁移/管理角色与请求角色分离。每个请求或后台任务在事务内设置可信租户上下文，连接归还连接池前不留下会话级租户状态；缺少或非法上下文时失败关闭；
4. RLS 同时约束读取与写入（`USING` 与 `WITH CHECK`）。故意漏写应用层租户条件时仍不得读到诱饵租户，A 租户会话写入 B 的 `tenant_id` 必须被数据库拒绝；
5. 权威数据、原件、全文/向量检索、缓存、后台任务、事件记录、运行日志、导出和备份恢复都保持同一租户范围；
6. 检索先把服务端可信租户及适用范围放进查询，再进行关键词或向量 Top-K；禁止先做全库召回、再在应用端删除越界结果。未来若采用 pgvector，向量表继续启用 RLS；外部向量引擎必须由服务端强制集合/命名空间与元数据过滤；
7. 使用“折线之间”与一个诱饵租户进行越界对抗，证明诱饵租户的商品、知识、素材、生成记录和私人资料不会进入当前结果；
8. 同租户多品牌的相似声纹、商品故事和连续记忆也必须通过反向泄漏测试，但只有真实多品牌租户出现时才能宣称真实运行验证。

本补充关闭数据库产品和租户根强制机制：第一阶段不采用每租户 schema。RLS 只兜住租户根，品牌、组织、账号与私人范围继续由应用核心按既有合同显式收窄；未来私人素材表成为当前消费者时，再对其增加必要的所有者级数据库策略，不预建通用权限策略平台。对象存储、索引引擎等仍在有直接消费者的里程碑内选择。

## 明确不做

第一版不建设：

- 每品牌、每区域、每门店或每账号独立数据库；
- 每租户 schema 及其重复迁移、索引和运维链；
- 同时维护共享库、租户独立库和专属实例三套生产架构；
- 为未来可能的专属部署预建微服务群或数据同步体系；
- 跨品牌或跨租户训练、匿名汇总和特征复用能力；
- 用“同属一个租户”替代品牌、组织或私人作用域过滤。

## 影响

D-008 正式关闭。第一版物理架构可以围绕一套共享 SaaS 和统一逻辑模型继续设计，不再等待每租户分库方案，也不需要为品牌、区域和门店复制数据库。

本 ADR 只是已接受的架构合同，不是生产隔离通过证明。物理数据库、检索、对象存储、后台任务和泄漏对抗仍须在后续软件纵向闭环中实现并验证。
