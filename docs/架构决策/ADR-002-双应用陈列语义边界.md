# ADR-002：双应用陈列语义边界

- 状态：已接受
- 接受日期：2026-07-17
- 对应决策：D-022

## 背景

系统第一版包含内容生产与陈列搭配两个应用。内容生产中的搭配/陈列用于视觉表达，允许明确标识的概念、摆拍和创意演绎；面向企业组织与渠道的陈列则用于培训、经营方案、门店执行、巡检和整改。两者如果共用同一产物和状态模型，会把内容效果图误当为可执行方案，或把建议误当为门店实况。

## 决策

1. 第一版交互 UI 在逻辑顶层明确提供两个应用入口：**内容生产（对外）**与**陈列搭配（对内）**。用户进入其中一个应用后仍使用自然对话，不被强制填写表单；当前顶层选择确定任务默认适用的应用合同，不是 P1—P5 的二次选择器。
2. 系统正式区分两个语义域：
   - **内容表达型搭配/陈列**，归属 `content-production`；
   - **零售经营型陈列**，归属 `display-merchandising`。
3. P1—P5 只属于 `content-production`。顶层选择 `display-merchandising` 后，即使输入了同一件商品并调用相同的色彩、造型或组合知识，也不启动 P5、不生成 `ContentItem`，而按独立陈列合同形成 `DisplayArtifact`。
4. 共享核心中仅保留含义中性的商品、属性、规则、证据、授权、经营上下文和 `ProductComposition`。`ProductComposition` 只表达组合关系，不证明已在门店陈列。
5. 内容侧使用 `ContentBrief` 与 `ContentItem`，必须区分 `concept`、`staged` 和 `actual`，并单独标记门店现状、可售库存或经营结果主张。
6. 经营陈列侧使用模块私有上位类 `DisplayArtifact`，子类型为：培训指南、经营陈列概念案、门店陈列执行方案、陈列巡检记录和陈列整改任务。这五者是子类型，不增加五个共享核心类别。
7. 经营侧可以创意设计、教学模拟和方案预演，但必须区分概念/建议与门店已经执行的结果。第一版不建设笛语内部审批状态机、库存有效性判断或现场真值验证；门店是否采用、怎样执行由经营主体决定。
8. 内容成品不得反向成为门店执行方案或现场证据；经营概念案被内容侧引用时，必须转成内容任务并重新经过内容侧的事实、授权和验收规则。
9. 第一版以用户手工提供的门店、空间、商品和数量条件生成建议，不核验现实库存，也不把建议标记为系统已经批准或现场已经执行。第一版不接 ERP；未来若接入，仍由外部经营系统负责库存真值，见修订后的 [ADR-007](ADR-007-外部经营系统可选接入与库存事实降级.md)。
10. 如果用户的明确成果要求与当前顶层应用冲突，系统只需用一句最短提示建议切换应用，不能静默混用另一应用的产物与验收合同；不为此建设第二个意图分类服务、状态机或重复知识库。

## 影响

- 两个应用分别拥有任务、产物、状态机、权限与验收合同，避免同名异义。
- 共享的服装搭配知识不需要复制，但不会因共享而丢失业务语境。
- 输入不足时仍可生成不依赖该输入的概念或培训建议；只有用户明确要求数据依赖方案时才询问最低输入。任何建议都不能由模型冒充门店已执行结果。
- 顶层应用选择只固定成果语义和默认路由，不牺牲任一应用内的自然对话与主动协作。
