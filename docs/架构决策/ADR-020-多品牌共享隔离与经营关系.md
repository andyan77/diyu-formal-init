# ADR-020：多品牌共享隔离与经营关系

## 背景

同一企业可能用一套组织和人员经营多个品牌，集团也可能由不同完整组织分别运营品牌；集合店、买手店和散货商户又可能销售并不由自己控制的品牌商品。系统既不能按品牌复制虚假组织和数据库，也不能因为对象同属一个租户就自动串用品牌人格、商品故事、连续前情和知识。

## 决策

### 1. 三条作用域轴与一次任务编译

`Tenant` 负责合同和数据隔离，`OrganizationalParty` 负责真实业务控制，`Brand` 负责内容、商品、知识和规则的适用范围；三者不得互相替代。组织、账号、角色和数据库不因品牌数量机械复制。

系统在任务时根据当前租户、有效经营关系、应用、组织、品牌、账号/角色或陈列责任组织、商品/知识范围及有效期编译本次上下文。可以唯一推断时不追问；多个合法解释会实质改变成果时才自然追问一次。

### 2. 对外内容拆开一个主要表达身份与引用品牌

每份 `ContentItem` 只有一个主要表达身份，并可以引用零到多个品牌。集团、集合店或买手账号使用自身身份谈多个品牌商品，不混合多个品牌人格，也不冒充上游品牌；若两个品牌需要各自发声，形成两份成品。

主要表达身份决定整篇观点、声纹和关系姿态；引用品牌当前有效的商品事实与明确禁区只约束涉及该品牌的主张。冲突时删改、收窄、省略或拆分，不平均品牌人格。

### 3. 组织、账号、角色、商品和知识的共享边界

- 真实组织按事实建立，同一组织可以服务多个品牌；品牌不是组织部门。
- `PublishingAccount`、`ContentRole` 各由一个组织控制，并显式声明企业共同、单品牌或明确品牌集合的适用上限；适用上限不等于每篇内容混用全部品牌。
- 自然人登录身份不按品牌复制，具体使用资格继续由 `AuthorizationGrant` 决定。
- 稳定 `Product` 有一个租户和目录控制组织，品牌关联可为空、单一或真实集合；商品关系和 ERP 品牌码不产生品牌人格或官方发声资格。
- 正式知识只分系统通用、企业共同、明确品牌/品牌集合和组织本地四种范围；私人素材不进入企业共同知识。

“企业共同”不是杂物桶。只有能在覆盖品牌中原义使用的内容，或明确以企业/集团为主体的共同事实，才能进入该范围；它可以为描述集团事实而提及品牌名称，但不得承载某一品牌独有的定位、声纹、商品故事、活动、账号连续记忆或成功文案。混合内容拆成最小知识陈述后分别定域。

只有 ERP 品牌码、没有企业确认的独立 D-014 战略基线时，品牌码只用于商品识别和过滤，内容继续沿用企业共同表达。

### 4. 对内陈列不套用对外 IP 人设

`DisplayArtifact` 按一个任务责任组织、实际商品、引用品牌和当前适用陈列规则形成，不读取 `PublishingAccount` 或 `ContentRole` 决定专业结论。集合/买手店默认使用领域知识、自己的店规和实际商品事实；上游规则只有来源清楚、版本当前且明确适用于当前店铺或商品时才调用。多品牌规则冲突时收窄或分区，不静默混用。

### 5. 四类承重经营关系

第一版只区分：

1. 真实组织控制；
2. 按必要品牌、区域和时间成立的渠道合作；
3. 品牌治理或运营；
4. 只销售、经销或选品。

同一组织对不同品牌可以具有不同关系。`OperatingRelationship` 只证明业务资格，不自动产生具体人员权限、发布账号所有权或操作权、品牌知识维护权、私人素材/记忆/本地事实读取权，也不产生总部向代理和加盟店的层级穿透。品牌治理、知识维护和官方账号运营可以分别成立；销售商品不等于取得品牌人格、设计原意或官方陈列规则。

## 明确不做

第一版不新增品牌子租户、每品牌物理数据库、品牌工作区、跨品牌授权对象、人设变体、知识副本、逐账号/逐知识 ACL、每品牌 Agent、完整业态状态机、合同规则引擎或关系审批流。

## 最小验证与证据边界

D-017-C 三例完成语义校验：同组织 A/B 强诱饵没有合法泄漏路径；集团账号能够以集团身份引用 A/B 而不混人格；集合店可以以自身身份解释 A/B 商品而不取得上游官方资格。既有总部—独立加盟店路径覆盖渠道合作负边界。3/3 通过，没有发现新增核心类别的必要。

这些是合成语义与合同证据，不是生产实现证据。首期正式试点仍是单品牌；DIFY/Agent 编排、物理检索与数据库隔离、真实多品牌资料接入、多品牌陈列运行、并发性能和企业采用仍未验证。未来实施验证不得反向改写本 ADR，除非真实业务出现新的控制、权限或生命周期语义。
