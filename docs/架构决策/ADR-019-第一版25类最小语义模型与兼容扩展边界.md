## 背景

D-012 已接受业务任务驱动的混合知识路线，并把第一版轻量语义模型控制在 15—25 个核心类别。P1—P5、媒体合同、品牌战略、账号角色、运行留痕和首期墙面挂杆合同关闭后，D-016 可以依据真实消费者判断哪些对象必须拥有稳定身份，哪些只应是词表、配置、任务输入、成品内部结构或运行投影。

本 ADR 不是数据库表设计或知识图谱建图方案。它冻结全系统共同理解的业务对象、承重关系和兼容扩展边界；物理对象可以分别落在关系库、文档、对象存储、事件记录或派生索引中。

## 决策

### 1. 一等对象准入门槛

一个候选只有同时满足以下条件才进入第一版核心类别：

1. 第一版内容生产、陈列搭配、权限、知识检索或留痕已经存在明确消费者；
2. 至少需要跨任务独立引用/查询、独立权限/归属、独立版本或独立生命周期中的一项。

字段多、有 ID、出现在 Prompt、可以画进图谱或未来可能有用，均不足以升格。25 类不等于 25 张表，也不表示全部进入知识图谱。

### 2. 第一版 25 类

| 组 | 一等语义对象 |
|---|---|
| 经营与内容身份（10） | `Tenant`、`OrganizationalParty`、`OperatingRelationship`、`UserIdentity`、`AuthorizationGrant`、`Brand`、`BrandStrategyBaseline`、`AudienceProfile`、`ContentRole`、`PublishingAccount` |
| 商品、知识、全局能力与素材（8） | `Product`、`ProductComposition`、`KnowledgeStatement`、`EvidenceRecord`、`GenerationMethod`、`IndustryRoleTemplate`、`PolicyRule`、`MaterialAsset` |
| 规划、任务、成果与留痕（7） | `ContentPlan`、`ContentSeries`、`BusinessTask`、`ContentItem`、`DisplayArtifact`、`GenerationRun`、`ActivityEvent` |

### 3. 领域库四类逻辑资产的归宿

| 逻辑资产 | 语义归宿 |
|---|---|
| A/B/C 领域知识卡 | `KnowledgeStatement`，类别、主题和同义词使用受控词表；来源由 `EvidenceRecord` 关联 |
| D/E 生成与制作方法卡 | `GenerationMethod` |
| F 行业角色模板卡 | `IndustryRoleTemplate`；只能派生企业确认的 `ContentRole`，不能直接运营账号 |
| H 评测规则卡 | `PolicyRule.evaluation_rule`；规则定义与某次评测结果分离，后者进入 `ActivityEvent` |

`DomainConcept` 不作为一等对象；面料、版型、色彩、穿着/陈列场景等继续作为受控词表和分类结构存在。

### 4. R1—R10 承重关系

| 编号 | 决策 |
|---|---|
| R1 | 每个租户业务组织、品牌、商品、内容角色、发布账号、任务、成果、生成运行和事件恰属一个 `Tenant`；租户知识、方法、模板、规则和素材也必须显式落在该租户，系统全局资产才使用全局作用域。稳定 `Product` 恰有一个目录控制 `OrganizationalParty`；品牌关联必须显式，可为空（无稳定品牌/企业共同商品）、为一个普通品牌，或在真实联名/集合关系下为明确品牌集合，且全部与商品同租户。品牌码或商品品牌关联不自动产生独立品牌人格。 |
| R2 | 每个 `OperatingRelationship` 恰有一个源组织、一个目标组织或品牌及一个租户作用域；同一端点可因品牌、区域和有效期存在多条关系。具体关系类型由 D-017 决定。 |
| R3 | 每个 `ContentRole` 和 `PublishingAccount` 恰由一个 `OrganizationalParty` 控制并属于一个租户；品牌是独立适用范围，只能明确为企业共同表达、单一品牌或一个明确品牌集合，未声明时不得默认跨品牌。每个 `AuthorizationGrant` 恰关联一个 `UserIdentity`、一个控制组织和一个授权目标/范围；同一角色或账号可先后或同时授权多人。 |
| R4 | 一个 `IndustryRoleTemplate` 可派生零到多个 `ContentRole`；一个 `ContentRole` 最多保留一个来源模板引用。派生后独立确认、独立换版，模板更新不得静默覆盖。 |
| R5 | 一个品牌或企业表达身份可以保留多个 `BrandStrategyBaseline` 历史修订，但任一有效时点最多一个启用版本；任务按其显式品牌范围保存本次实际使用的对应基线版本，单品牌任务通常只有一份。 |
| R6 | 每个 `BusinessTask` 恰有一个发起/控制 `OrganizationalParty`、一个顶层应用和一个明确品牌适用范围；品牌范围只能是企业共同表达、单一品牌或显式品牌集合，不能因同属一个租户而自动串用。内容任务只产生零到多个 `ContentItem`，陈列任务只产生零到多个 `DisplayArtifact`；同一成果不得兼具两种身份。 |
| R7 | 每个 `ContentItem` 恰属一个任务、一个代表组织、一个发布账号、一个当次有效内容角色和一个当前已注册的主要内容产品合同；第一版注册集为 P1—P5。它保存本次实际使用的有效品牌/企业表达基线集合，单品牌任务通常恰为一份，显式跨品牌任务才可为多份。商品关系按产品合同条件触发，不把账号与角色永久锁为一对一。 |
| R8 | 每个 `DisplayArtifact` 恰属一个陈列任务和一个执行/控制组织；不绑定发布账号、对外内容角色或 P1—P5。D-005 执行型成果必须满足本次商品来源和数量约束。 |
| R9 | 一个 `ContentItem` 最多属于一个连续系列，一个系列可包含零到多个成品；“上一期”等公开承接按当前成品发布账号及用户明确前提判断，矩阵共创不自动共享前情。 |
| R10 | 每个 `GenerationRun` 恰属一个任务和一个实际操作者，可产生零到多个成果修订；`ActivityEvent` 追加记录明确操作者、对象及必要版本，采用、导出、发布等不组成互斥状态机。 |

补充关系：每条 `EvidenceRecord` 连接一条 `KnowledgeStatement` 与一个明确来源定位；是否必须有证据取决于知识硬度和风险。`ContentPlan` 中的建议项只有被用户选择执行后才形成 `BusinessTask`。`ProductComposition` 是中性组合，不证明已穿着、拍摄、执行或核验。

### 5. 内容生产与经营陈列的商品来源不对称

内容成品不设统一商品或逐句证据硬门。P1、P3、P4 在各自合同成立时可以有零个 `Product`；P2 至少保留一个真实商品专属锚点，P5 至少绑定一个真实品牌商品，任何承重商品主张仍须来自品牌商品资料、用户本次明确输入、可观察信息或有边界的专业解释。所有品牌内容仍须绑定任务、有效品牌表达、代表组织、发布账号、内容角色和一个主要产品合同。

D-005 型 `DisplayArtifact` 的上墙、不上墙、替代及数量必须逐项回指用户本次提交的商品来源集合，不得增加清单外商品或超过提交数量。有稳定商品编号时关联 `Product`；没有时保留原始描述并收窄方案，不得虚构商品。用户声明“本次可用”不等于 ERP 或笛语已经核验库存真值。

### 6. 多品牌与作用域的证据边界

租户、组织和品牌是三条不可互相替代的作用域轴：`Tenant` 负责隔离，`OrganizationalParty` 负责业务控制，`Brand` 负责内容与商品的适用范围。`ContentRole` 和 `PublishingAccount` 的业务控制权归一个明确组织，品牌只作为显式适用范围；`BusinessTask` 也必须声明企业共同表达、单一品牌或明确品牌集合，不能因为对象同属一个租户就自动跨品牌调用。这允许同一组织在获得相应品牌表达授权后复用或派生角色与账号配置，而不把角色永久归属于某个品牌，也不把品牌当组织子树。

第一版黄金试点由 D-019 限定为单品牌。因此同一组织多品牌、集团多品牌以及跨品牌角色/账号适用关系目前属于**已设计、未验证**；不得写成已经通过真实或黄金实证。D-017 和第一个多品牌租户可以在不改写既有单品牌数据含义的前提下，版本化细化关系类型和品牌适用范围。

### 7. 横向扩展、前后向兼容与鲁棒性

本项目不承诺对所有未知未来“零改造完全兼容”。正式兼容承诺为：

1. 已保存对象的稳定标识、所属应用和业务含义不被后续版本重解释；`ContentItem` 不能通过修订变成 `DisplayArtifact`。
2. 内容应用新增题材、媒体语法或内容产品时，优先新增版本化合同/子型并继续使用 `ContentItem` 根；陈列应用新增目的、区域、载体或培训/概念/巡检等成果时，优先新增版本化合同/子型并继续使用 `DisplayArtifact` 根。
3. 可选能力使用加法扩展；改变旧字段含义、旧必填条件或旧关系基数属于破坏性变化，必须发布新合同并显式迁移。
4. 新版本在承诺范围内继续读取旧任务、成果、规则和修订，并保留任务实际使用的品牌基线、角色、规则、方法及合同版本。
5. 旧消费者可以安全忽略不影响旧语义的可选信息，或明确拒绝未知合同/子型；不得静默误路由或把未知陈列成果当作可执行方案。
6. 新业务若产生新的数据真源、权限和独立生命周期，例如库存管理、空间数字孪生、多媒体生成或新应用模块，应通过新模块合同及必要的新对象接入；不得为了兼容预建空表、空接口或万能 JSON。
7. 未知合同、缺失来源或迁移失败时必须安全拒绝或降级，并保留原任务、成果修订及 `GenerationRun`/`ActivityEvent`，不得损坏历史数据。

未来能力的最小判断法是：若它不改变成果的真实性、业务控制、权限和生命周期，优先增加版本化合同或子型；若任一项发生实质变化，应采用新主版本、显式迁移或模块私有成果对象。只有某个候选已经被多个模块反复引用且具有独立生命周期时，才重新通过本 ADR 的一等对象准入门槛；不能为了横向扩展提前把所有载荷塞进万能 JSON。

因此，当前模型保证的是两个已知应用内部的稳定扩展缝、版本化向后兼容和安全失败，不是对所有未知未来的无条件兼容。

## 最小验证与证据边界

两条既有合成黄金路径完成关系校验：

- P3 总部品牌账号路径在补齐 `ContentRole`、`PublishingAccount` 控制组织后无语义歧义；
- 南城店墙面挂杆路径在补齐 `BusinessTask` 组织及 `Product` 租户、目录控制方和显式品牌关联后无语义歧义，二十件输入、十五件首次使用、十四件修订使用全部可追溯且不超量。

该验证只证明单品牌内容与陈列语义合同自洽。证据分级如下：

- **本轮关系级已验证**：单品牌总部 P3 `ContentItem` 路径，以及 D-005 双层挂杆 `DisplayArtifact` 路径；
- **已关闭的上游合同基线**：P1—P5 与 D-032 已分别通过各自黄金合同验收，但本轮未逐类重跑 D-016 关系路径；
- **设计兼容、尚未验证**：多品牌与跨品牌角色/账号、其他陈列载体、ERP 实时输入、多模态、复杂共创和新应用；
- **生产级未验证**：物理表/API、真实数据迁移、并发、故障恢复、性能和实际企业运行。

因此，D-016 关闭不证明 DIFY 工作流、真实陈列效果、真实企业采用、多品牌运行或未来扩展已经实现。

## 明确不做

第一版不把以下内容升级为核心类别：`Workspace`、`DomainConcept`、组织岗位、平台、内容产品合同注册表、抽象版本、商品款/变体独立类、内容造型规范、系列状态、账号连续记忆、私人偏好、任务种子、人工库存、空间/挂杆、陈列内部布局、趋势信号、权利记录、发布/反馈/评测独立对象、顾客/CRM、家庭/剧情人物及应用模块注册表。

不画全量 ERD，不冻结数据库表、JSON 结构、外键、级联删除、DIFY 节点、ERP 真值或发布账号—内容角色终身一对一。

## 影响

D-016 正式关闭。第一版已有一套可由 SQL、知识层、素材存储和事件记录共同消费的 25 类最小语义模型，同时为内容生产和陈列搭配保留有边界的横向扩展路径。下一项进入 D-017，精确描述共享组织多品牌、集团多品牌和零售经营关系；在真实多品牌租户进入前，相关关系只能标为设计兼容，不能冒充实证。
