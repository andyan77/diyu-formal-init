# 首次条件档案与日常输入合同 v0.1

## 1. 核心原则

> **首次自然说明一次固定条件；日常只提交本次实际可用商品及数量。**

系统内部可以结构化，但门店不填写复杂表单。固定条件只有发生变化时才更新。

## 2. 首次条件档案

### 最低必须可解析

| 条件 | 为什么必要 | 用户怎样说即可 |
|---|---|---|
| 挂杆大致容量或分层容量 | 决定能挂多少、间距和正侧挂比例 | “上杆侧挂最多六件，下杆最多八件，另有两个正挂点” |
| 上下/正侧挂结构 | 决定垂直搭配区和重点位置 | “上面一根、下面一根，左右各一个固定正挂” |
| 主要来客方向 | 决定第一视觉入口和左右顺序 | “顾客主要从左边过来” |
| 固定硬限制 | 防止生成不可执行步骤 | “右边不能挡消防设备”“左边正挂不能移动” |

### 可选但有价值

- 主要视线带或已知安装高度；
- 不能移动的层板、标牌或设备；
- 商品拿取方式；
- 用户希望保留的现状部分；
- 现场照片（首期不用于识图巡检，只能作为用户主动选择的参考素材）。

### 写入边界

- 档案属于指定门店和指定挂杆组；
- 首次自然说明可形成候选摘要，用户纠正后复用；
- 临时活动限制不自动写成永久条件；
- 条件变化时保留版本，不覆盖历史方案的生成时语义；
- 不要求精确到毫米，除非用户明确要求的数据依赖成果确实无法用相对条件成立。

## 3. 日常输入

日常最小输入是一条自然库存清单，例如：

> 今天这组墙可用：ZX-C218 3 件、ZX-S104 3 件、ZX-K126 4 件、ZX-P211 3 件、ZX-V113 3 件、ZX-Q117 4 件。

它只表示：

- 用户本次希望系统按这些商品和数量生成；
- 系统不得改数；
- 系统不核验现实库存；
- 不自动写入长期库存或 ERP 真值；
- 未上墙商品可以成为后备。

用户可顺手补充：

- “今天重点推外套”；
- “右侧施工，暂时不能用”；
- “顾客主要从另一侧过来”；
- “这次希望更疏一点”。

## 4. 默认目标

用户没有说明特殊目标时：

1. 调用品牌当前有效的墙面挂杆陈列标准；
2. 调用指定门店和挂杆组的固定条件；
3. 调用当前商品资料与 A/B/G 通用知识；
4. 生成一份完整执行方案。

品牌标准缺失时，只能：

- 生成明确标为通用候选的方案；或
- 提出一个最小方向问题。

不得把通用候选冒充品牌正式标准。

## 5. 最小追问规则

只在缺失信息会改变以下任一结果时追问：

- 是否仍属于墙面挂杆合同；
- 能挂多少；
- 主焦点放哪边；
- 固定正挂/上下结构是否成立；
- 用户明确要求的商品是否真实存在于清单；
- 权限、租户或品牌作用域是否成立。

每次只问当前最关键的一个问题。其他空白通过：

- 复用品牌标准；
- 使用相对数量和间距；
- 少挂；
- 删除无依据主张；
- 降级为通用候选。

## 6. 自然反馈输入

反馈只需说明差异和未变化条件，例如：

> 已经照这版挂了。右上正挂外套袖子压到旁边两件马甲，最近一件不好拿。其他位置、动线和库存都没变。

系统内部只提取：

- 受影响区域；
- 受影响商品；
- 新的操作限制；
- 明确未变化的条件；
- 是否需要局部修订或整体重算。

不生成巡检表、评分表或整改工单。
